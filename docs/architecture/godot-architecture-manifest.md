# Godot 架构约束与最终形态规范 (Godot Architecture Manifest)
**Version**: 1.0 (Alpha 阶段入口基线)

## 1. 核心架构纪律 (Core Architecture Discipline)
### 1.1 逻辑与表现绝对隔离 (MVC/ECS 变体)
*   **Model (C# 内核)**：持有世界状态、战力推演、剧情状态机与空间网格。绝对纯净，**严禁**引入任何 Godot 引擎层的库（如 `Godot.Mathf`, `Godot.RandomNumberGenerator`）。
*   **View (Godot 表现层)**：仅作为渲染器和输入捕获器。通过观察者模式（Signals/Events）监听 C# 内核的状态变化并触发对应动画/UI。
*   **Controller (桥接层)**： Godot 的 `_Process` 仅负责时间流逝的 UI 表现与输入转发，核心逻辑的 `Tick()` 步进由玩家输入严格驱动。

### 1.2 确定性与回放安全 (B.2 & B.3 红线)
*   核心推演必须维持 100% 纯整数数学模型。
*   **输入序列化**：录像/回放系统仅记录 `[初始 Seed] + [每回合离散指令序列]`。任何会导致浮点数漂移的即时物理碰撞结算，严禁进入 C# 内核。

## 2. 宏微观双层世界架构 (Macro-Micro Dual Layer)
### 2.1 大世界层 (Macro Overworld)
*   **表现形态**：微缩沙盒景观（类似《33号远征队》）。
*   **时间推演**：同步回合制。玩家不操作时，世界绝对静止；玩家移动/决策一次，底层内核全局结算一次 `Tick()`。
*   **实体降维**：视野外的 NPC 仅为网格坐标数据点，不实例化 Godot 节点。
*   **地形跨越**：底层 A* 寻路权重与实体轻功/境界能力挂钩，支持跨越特殊地形（高山、水域）。

### 2.2 箱庭与战斗层 (Micro Arena)
*   **触发机制**：实体坐标在大世界重合，即时判定转场。
*   **环境继承 (Context-Aware Arena)**：切入独立斗法台，但场景初始化时必须动态读取大世界当前坐标的地形、天气数据，并应用对应的材质贴图与底层属性乘区 Buff。
*   **多角度切入**：记录玩家在大世界触碰目标的矢量方向，映射为微观场景的出生点坐标。

## 3. 反应式回合制战斗 (Reactive Turn-Based Combat)
*   **核心范式**：指令下达 + 实时动作反馈（非纯 ARPG）。
*   **进攻判定**：玩家选择技能后，View 层触发 QTE（缩圈/节奏点击），将结果映射为离散乘子（如 `0.5x`, `1.0x`, `1.2x`）传回内核结算。
*   **防守博弈**：内核闪避/命中属性转化为 View 层的“反应窗口帧数宽容度”。弹反（Parry）成功可抵消内核原定的伤害。
*   **控制衰减 (balance-007 落地)**：强控技能（如迷魂）表现为敌方释放必须弹反的杀招；同类控制连续释放时，弹反判定窗口呈阶梯式放大（抗性递减）。

## 4. 动态江湖生成器 (Procedural Generation)
*   **数据驱动**：宗门分布、势力关系、剧情模板必须通过外部配置文件（JSON/YAML）定义，杜绝硬编码。
*   **生成管线**：读取剧情模板 → 宏观空间分割（Voronoi/BSP）划分势力版图 → 微观柏林噪声填充地形特征 → Godot TileMap 根据逻辑网格生成等距（Isometric）图块。

## 补充条款：实体、表现与叙事扩展

### 1.3 实体平权与宏大规模 (Entity Parity & Scale)
*   **主角平权**：主角在 C# 内核中仅是一个具备“外部输入特权”的标准实体，其灵根、出身、初始坐标遵循全套随机演化逻辑，不设写死的特殊机制。
*   **性能天花板**：为支撑海量同屏/全局实体推演，内核架构严格限制深层 OOP 继承，强制向 ECS（实体组件系统）靠拢，降低 CPU 遍历开销。

### 3.2 霸体与硬直转化 (Poise & Stagger System)
*   **韧性条映射**：底层的“体质/防御”属性在战斗场景中具象化为实体的韧性条（Poise Meter）。
*   **控制结算**：附带强控标签（如迷魂）的攻击不仅造成 HP 伤害，更产生极高的削韧（Poise Damage）。韧性清零前受击具备霸体（不断动作）；韧性击破后方可触发真实硬直（Hitstun）与控制状态。

### 4.3 动态叙事与文本生成 (Dynamic Narrative & LLM Integration)
*   **江湖记事本**：前端维持动态历史图谱 UI，用于追踪 NPC 恩怨链条与生平事件。
*   **生成器预留**：系统架构必须为外部文本生成（如本地大模型 / RAG 智能体工作流）留出接口。在每次输入初始 Seed 时，交由生成系统自动书写当前轮次的背景故事、宗门历史及功法视觉描述。
*   **数值变异**：功法等实体特征支持基于生成模板的随机数值浮动，但变异结果必须被限制在底层 PE（战力估值）平价公式的阈值内。
