## Summary

废弃"纯 PE 差值定胜负"（现状 100% 确定性碾压，胜率非 0 即 100），确立 **「Margin→关键事件触发概率」为主轴** 的战斗内核：PE 差（Margin）不再直接决定伤害数值，而经**整数查表**映射为命中/暴击/招架/破绽等**关键事件的伯努利概率**，逐次掷骰累加定胜负。伤害掷值带（±X%）降为次要风味轴。

内核（Model）只提供 **"方差基准 + 判定钩子（防守窗口/概率）"**；真正的状态反转（弹反清零、QTE 质量档）由**表现层（View：Godot QTE / 玩家 或 后台 NPC 检定）接管**——这是与《反应式回合制战斗》愿景榫卯咬合的关键（承 ADR-0004 Model→View 单向流 + 命令端口回写）。

**非对称结算**是本模型的自洽枢纽：同一张 `p_permille` 表对 **NPC vs NPC** 是"掷骰概率"、对 **Player vs NPC** 是"防守窗口帧数"——数值决定容错率（难度），玩家操作决定实际结果，故"无必中伤害则可靠手速无限格挡/越级斩杀"（魂系/只狼逻辑）在架构上成立。辅以**标签门控**（Blunt 关招架 / Elemental 关闪避 → 读招策略）与**阈值溢出 >1000‰**（高阶威压定身），战斗从"纯反应测试"升级为"读招策略（石头剪刀布）+ 反应速度"。

**三条地基一字不改**：整数确定性（ADR-0001 / B.2）、off 逐字节（ADR-0003 / B.3）、模块工厂（ADR-0002 / B.9）。方差全链路整数（禁浮点 Sigmoid，改整数查表 permille），同种子逐字节复现（B.2）。

## Context

### 现状与问题

当前战斗内核（`DuelEngine.ResolveR2`）基于纯 PE 差值：

- **基础伤害 = `PE/10` 每回合**（`BaseDamageDivisor=10`），**全程零 RNG**（源码注释明写"纯整数、确定性、无 RNG"，`DuelEngine.cs:12`）。
- HP = PE；双方每回合同时互扣 `PE/10` → 任何 PE 优势**线性累积**成决定性 margin。
- **`UT gap≥2` 直接 `auto-win`**（`margin=int.MaxValue`，`DuelEngine.cs:67-77`）——这是实测"差 999 / 差 1111"碾压的直接来源。

**实测证据**（2026-07-06 无头对局，seed=42）：切磋录充斥"差 999""差 1111"级碾压，跨路 PE 分化 18→649（同 UT 同四维），战斗沦为**数值比大小**而非博弈。此模型与《反应式回合制战斗 3.1/3.2》愿景**根本性冲突**——无法为前端 QTE、弹反窗口、韧性打断提供任何判定切面（结果已被内核掷死，View 无劫持空间）。

### balance-006 已判死"确定性 → 公平不可达"

`balance-006`（`53e83f4`，done）经方案 B `calibrationMode` 中性化设施 + PE-band 硬 gate，**证明了确定性对拍模型下同 UT 胜率 [40,60]% 数学不可达**（DuelEngine 无 RNG → HP=PE、dmg=PE/10 → 小 PE 差=大 margin → 胜率非 0 即 100），遂将 TR-BAL-001 的 [40,60]% 硬指标**降级为裸 PE-band 代理**。retro action item 明列"评估方差战斗模型（解胜率平价天花板，大架构决定需专门立项）"。**本 ADR 即该专门立项。**

### 目标战斗类型（用户裁定）：反应式回合制战斗

- **3.1 指令与实时反应**：进攻——玩家选技后 View 触发 QTE（缩圈/节奏点击）→ 结果映射为离散乘子传回内核结算；防守——内核闪避/命中转化为 View 的"反应窗口帧数"，弹反成功抵消原定伤害。
- **3.2 霸体与硬直转化（承 balance-007）**：体质/防御 → 韧性条；强控标签附带高削韧；韧性清零前具霸体，击破触发硬直；同类控制连续释放时弹反判定窗口阶梯放大（抗性递减，复用 balance-007 DR）。

### 约束（不可破）

- **B.2 整数确定性**：`Jianghu.Cultivation` 禁浮点（IL 扫描守）→ **概率映射禁用浮点 Sigmoid，改整数查表**。同种子逐字节复现。
- **B.3 off 逐字节**：`cultivation=false` 与 v1.0 逐字节一致；off 走 legacy `SparAction`，**不入 DuelEngine**。
- **B.9 模块工厂**：概率事件的效果仍经 `Modules` 工厂 / `SpecialModuleRegistry`，禁裸 `new EffectOp(七参)`。
- **ADR-0004 判定权移交**：View 劫持只经命令端口（整数意图），浮点/帧时/坐标不进 Core。

## Decision

**废弃"纯差值定胜负"，确立以「Margin→关键事件概率映射」为主轴、"伤害掷值带 + 命中质量档"为落地形态的战斗演算架构。**

### ① 主轴：Margin→概率映射（α，用户裁定）

PE 差（Margin）**不直接决定伤害数值**，而映射为**关键事件的伯努利触发概率**（命中 / 暴击 / 招架 / 破绽）：

- **同战力交锋**：关键事件概率趋近 50/50，悬念拉满。
- **强打弱**：低战力方仍保留**理论上极小非零概率**触发"连环招架"完成反杀——**弱胜强的叙事地基**（武侠张力不来自"多打 10 点血"，而来自"生死关头触发完美闪避"）。
- **胜负 = 一系列伯努利试验（命中/不命中/招架/暴击）累加的结果**，而非单次伤害比较。

### ② B.2 兼容化：整数查表 permille，非浮点 Sigmoid（Claude 落地裁定）

> **用户草案提及"可用 Sigmoid 映射函数"——但 Sigmoid 是浮点，B.2 铁律禁 `Jianghu.Cultivation` 出现 float（IL 扫描守，ADR-0001 明列"需要时用整数查表或定点近似"）。故落地为整数查表。**

- **映射 = 整数查表**：Margin（clamp 后）分桶 → 每桶预生成一个 **permille 概率 `p ∈ [0,1000]`**（千分比，1000=100%）。表形状仿 Sigmoid（数值策划在图上画 Sigmoid 曲线作**设计语言**，落地烘焙为整数桶表作**实现真相**）。
- **伯努利判定**：`bool hit = duelRng.Next(1000) < p_permille;`——纯整数，无浮点。
- **千分比全链路统一**：概率 permille [0,1000]、QTE 质量档乘子（会心 1500‰ / 普通 1000‰ / 擦伤 500‰）、削韧值全整数，最终步**向下取整**（承现有 `dmg * ratio / 10`、`Scale=100` 定点惯例，本 ADR 统一到 /1000 粒度用于概率与质量档）。
- **可校准性**：胜率 = 伯努利累加的闭式/蒙特卡洛可算量 → 数值策划可直接锁死"**战力差为 X 时胜率必落 [40,60]%**"（调桶表斜率即调曲线）。**balance-006 判死的 [40,60]% 硬闸门由此复活**（每场对局仍逐字节确定，胜率经 seed-sweep 统计得出）。

### ③ 核心推演管线（Pipeline）

```
每回合 / 每次交锋（承 DuelEngine.ResolveExchange 现有骨架，方差前置注入）：

  1. 基准核算 (Base)
     依 PE 差 + 武学面板 → 【基准伤害 = PE/10】+【基准防守帧数窗口】+【Margin 分桶】

  2. 概率映射 (Margin → p‰)         ← 本 ADR 主轴 α
     查表：clamp(Margin) → 桶 → 关键事件 p‰（命中/暴击/招架/破绽）

  3. 判定劫持 (UI Hook - Phase 1：命中质量档)
     · 有玩家介入：Godot 缩圈 QTE 结果 → 覆写质量档（Perfect/Good/Miss
       ⟺ 会心 1500‰ / 普通 1000‰ / 擦伤 500‰ 离散乘子）传回内核
     · 无玩家介入（后台推演）：内核 duelRng 代掷质量档（保底层逻辑统一）

  4. 伤害带方差 (Variance，次轴)
     基准伤害 × 质量档乘子 × roll(±X% permille) → 整数向下取整

  5. 【既有阶段不变】OnUse→OnDefend→SuppressionMatrix→软情境
     （经 ModuleResolver / Modules 工厂，B.9 守；calibrationMode 旁路仍可用）

  6. 防御劫持 (UI Hook - Phase 2：弹反)
     内核抛出【防守帧数窗口】（由敏捷/反应差 → 整数帧数，非掷死结果）
     · 防守方（玩家 QTE / 高敏 NPC 检定）在窗口内响应成功 → 伤害带清零，
       触发反向硬直事件
     · 承 ADR-0004：窗口帧数是 Model 出的整数钩子，玩家响应经命令端口回写

  7. 韧性副轴 (Poise & Stagger，承 balance-007/008)
     伤害结算后同步计算独立【削韧值】：
     · 韧性 > 0：保持霸体（Action 队列继续）
     · 韧性 ≤ 0：触发硬直（Stagger）→ 强制清空行动队列
                → 触发抗性递减（DR）修正下一次【防守帧数窗口】
```

**判定权分层（本 ADR 灵魂）**：内核铺"基底 + 悬念"（方差基准 + 概率钩子 + 削韧），View 用 QTE 质量档 + 反应弹反窗口"驾驶"，削韧打断作"刹车"。**内核绝不把"闪避/招架"结果掷死**——只出"该给几帧防守窗口"，成败由 View（玩家/NPC 检定）接管（否则 QTE 沦为无意义摆设）。

### ④ B.2 兼容：Duel PRNG 流派生（Claude 落地裁定）

方差需 PRNG，而现状 `DuelEngine.ResolveR2` 全程无 RNG、其唯一调用方 `SparAction` 亦不持 `IRandom`。落地规矩：

- **新增 `RngStreamIds.Duel = 9`**（append-only；现最大 id=8=Faction，冻结不复用 1–8，守 B.2"新随机流追加新 id，绝不复用既有，保 off 逐字节"）。
- **per-duel 种子派生**：`duelRng = cultRng.Split(RngStreamIds.Duel).Split(hash(clock, 排序后的双方 CharacterId))`——**纯世界状态派生，无挂钟**（`clock`=逻辑 tick，双方 id 排序保交换律无关）。同种子 → 同对局逐字节。
- **duel-local，不入 `World.Clone`**：`duelRng` 与削韧/概率状态生活于 `ResolveR2` 作用域，**非持久字段**——完全比照现有 `ControlLimiterState`（balance-007，`DuelEngine.cs:431` 注释"duel-local → CD/DR 计数同 duel-local，不入 Clone → B.3 天然安全"）。**off 更不调 DuelEngine，逐字节铁律天然守住。**

### ⑤ 碾压/越级边界：硬 auto-win 降级为映射表极端桶（解 α 顺带）

现 `UT gap≥2 → margin=int.MaxValue` 硬 auto-win，降级为**映射表的极端桶**（如 `p≈999/1000`）：

- 满足 C2 gate（gap≥2 高 UT 应 ≥80% 胜；999‰ 远超）。
- 保住 α 的"理论反杀"叙事（极小非零概率，弱胜强地基）。
- **极大 UT gap（≥3 大境界）的性能截断已定案**（决策 ⑩.2）：仅 NPC-vs-NPC 后台跳查表直接 Auto-Win（省 CPU）；Player-vs-NPC 不截断，交 ⑨.2 溢出机制吃保底帧（按键碎裂定身）。

### ⑥ 数值外置 + 版本锁定（承 A.10）

映射桶表、方差幅度 ±X%、质量档乘子（1500/1000/500‰）、削韧阈值、防守窗口帧数系数、保底帧数、溢出惩罚系数、标签门控表——全部**外置为整数 tuning knob**（`LimitsConfig` 扩展或独立 combat config），**非浮点 JSON**（整数/千分比）。新战斗模型 + 数值版本**锁定**（承 A.10 配置驱动·确定性张力）；旧无方差 replay 显式标记为不同模型版本（同种子跨模型版本不保证一致，属预期）。

### ⑦ 非对称结算（Asymmetric Resolution）：同一 permille 表的双重物理意义（用户裁定）

**关键洞察**：`p_permille` 映射表对不同结算对象有**两种物理意义**，同一套查表逻辑下分流——这让"人打 NPC 可靠手速无限格挡（只要敌方无必中伤害）"在架构上完全自洽（ACT 元素反应式战斗的核心灵魂 = 魂系/只狼式无伤/越级斩杀逻辑）：

| 结算对象 | `p_permille` 的物理意义 | 生死裁决者 |
|---|---|---|
| **NPC vs NPC**（后台纯模拟）| 闪避/弹反的**数学触发概率** | 内核 `duelRng.Next(1000) < p` 掷骰 |
| **Player vs NPC**（前端实战）| Godot 前端的**防守判定窗口帧数**（Window Frames）| **玩家物理操作**（在窗口内响应成功/失败）|

- **底层数值决定"容错率（难度）"，玩家物理操作决定"实际结果"**：高 p（玩家数值优势/碾压局）→ Godot 翻译为极宽裕反应窗口（如 15~20 帧）；低 p（越级受虐局）→ 窗口压缩至 2~3 帧。只要玩家拥有机械般精准反应、且攻击**未带必中标签**，允许**靠操作抹平数值鸿沟、实现跨大境界斩杀**。
- **这是"反应式回合制"相较"纯数值回合制"的最大魅力**：数值鸿沟不是判决书，是难度旋钮。
- **承 ADR-0004**：帧数是 Model 出的整数钩子，玩家响应经命令端口回写；帧→毫秒/输入采集全在 Godot 宿主层（浮点/帧时不进 Core）。

### ⑧ 两条硬性规则（用户裁定，写入决议）

**规则 A：防守保底帧（Minimum Frame Window）**
> 即便战力差达 `int.MaxValue`、触发映射表最极端桶（p 达/超 1000‰），**只要攻击方技能未携带 `Unblockable`（必中）标签，引擎必须向表现层输出 >0 的保底判定帧**（如 2 帧 / ≈33.3ms）——**捍卫"理论可操作性"**（无必中 = 理论可无限格挡）。保底帧数外置为整数 knob。

**规则 B：UI 劫持绝对优先（UI Override Supremacy）**
> 玩家 QTE 抛回内核的离散乘子（质量档/弹反成败），拥有**超越底层 `duelRng` 的最高优先级，无条件覆写该回合的伤害与削韧结算**。底层 RNG 掷值仅在"无玩家介入（后台推演 / NPC 侧）"时生效。

> 由此：底层依然是 B.2 纯整数查表确定性，前端获得合法且自洽的"**操作凌驾数值**"特权——二者不冲突（RNG 是 NPC 的手，QTE 是玩家的手，同一张表分流）。

> **连续格挡的资源与收益约束（用户确认"是"）**：为规避"单键弹反打天下"的音游化陷阱，连续格挡/弹反须挂**资源与收益约束**（如格挡消耗内力/耐力资源、连续弹反收益递减、崩招后惩罚窗口）——具体数值留实现期，本 ADR 登记为**必须存在的约束轴**（与规则 A 的"理论可操作"平衡：可操作 ≠ 零成本无限）。

### ⑨ 标签门控（Tag Gating）+ 阈值溢出（Threshold Overflow）（用户裁定）

**⑨.1 必中类型与防御策略互斥（Tag Gating）**——单纯"弹反"不再万能解，内核据攻击 `DamageType` 标签**动态开关** Godot 判定钩子：

| DamageType 标签 | Block（招架）钩子 | Dodge（闪避）钩子 | 强行错误操作后果 |
|---|---|---|---|
| **Slashing/Normal**（物理锐器/常规）| 开 | 开 | 玩家自由选择 |
| **Blunt**（重型钝击）`Unblockable_Weapon` | **内核强制关闭** | 开 | 强行 Block → **招架崩坏**（大量削韧 + 全额伤害）；必须交位移 Dodge |
| **Elemental/AOE**（元素/范围法术）`Undodgeable_Space` | 开 | **内核强制关闭** | 无敌帧无法躲；须靠法宝/元素护盾 Block，或霸体硬吃（Tank）|

> 由此博弈从"纯反应速度测试"升级为 **"读招与策略选择（石头剪刀布）+ 反应速度"**——看敌方招式标签选正确防御姿态，再拼反应。

**⑨.2 难度函数越界 > 100%（Threshold Overflow）**——permille 体系下难度**允许突破 1000‰ 硬上限**，形成绝对数值压制：

- **运算逻辑（全整数）**：高阶修士威压攻击算出"闪避难度"1500‰，防守方身法修正仅 −200‰ → 最终结算难度 1300‰（可 >1000）。
- **NPC 后台推演**：`duelRng.Next(1000)` 最大 999 → 难度 ≥1000‰ 时**数学上绝不可能掷成** → 实现高阶对低阶**绝对秒杀**。
- **Godot 帧数转化（UI Hook Suppression）**：难度溢出 1000‰ 时，溢出部分作**帧数惩罚因子**削减保底帧：
  ```
  Overflow      = max(0, 难度‰ − 1000)
  有效帧数      = 保底帧数 − (Overflow / 惩罚系数)      // 全整数，向下取整
  若 有效帧数 ≤ 0 → 引擎不向 UI 派发 QTE 信号
  ```
  表现：角色被绝对力量锁定，按键图标灰态/碎裂、无法操作——符合修仙"**高阶威压定身**"叙事。
- **与规则 A 的边界**：规则 A 的保底帧捍卫"无必中标签 → 理论可操作"；⑨.2 的溢出定身是**通过 Overflow 惩罚吃掉保底帧**实现的"境界威压"——二者经"是否带必中标签 / Overflow 是否吃穿保底帧"共同裁定，实现期须定死优先级（必中标签硬关钩子 > Overflow 吃保底帧 > 规则 A 保底）。

### ⑩ 开放子决策的最终定案（Finalized Decisions，2026-07-06 用户裁定）

框架内的 4 项残留子决策已全部定案，架构封闭：

**⑩.1 元素伤害格挡穿透（Chip Damage）——确立存在穿透伤害**
- 带 `Elemental/AOE`（`Undodgeable_Space`）标签的攻击被玩家 Block 或特定护盾成功格挡时：**免疫控制与硬直（保持霸体），但必须承受基础穿透伤害**。
- **纯整数计算**：穿透比例 = 基础固定千分比（如 `300‰ = 30%`），由双方 PE 差（Margin）提供修正系数。
- **公式示例（全整数向下取整）**：
  ```
  实际受击 = (基础伤害 × 300 / 1000) + Margin修正偏移
  ```
- **设计意义**：元素/法术无法靠格挡完全免伤 → 逼玩家用法宝/元素护盾/位移预判，而非无脑 Block。穿透比例/修正系数外置为整数 knob。

**⑩.2 极大 UT gap 的纯 Auto-Win（性能截断）——启用**
- **应用场景**：双方境界差 ≥3 个大境界（`margin ≥ 超大阈值常量`）**且为 NPC vs NPC 后台演算**时，**跳过查表与掷骰，直接返回绝对胜利（Auto-Win）**——千人同屏推演节省 CPU 周期。
- **Player vs NPC 不走此截断**：交由 ⑨.2「阈值溢出」机制自然吃掉保底帧，表现为按键碎裂定身（保住玩家侧"理论可操作"的语义一致性）。
- **注**：此为决策 ⑤ 极端桶之上的**性能优化短路**，仅后台 NPC 生效；阈值常量外置。

**⑩.3 连续格挡资源/收益约束（防音游化）——防守帧数递减（Frame Diminishing Returns）**
- **不引入额外体力条**（保持底层状态极简），改用**保底防守帧递减**：
  - 目标在设定窗口内（如 3 秒 / N Tick）**每成功触发一次连续格挡，其下一次判定的"保底防守帧"强制衰减 1 帧**。
  - 衰减至归零 → 触发**招架崩坏**（大量削韧 + 全额伤害，同 ⑨.1 强行错误操作后果）。
  - **衰减计数器在停止受击 X 个 Tick 后清零**（喘息重置）。
- **与规则 A 的关系**：递减吃的是"保底帧"本身 → 连续格挡会**动态压缩**规则 A 的保底窗口，故"无限格挡"有代价、有上限，杜绝单键弹反打天下。计数器 duel-local（不入 Clone，同 balance-007 DR 纪律）。

**⑩.4 裁定优先级（Resolution Priority）——判定链顺序（确保自洽无死锁）**
> 每次防守判定，钩子是否派发 + 帧数几何，按以下**绝对优先级**从高到低裁定：
1. **标签强制干预（Tag Overrides）**：必中/必防标签（`Unblockable_Weapon`/`Undodgeable_Space`）**硬性开关** QTE 钩子——**最高绝对优先级**（决策 ⑨.1）。
2. **难度溢出截断（Overflow Suppression）**：若难度 >1000‰，计算帧数惩罚 `Overflow/惩罚系数`，吃掉常规判定帧（决策 ⑨.2）。
3. **规则 A 保底（Base Frame Safety）**：若未被上述两项阻断，**必须输出 >0 保底帧**（决策 ⑧A）；连续格挡递减（⑩.3）在此层内动态压缩保底帧。

> **至此，adr-0008 的数学内核、判定边界与跨层通信协议已完全封闭。**

## Alternatives Considered

### 备选：方案 2（离散过招判定为主）—— **否决**
- **描述**：内核每次交锋直接掷 命中/闪避/招架/暴击 离散结果，最贴传统回合制（如梦幻西游）。
- **拒因**：**若底层直接把"闪避/招架"结果掷死，前端 Godot 就不该弹 QTE（结果已定）——直接摧毁 3.1 实时反应机制**，玩家 QTE 沦为摆设。适合纯回合制，不适合带弹反的 ACT 元素回合制。**（用户裁定否决）**
- **注**：α 主轴保留了"概率决定是否抛判定钩子"，成败仍由 View 接管——这正是方案 2 做不到的判定权移交。

### 备选：方案 3（CTB 势能/行动序为主）—— **否决**
- **描述**：Speed 决定行动序 + 连击/破势势能累积，悬念来自节奏。
- **拒因**：项目已有"霸体/削韧/硬直"（balance-007/008）作节奏控制系统；再引 CTB 会**双重切割时间轴**（既看速度条又算硬直），系统耦合过重、改动面大、极易死循环。**（用户裁定否决）**

### 备选：浮点 Sigmoid 直算概率 —— **否决（B.2）**
- **描述**：`p = 1/(1+e^(-k·margin))` 浮点直算，映射平滑。
- **拒因**：浮点进 `Jianghu.Cultivation` 破 B.2（IL 扫描 = 编译错误），且跨运行时（CoreCLR JIT / AOT / Mono）舍入分歧破逐字节复现。**改整数查表 permille（形状仿 Sigmoid）**，见决策 ②。

### 备选：保留纯确定性 + 仅方差伤害带（无概率映射）—— **不足**
- **描述**：只在 `PE/10` 上叠 ±X% 伤害带，不引关键事件概率。
- **拒因**：伤害带方差**不足以产生"生死关头完美闪避"级戏剧**，也给不了 QTE 弹反判定切面（无"是否抛钩子"的概率语义）；同 UT 胜率仍主要由 PE 差决定，[40,60]% 校准力弱。降为 α 主轴下的**次要风味轴**（决策 ③ 步骤 4），非主轴。

## Consequences

### Positive
- **完美解耦 Model 与 View**：内核出"方差基准 + 判定窗口"，Godot 表现层经 QTE/动画树"落实结果"（承 ADR-0004 判定权移交）。
- **满足武侠悬念与博弈感**：同战力 50/50，弱有理论反杀，强不必然全胜——戏剧涌现最大化。
- **balance-006 [40,60]% 硬闸门复活**：伯努利累加 → 胜率成可校准曲线，TR-BAL-001 可完整达成（不再是裸 PE-band 代理）。
- **QTE 榫卯**：概率直接决定"是否抛判定钩子 + 窗口宽度"，与 Godot 缩圈 QTE 完美嵌套。
- **操作凌驾数值（ACT 灵魂）**：非对称结算 + 保底帧 + UI 劫持优先 → 玩家可靠手速无伤/越级斩杀（魂系/只狼逻辑），同时后台 NPC 走概率掷骰，同一张表分流不冲突。
- **策略深度（防音游化）**：标签门控（读招选 Block/Dodge）+ 溢出定身 + 连续格挡资源约束 → 战斗是"读招策略（石头剪刀布）+ 反应"，非单键弹反。
- **B.2/B.3 守住**：整数查表 + duel-local PRNG，同种子逐字节复现，off 不受扰。

### Negative / 成本
- **1087 测试大规模重写**：现存战斗测试全基于"无方差确定性 margin"断言（如"差 X 必胜"）→ 需改为"seed-sweep 胜率落 [lo,hi]%""同种子逐字节复现"两类断言。**最大成本项。**
- **全流派重标定**：各路 PE 标定 + 伤害映射需重新拉表（承 balance-003 的 §5 校准 harness `BalanceMatrixDump`/`InvCrossDuelTests` 复用）。原 21 路 mul 校准需在概率模型下复算。
- **判定钩子契约需前置定义**：防守窗口帧数 / 质量档乘子的 Model→View 契约须在 Godot 战斗 View 前定死（本 ADR 定语义，实现 story 定字段）。

### Mitigation
- **重标定复用现有 harness**：`BalanceMatrixDump`（跨路派生视图）+ `InvCrossDuelTests`（C1 硬闸门）+ balance-006 的 `calibrationMode`（标定期旁路 Control/Suppression/CounterMul）迁移到概率模型，非从零起。
- **off 逐字节零成本守**：off 走 legacy SparAction 不入 DuelEngine + duel-local 不入 Clone → B.3 回归守自动通过（无需改 off 路径）。
- **版本锁定隔离旧 replay**：新模型标版本，旧无方差 replay 不跨版本比对（A.10）。
- **分阶段落地**：Phase 1（命中质量档 + 概率映射，后台 NPC 内核代掷）先跑通无头戏剧涌现 + [40,60]% 校准；Phase 2（弹反 View 劫持）待 Godot 宿主（承 A.10：无头数据日志证核心无死锁才接 View）。

## 附录 A：Margin→概率 permille 示例映射表（数值策划起手模板）

> **示意，非最终标定值**。形状仿 Sigmoid（同战力 500‰ 中点，战力差拉大趋近极端桶），落地为整数桶表。`p_hit` = 攻方关键事件（命中/有效伤害）触发 permille；防守方对称取 `1000 − p_hit` 的镜像逻辑（招架/闪避概率）。实现期由 `InvCrossDuelTests` seed-sweep 反标定，锁死 [40,60]% 闸门。

| Margin 桶（攻−防 PE 差，clamp 后） | `p_hit` (‰) | 单次事件语义 | 累加涌现胜率区间* |
|---|---|---|---|
| ≤ −60%（被压制） | 80 | 极难命中，多招架 | ~5%（弱胜强的理论窗口）|
| −60% ~ −20% | 300 | 劣势，偶有破绽 | ~25% |
| −20% ~ −5% | 440 | 微劣 | ~40% |
| **−5% ~ +5%（同战力）** | **500** | **50/50 悬念拉满** | **[40,60]%（硬闸门目标）** |
| +5% ~ +20% | 560 | 微优 | ~60% |
| +20% ~ +60% | 700 | 优势 | ~75% |
| ≥ +60%（碾压） | 920 | 极易命中 | ~95% |
| UT gap≥2（极端桶，替代硬 auto-win）| 999 | 近乎必中 | ≥80%（满足 C2）|

\* "累加涌现胜率" = 多回合伯努利试验累加后的整局胜率（示意值）；与"单次 `p_hit`"不同——单次 50/50 经多回合累加后整局胜率仍 ≈50%，但战力差经**每回合复利**放大整局差距。精确曲线由蒙特卡洛/闭式在实现期标定。

**校准手柄（tuning knobs）**：
- **桶边界**（Margin 分桶阈值）→ 控制"多大战力差算同级"。
- **桶 `p_hit` 斜率** → 控制胜率曲线陡峭度（直接锁 [40,60]% 宽度）。
- **回合数上限**（现 `MaxRounds=20`）→ 控制伯努利累加次数（回合越多，战力差复利越显 → 反杀窗口越窄）。
- **极端桶阈值 + 是否保留纯 auto-win**（决策 ⑤ 残留子决策）。
