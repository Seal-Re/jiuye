## 1. 概述与目标

构建一个**武侠"江湖"多智能体模拟内核**：程序化生成带背景设定的武侠角色（人设），让角色在**开放式连续时间中按各自节奏自主行动**（世界自由发展、不设固定回合数），产出可读的"江湖编年史"。核心内核是**纯 C# 逻辑库**，现在用 CLI 驱动，后期 Core 直接被 Unity 引用。

设计的两条"行动"路径统一在同一套**枚举动作 API**上：
- **规则脑（RuleBrain）**：目标驱动的效用机，确定性，无需任何外部依赖即可运行。
- **LLM 脑（LlmBrain）**：通过黑盒 API（function-calling）从枚举动作中选择，增强决策的人设一致性与涌现深度。

"随机但有上限"是核心数据原则：角色人设全随机生成（带硬上限），门派/功法/地图用手写模板池 + 随机实例化。

> 形式化补充（对照业界 spec 范式新增）：EARS 需求见 **§17**、质量目标见 **§18**、备选方案与取舍见 **§19**、ODD（ABM 文档标准）对照见 **§20**。本地仓三件套对位：`调研报告.md`(research) → 本文(design.md) → writing-plans 产物(tasks.md)。

### 1.1 分阶段交付（关键决策）

"涌现江湖"是**目标**而非 v1.0 默认。按 walking skeleton 竖切分阶段：

| 阶段 | 范围 | "完成"验收判据 |
|---|---|---|
| **v1.0 竖切** | 自带确定性 PRNG + 属性生成（定和+封顶，偏中庸分布） + **事件驱动调度器（开放式连续运行）** + RuleBrain 目标驱动效用机 + 3 个自包含动作（修炼/游历/切磋） + **角色生老死与新人设涌现（renewal）** + 1 个硬编码门派 + DomainEvent→文本编年史 + 不变量/分布测试 | 开放式长跑（数千逻辑步）稳定不崩、**单步计算量有界（无 freeze）**、江湖经生老死+新人可持续；确定性测试在固定 N 逻辑步上同种子逐字节一致；属性分布通过 KS/卡方形状测试；编年史人类可读 |
| **v1.1 涌现** | LlmBrain（GIGAX 式 function-call + strict/校验层） + 黑盒 API 限流/Batch/分层用脑 + 最小规划（意图槽 + 周期反思） + 防漂移护栏 | 行为层指标（活跃故事线数 / 关系图非平凡边占比 / 目标推进率 / 动作分布熵）落在合理带内；抽样人读编年史判定"像一段江湖" |
| **v1.2 戏剧** | 复仇/叛门/夺嫡/情仇 多阶段状态机 + 恩怨产生/继承/化解 + 门派内政（掌门更替/收徒逐出） + 模板池 JSON 扩容 | 自发产生 ≥1 条跨角色恩怨链并收束 |

本规格**完整定义 v1.0**，并为 v1.1/v1.2 划定接口预留与不做项，使后续阶段无需返工地基。

## 2. 已锁定约束（决策记录）

| 决策 | 结果 | 理由 |
|---|---|---|
| 玩家模型 | 纯模拟内核优先；玩家介入留到后期 Unity | 最干净、最可测；v1.0 无玩家耦合 |
| 语言/技术栈 | C# / .NET，Core 为纯逻辑库 → 后期 Unity | 避免重写（Core 纯逻辑可零改写进 Unity） |
| LLM 路线 | 黑盒 API（provider 无关，可插拔） | 零训练、低门槛；v1.1 才接入 |
| 降级 | 规则脑常驻；LLM 脑增强 | 无 API 也能跑（确定性单机），有 API 才涌现 |
| 随机生成范围 | 角色人设全随机（带 caps） + 门派/功法/地图 手写模板池 + 随机实例化 | 既有小说味骨架又有随机变化 |
| 世界观来源 | 借鉴武侠小说 | 门派/功法/恩怨体系 |
| 涌现 | 分阶段：v1.0 RuleBrain 确定性 → v1.1 LLM + 护栏 | 涌现是目标非默认；先验证"江湖好看" |
| 属性分布 | 偏中庸（多数平庸、少数极端） | 真实江湖感，高手稀缺才有戏 |

## 3. 架构总览

三层 + 依赖反转。**铁律：`Jianghu.Core` 永不引用 `Jianghu.Llm` / `Jianghu.Cli`。**

```
Jianghu.Core   纯逻辑库（netstandard2.1）。无 Console / 无 HttpClient / 无 UnityEngine。
               只依赖端口接口：IBrain / ILlmClient / IRandom / ISerializer / ITemplateLoader。
               含：World, Character, Sect, Stats, Memory, Relations, ActionSystem,
                   RuleBrain, DomainEvent, Chronicle, VariedSelector, LimitsConfig,
                   Scheduler（事件驱动调度）, Lifecycle（生老死+renewal）。
Jianghu.Llm    黑盒 API 适配器（引用 Core）。实现 ILlmClient（HttpClient）。含 LlmBrain（Core 内的 IBrain 实现，注入 ILlmClient）。
Jianghu.Io     序列化适配器（引用 Core）。实现 ISerializer / ITemplateLoader（System.Text.Json source-gen）。
Jianghu.Cli    控制台宿主（引用 Core + Llm + Io）。配置、种子、驱动模拟推进、渲染编年史/人设卡。
```

> 注：`LlmBrain` 是 Core 内 `IBrain` 的一个实现，但它**只依赖 `ILlmClient` 端口**，不依赖任何 HTTP 实现；真正的 HttpClient 适配器在 `Jianghu.Llm`。这样 Tick 循环只见 `IBrain`，Core 不含传输代码。
>
> Unity 阶段：直接引用 `Jianghu.Core`；`Jianghu.Llm` 的 HttpClient/Task 需替换为 UnityWebRequest/UniTask 的等价适配器（见 §6.4 已知迁移代价）。**"零改写"承诺收窄为：Core 纯逻辑零改写；Llm 传输层会换实现。**

## 4. 地基规格（6 项 must-fix，动工前钉死）

这些是评审认定的"承诺证伪点"，早做便宜、晚做灾难。

### 4.1 确定性整数 PRNG（`IRandom`）

- Core 自带确定性 PRNG 实现（**PCG32** 或 **xoshiro256\*\***），全程 `uint`/`ulong` 整数运算。
- **禁用** `System.Random`（跨 .NET Framework/.NET 6+/Mono/IL2CPP 同种子序列不同）与 `UnityEngine.Random`。
- **确定性状态推进路径禁用 `double`/`float`**（IEEE754 在不同后端/优化等级舍入可能不同）；需要比例时用整数或定点。
- PRNG 内部 state 必须可 `GetState()`/`SetState()`，并写入 `World` 快照（确定性续跑的最小充分条件，非"复杂存档"）。
- 验收：固定向量测试——硬编码前 N 个输出值，在 CLI 与（后期）Unity 双端比对一致。
- **运行清单（run manifest）**：每次运行/快照持久化 `{种子, LimitsConfig 哈希, 内核版本, 规则集版本}`。自然语言描述不足以唯一确定可复现状态——执行态须连同种子/配置/版本一并保全（reproducibility 文献共识）。

### 4.2 Core 纯净的工程硬门控

- `Jianghu.Core` 目标框架 **`netstandard2.1`**（Unity 默认 API 兼容级），桌面专属 API 在编译期即报错。
- 挂 **BannedApiAnalyzers**，把 `UnityEngine.*`/`System.Console`/`System.Net.Http`/`System.IO.File`/`DateTime.Now`/`System.Threading.Thread`/`System.Random` 列入 Core 禁用清单，CI 红线。
- `Jianghu.Core.csproj` 禁止 `ProjectReference` 到 Llm/Cli/Io。
- 验收：S9 增一条测试——"Core 在裸 netstandard2.1 下能编译且不引用任何禁用 API"。

### 4.3 序列化与传输反转成端口

- **传输端口** `ILlmClient`（纯传输，放 Core）：`ValueTask<LlmResponse> CompleteAsync(LlmRequest req, CancellationToken ct)`。Provider 差异封死在 `Jianghu.Llm` 的适配器内部。
- **序列化端口** `ISerializer` / `ITemplateLoader`（放 Core）：返回**已构造好的 POCO/record**。Core 内**禁止出现任何 `System.Text.Json`/`Newtonsoft` 的 using**。
- 具体 JSON 实现放 `Jianghu.Io`，用 **System.Text.Json source-generation**（`JsonSerializerContext`），非反射（IL2CPP/AOT 反射会抛 "no AOT code was generated"）；加 `link.xml`/`[Preserve]` 防裁剪，并在 IL2CPP 真机 build 验证（v1.1 进 Unity 时）。
- **删除 `ILlmBrain`**：它是过度具体化依赖。Core 依赖收敛为 `{ IBrain, ILlmClient, IRandom, ISerializer, ITemplateLoader }`。

### 4.4 异步脑接口

```csharp
public interface IBrain {
    ValueTask<ActionChoice> DecideAsync(DecisionContext ctx, CancellationToken ct);
}
```
- `RuleBrain` 用 `ValueTask.FromResult(...)` 零开销同步返回。
- `LlmBrain` 自然 `await`；超时/重试/fallback 用 `CancellationToken` + try/catch 封在 `LlmBrain` 内部，对调用方透明（fallback 时内部委托给一个 `RuleBrain` 实例）。
- **`IBrain.Decide` 只吃不可变 `DecisionContext`，不吃 `World`** → 从根上杜绝 Brain 越权改世界（只有 `ActionSystem` 能改）。
- Core 不感知线程；I/O 与主线程 marshalling 归宿主（Cli 用 `Task`，Unity 用 `UniTask`/dispatcher）。

### 4.5 CharacterId 不可变主键

- 引入 `readonly record struct CharacterId(long Value)` 作为唯一主键。武侠重名（无名僧）、改名（韦小宝→小桂子）、称号即名是常态，**姓名不可作 key**。
- `name` 降级为 `Persona` 上的普通可变字段。
- `Memory` 与 `Relations` 全部以 `CharacterId` 键控。
- `MemoryStore` 作为 `Character` 聚合根的**私有字段**，对外只暴露 `Remember(entry)` / `Recall(query)`；外部拿不到可变 `List` 引用——这才是真隔离。

### 4.6 确定性 DomainEvent 层 + 单一钳制 chokepoint

- **`DomainEvent`（不可变 record）是唯一真相源**，确定性、可重放。`Memory`/`Relations`/`Chronicle` 三方各自 `project`（投影）消费同一事件流，消除"从动作各自抠数据"的重复与漂移。
- **LLM 润色只作用于渲染层**，产出独立的 `FlavorText`，**绝不回写** `DomainEvent` 流 → 把非确定性隔离在领域边界之外，保住"同种子同输出"。
- **所有属性变更只能经单一 chokepoint `Stats.Apply(StatDelta)`**，在其中统一钳制并维护选定不变量；任何绕过 `Apply` 的直写在测试中被断言拦截。

## 5. 随机但有上限

### 5.1 两层模型

- **第一层·属性（生成期：定和 + 单项封顶）**：四主属性 `武力/内力/根骨/悟性`，总和固定 `S`、单项 `≤ C`。
- **第二层·抽取去重（最少使用加权）**：`VariedSelector<T>` 在模板池上抽取，优先最少使用、不重复直至覆盖全集。用于门派/功法/姓名/（v1.1）事件实例化。

### 5.2 属性采样（偏中庸，无拒绝构造）

- **目标分布写死为"偏中庸"**：多数角色四维接近 `S/4`，少数极端。写入 `LimitsConfig`（含集中度参数）。
- **采用无拒绝构造法**（Stafford randfixedsum 等价的整数版 / 带集中度的 Dirichlet 采样后投影），**禁止序贯逐项 rejection 采样**（在单纯形上非均匀：质量堆在等分点、边中点次峰、偏离极端 build）。
- **整数化**用最大余数法补足到精确和 `S`，并 clamp 到 `[min_i, C]`。
- 属性是**整数**。
- 验收：S9 用 **KS/卡方检验**对齐目标分布形状，而非只测"总和守恒+不越界"这种暴露不了偏置的弱不变量。

### 5.3 可行域非空断言

- `LimitsConfig` 加载期断言可行域非空：`Σ min_i ≤ S ≤ Σ C_i`，且每个 `C_i ≥ S − Σ_{j≠i} C_j`。给每个属性轴显式下限 `min_i`。
- 采样器在退化/空可行域时 **fail-fast 抛配置错误**（指向可行域过窄），绝不死循环。
- 默认值（`S=80`、`C=30`、`min=5`、4×30=120≥80）可行域非空，已验证。

### 5.4 运行期成长的钳制语义（消歧）

- **区分两套不变量**：
  - **生成期**：定和（`Σ=S`）+ 封顶（`≤C`）。
  - **运行期成长**（修炼 +delta）：**只受单项 cap（`≤C`）约束，总和允许增长**（不再定和）。
- `Stats.Apply(delta)` 内 hard-clamp 单项到 `C`；溢出部分丢弃（不重撒，避免重新引入分配偏置）。
- 验收：S9 对"每个枚举动作 × 随机 delta"做 property test（恒满足 `≤C`、无负值），而非逐动作 happy-path。

### 5.5 VariedSelector 形式化规格

- 状态 = 每元素使用计数（`long`，防溢出）。
- 选择 = 在**最小计数集合**内，用**注入的 `IRandom`** 均匀抽，抽后计数 +1。
- **单调计数天然实现 LRU 式去重，无"循环重置"特判**（避免周期性 artifact）。
- 每个用途持具名 selector 实例（`traitSelector`/`nameSelector`/`eventSelector`），各自历史，不共享桶。
- 池动态增删（v1.1 角色/门派增减）时新元素初始计数 = 当前最小计数（防永久饥饿或永久优先）。
- `Select` 接收 `IRandom` 参数，不内藏 RNG。
- 验收：边际频率收敛测试 + 池动态增删后无饥饿测试 + 连续抽 N 次覆盖全集且每元素出现次数差 ≤1。

### 5.6 确定性 vs 并发边界（关键）

- **v1.0 单线程顺序执行 + 全 RuleBrain + Selector/RNG 串行消费 → 完全确定性**（"同种子同输出"仅在此模式下成立，文档显式声明）。
- **v1.1 LLM 并发**：给每个角色派生**独立子 RNG 流**（主种子 + 稳定 `CharacterId` 经 SplitMix/PCG split 派生），角色处理顺序随机化用**独立的 order-RNG 流**；**LLM 决策阶段禁止消费领域 RNG**；多角色决策回写世界按**确定性 CharacterId 顺序**提交。
- **LlmBrain 路径不纳入"同种子同输出"不变量**（LLM API 即使 temperature=0 也非 bit-deterministic）；测试时用 mock `ILlmClient` 注入固定响应获得测试确定性；`World` 持久化记录每个 LLM 决策的原始响应以支持回放。
- 所有集合迭代必须基于**确定性有序结构**（`List` 或显式排序 key），**禁止 `Dictionary`/`HashSet` 枚举顺序**参与影响世界状态的逻辑；随机打乱用注入 `IRandom` 做 Fisher–Yates。

## 6. 领域模型

### 6.1 实体与值对象

```
CharacterId         : readonly record struct (long)   // 唯一主键
Persona             : record  { Name, Title, 出身, 性格(标签集/OCEAN), SectId?, 师承?, ArchetypeWeights }
                                // v1.1+ 追加：官方背景、秘密（仅在其机制落地时加入，避免"有字段无机制"）
Stats               : 武力/内力/根骨/悟性 (int)，经 Stats.Apply(StatDelta) 统一变更
MemoryEntry         : record  { Tick, EventRef, SubjectId, ObjectId?, Valence(int) }   // 结构化，非纯文本
Relation            : record  { FromId, ToId, Affinity(int -100..100) }  // 有向；v1.2 加恩怨强度/传播
Character (聚合根)   : { Id, Persona, Stats, NodeId(位置), Goal, Age, Lifespan, NextActAt(调度), Alive, private MemoryStore, Brain }
Sect                : { Id, Template, 弟子名册(CharacterId[]), 掌门Id?, 声望 }  // v1.0 单门派硬编码
WorldNode           : { Id, 名称 }   // v1.0 地图=抽象节点集（无寻路）
World               : { Clock(逻辑时钟), Characters(有序), Sects, Nodes, Chronicle, RngState,
                        SelectorHistories, Scheduler(优先队列状态), Deceased(逝者档) }
```

### 6.2 模板（手写数据；v1.0 极简，代码内常量或最小 JSON）

```
SectTemplate   : { 名称, 武学路线, 门规(行为约束), 收徒条件, 优劣, 地理 }   // v1.0 仅用 名称+武学路线
SkillTemplate  : { 名称, 类型(招式/内功/轻功), 威力区间[min,max] }         // v1.0 3-4 个
TraitPool      : 性格/出身/动机 词条池
NamePool       : 姓名/称号池
```
> v1.0 模板池**极简化**：`NamePool` + 1 门派 + 3-4 功法（仅 类型+威力区间）。克制关系/收徒条件/地理/优劣等字段推迟到 v1.1+，避免"第一次跑通卡在填表"。

### 6.3 目标（Goal，驱动 RuleBrain）

- v1.0 `Goal` 是带进度的活动状态，类型枚举（v1.0 仅两类）：`精进武学`、`游历访友`。
- 含达成条件、进度、达成后更替规则。角色的 `ArchetypeWeights`（如 武痴/游侠）给各动作效用加权。
- v1.1+ 扩展：`复仇<某Id>`、`扬名`；v1.2：`晋升掌门`、`求娶<某Id>`。

## 7. 时间模型 · 调度 · 双脑 · 生命周期 · 性能

### 7.1 事件驱动调度循环（开放式、单线程，v1.0）

世界**不设固定回合数**，按事件驱动的逻辑时钟**自由、连续推进**，可长跑、可中断快照、可续跑。**不是每个角色每 tick 都计算**——只处理"到期"角色（见 §7.5 性能）。

```
// World.Clock 是单调整数逻辑时钟；Scheduler 是按 (NextActAt, CharacterId) 排序的优先队列。
// 初始：每个角色按其行动间隔入队。
Advance(budget):                          // 宿主反复调用；budget=本次最多处理的决策数（或墙钟切片）
  processed = 0
  while Scheduler.HasDue() and processed < budget:
    actor = Scheduler.PopDue()            // 取到期且 CharacterId 最小者（确定性）
    World.Clock = max(World.Clock, actor.NextActAt)
    ctx    = BuildContext(actor, world)   // 纯映射 → 不可变 DecisionContext
    choice = await actor.Brain.DecideAsync(ctx, ct)
    events = ActionSystem.Execute(choice, actor, world)   // 校验 + 经 Stats.Apply 改世界 → DomainEvent[]
    foreach e in events:
        actor.Remember(project_memory(e)); world.Project(e); world.Chronicle.Append(project_text(e))
    Lifecycle.Tick(actor, world)          // 衰老/死亡判定（§7.4）
    if actor.Alive:
        actor.NextActAt = World.Clock + ActionInterval(actor)   // 角色按自身节奏重新入队
        Scheduler.Push(actor)
    processed++
  Lifecycle.MaybeSpawn(world)             // 周期性新人涌现（§7.4），用 spawn-RNG 流
// 宿主循环：while not stopRequested: Advance(budget)  —— 不固定终止；停于用户中断/灭绝/稳态
```

- 时间是连续逻辑 tick；`ActionInterval` 可因状态而异（活跃者频繁、隐居者稀疏），映射到世界时间（旬/月）。
- 确定性：同到期 tick 多角色按 `CharacterId` 顺序处理，平手用注入 RNG 打破；`World.Clock` + `Scheduler` 状态纳入快照 → 中断续跑逐字节复现。
- v1.1：世界级随机事件也作为调度项入队（VariedSelector 抽事件表，带频率 caps）。

### 7.2 DecisionContext（一等不可变对象）

```
DecisionContext : record {
    ActorSnapshot,                 // 只读快照：属性/目标/位置（声望维度 v1.1 补）
    IReadOnlyList<MemoryEntry>,    // 该角色相关记忆（已裁剪）
    IReadOnlyList<ActionType> AllowedActions,
    IReadOnlyList<NearbyActor>,    // 同节点/可交互对象（只读快照）
    IReadOnlyList<RecentEvent>
}
```
`BuildContext : World × Character → DecisionContext` 是纯映射函数，可单测。

### 7.3 RuleBrain = 目标驱动效用机（v1.0 可玩性核心，非龙套兜底）

对每个 `AllowedAction` 计算上下文相关效用，取最高（平手用注入 RNG 打破）：

```
Utility(action | actor, ctx) =
    w_goal   · 目标推进度(action, actor.Goal)        // 本动作对当前目标的推进，效用主项
  + w_rel    · 关系梯度项(action, ctx.NearbyActors)   // 好感/敌意连续值驱动 切磋/结识
  + w_opp    · 机会窗口项(action, ctx)                // 邻近有实力相近者/可探索新节点
  + w_arch   · 原型偏好(action, actor.ArchetypeWeights)
  − w_decay  · 重复衰减(action, 近期同动作连击)        // decay^streak，防自相似刷循环
```
- v1.0 三动作的效用：
  - **修炼**：∝ 精进目标权重 × (1 − 目标属性当前值/单项硬上限 C) × 悟性 − 重复衰减。
  - **游历**：∝ 游历目标权重 × (本节点无新交互对象) + 探索欲 − 重复衰减。
  - **切磋**：∝ 较技偏好 × (邻近有实力相近且非敌对者) + 求胜欲 − 重复衰减。
- 消费 §6 的丰富状态（目标进度、关系梯度、门规、机会窗口），**不是 3 条 if 反射**。
- v1.1：仇敌/反派角色**强制用 RuleBrain** 作为冲突注入的稳定锚（不受 LLM 对齐偏置影响）。

### 7.4 时间模型与生命周期（v1.0 最小版）

开放式长跑要"江湖代有才人出"，否则人口停滞或全灭。v1.0 最小生命周期：

- **衰老/死亡**：每次行动按经过的逻辑时间推进 `Age`；`Age ≥ Lifespan`（随机生成、带上限）或战斗致死 → 标记 `Alive=false`，移出 Scheduler，转入 `World.Deceased`（逝者档保留，供编年史与 v1.2 恩怨/师承传承；其 `CharacterId` 仍被他人记忆/关系稳定引用，不串台）。
- **新人涌现（renewal）**：`Lifecycle.MaybeSpawn` 按活跃人口下界/节奏触发，**复用核心人设生成器**随机生成新江湖人（门派收徒 / 后辈登场），用独立 **spawn-RNG 流**（确定性、且不与决策 RNG 串扰）。这正是"随机人设生成"核心特性的持续运转。
- **人口调控**：活跃人口维持在 `LimitsConfig` 配置带内（下界触发涌现、上界抑制涌现），防爆炸或灭绝。
- 门派兴衰 / 世界结构演化 → v1.2（v1.0 单门派固定）。

### 7.5 性能与可扩展性（防"人人每 tick 全算"卡死）

核心策略：**不轮询全员、单步有预算、按重要性分级**。

**v1.0（随调度器一起落地）**
- **事件驱动调度分摊**：优先队列只处理到期角色（§7.1），把 N 个角色的计算摊到时间轴，避免每 tick `O(N)` 全量决策齐发。
- **单步预算上界**：`Advance(budget)` 每次最多处理 `budget` 个决策（或按墙钟切片），超额留到下次 → **无论 N 多大都不 freeze**（牺牲推进速度而非卡死）；宿主可据帧预算调用。
- **空间索引**：角色按 `NodeId` 分桶，"邻近角色"查询 `O(节点占用)` 而非 `O(N²)`。
- **行动间隔分散**：不同角色 `ActionInterval` 不同，天然错峰，避免同 tick 扎堆。

**v1.1+（规模变大时再加）**
- **LOD 重要性分级**：重要角色高频精算；龙套低频；**远处/离屏角色用粗粒度聚合更新（统计近似，不逐个全量决策）**——RimWorld/Dwarf Fortress 式离屏抽象。
- **LLM 分层 + Batch + 并发上限**（已在 §9.4）：LLM 仅覆盖少数重要角色，是 v1.1 最大的成本/延迟杠杆。
- **增量 BuildContext / dirty-flag**：仅在相关状态变化时重建上下文。

> 性能与确定性不冲突：调度、预算、索引都建在注入 RNG 与确定性有序结构上，长跑/中断续跑仍逐字节复现（§5.6）。

## 8. 枚举动作系统（受约束 API，三方共用）

每个动作 = 前置校验（纯查询）+ 效果（产 DomainEvent）+ 文本投影模板。按命令模式 + 纯函数拆分，使其可脱离 World/Chronicle 单测：

```csharp
interface IAction {
    ValidationResult CanExecute(WorldView world, Character actor, DecisionContext ctx); // 纯查询，零副作用
    IReadOnlyList<DomainEvent> Apply(World world, Character actor, ActionChoice choice); // 经 Stats.Apply 改世界
}
// 文本渲染独立成 IChronicler 消费 DomainEvent；LimitsConfig 钳制在 Stats.Apply 内
```

### 8.1 v1.0 动作集（3 个自包含动作，构成最小涌现回路）

| 动作 | 效果 | 产出 DomainEvent |
|---|---|---|
| **修炼** | 按悟性提升某属性（经 Stats.Apply，单项封顶） | `CharacterTrained(actorId, stat, delta, tick)` |
| **游历** | 移动到相邻节点，触发邻接 | `CharacterTraveled(actorId, from, to, tick)` |
| **切磋** | 与邻近角色友好对战，按战力 + RNG 定胜负，改双方关系 | `DuelResolved(winnerId, loserId, margin, tick)`、`RelationChanged(a, b, delta, newValue, tick)` |

`切磋` 的关系增量在 v1.2 成为恩怨链的种子（输家可记恨/敬服）。

### 8.2 ActionChoice（强类型，非弱类型袋子）

```
ActionChoice = ActionType + 强类型参数载荷(record 封闭层级)
  修炼: { StatTarget }          游历: { ToNodeId }          切磋: { TargetId }
```
校验层据 `ActionType` 反序列化到对应参数 record，缺参/多参/非法即 reject → fallback。

### 8.3 后续 backlog（不在 v1.0）

`结交/拜师入门/收徒/夺宝/奇遇/闭关/行侠/隐退` → v1.1+。其中：
- `夺宝奇遇` 拆为 **夺宝**（带竞争对象/持有者参数，可生恩怨）与 **奇遇**（归周期性世界级随机事件表）。
- `行侠` 需 `声望/道德` 维度（v1.1 补维度后加入）。
- `收徒` 需师承链子系统（v1.2）。
- 复仇/叛门/夺嫡/情仇按戏剧轴各做"起-承-转-合"小状态机（v1.2）。

## 9. LLM 集成（v1.1）

### 9.1 决策协议（GIGAX 式）

- prompt 含 `PROFILE / MEMORIES / RELATIONS / SCENE + ALLOWED_ACTIONS`（参照 GIGAX）。
- 把 `ALLOWED_ACTIONS` 作为 tool schema 暴露，模型返回 tool call → 解析为 `ActionChoice`。
- **只从枚举选 + 填实体参数，绝不放任自由文本**（尊重调研报告 0-3 否决：纯 prompt 不可单独承载决策）。
- 接口预留**约束解码位**（grammar/JSON-schema），把"格式保证"前移，不把全部正确性压在事后校验。

### 9.2 forced tool 只是格式保证，非正确性保证

- OpenAI：`strict:true` + `additionalProperties:false` + **全字段 required**（否则 strict 不生效）。
- Claude：无等价 strict，**客户端必须补闭集枚举校验**。声明"可靠性下限由最弱 provider 决定，校验层不可省"。
- **实体参数（目标角色/门派/功法）的动态值域**：schema 无法表达"值域=本 tick 上下文子集"。`BuildContext` 每 tick 把当前合法目标**动态注入为 enum 候选**（缓存 schema 骨架仅换候选值，降低 grammar 重编译开销）；`Execute` 前做"存在性 + 可达性"校验，非法即判解析失败走 fallback。

### 9.3 thinking 与 forced tool 取舍（二选一，写明）

- Claude 上强制工具与 extended thinking 互斥。v1.1 默认 **方案 B**：接受 LlmBrain 主要贡献是"人设一致的风味 / 编年史润色 + 在 RuleBrain 候选上做人设微调"，**战术智力仍交 RuleBrain 效用机**。
- 可选 **方案 A**（两段式："先 auto 自由推理出草案 → 再 strict tool 规整"，成本翻倍但决策有深度），作为高价值角色的升级项。

### 9.4 成本 / 限流 / 退避治理

- **分层用脑设为 v1.1 默认**：LLM 仅覆盖少数"重要角色"，龙套 + 反派/仇敌强制 RuleBrain。
- **token-bucket 按 TPM 限流** + 并发上限；`BuildContext` 给 MEMORIES **按 token（非按条目）裁剪**。
- 非实时模拟优先 **Batch API**（约 50% 折扣，天然适配异步内核）+ **prompt caching**。
- **重试按错误类型分流**：
  - 429 → 尊重 `Retry-After` 指数退避 + jitter；超预算直接降级，**不盲目重试**。
  - 5xx/超时 → 有限退避重试。
  - schema 越界/非法 enum → **不原样重试**（会再错），纠错提示重试一次或立即 fallback。
  - **限流时整批降级 RuleBrain** 作为一等降级路径。
- **校验失败分类**：区分"LLM 选了非法动作（语义越权，记为模型不可信）"与"网络超时（基础设施故障）"，分别可观测。

### 9.5 安全

- LLM 自由文本（编年史润色 / 含"秘密"字段的内容）若回填进世界状态或后续 prompt，存在 prompt 注入与不当内容风险。**润色文本只进渲染层，绝不回写 DomainEvent 或喂回决策 prompt**（见 §4.6）。

### 9.6 已知 Unity 迁移代价（记录在案）

`Jianghu.Llm` 的 `HttpClient` 在 Unity 抛兼容异常；WebGL 无线程，`Task` 无法正常 await。进 Unity 时该层需替换 `HttpClient → UnityWebRequest`、`Task → Awaitable/UniTask`。Core 纯逻辑不受影响。

## 10. 涌现与护栏（v1.1）

### 10.1 最小规划（补回被 YAGNI 砍掉的认知前提）

斯坦福 Generative Agents 消融证明：去掉 reflection + planning 后涌现显著退化。v1.1 加入**最小**版本（非全量向量库）：
- **意图槽**：每角色一个"当前意图 / 多步计划"，由目标生成，每次行动推进或重规划 → 让目标转化为跨多次行动的连贯行为链。
- **周期反思**：每隔若干次行动把记忆摘要成 1-2 条高层判断写回，影响后续效用权重。

### 10.2 防漂移与反对齐偏置护栏

- **冲突压力注入**：动机硬约束（门规优劣 / 目标排他 / 资源稀缺）让合作有代价。
- **动作分布监控**：若全局合作动作占比超阈值，则提高世界级冲突事件频率施加外部压力。
- **反派/仇敌强制 RuleBrain**：不受商用对齐 LLM 的"过度友善"偏置影响。
- **行为漂移检测**：长程监控 §10.3 指标偏移，超带告警。

### 10.3 涌现质量的可证伪指标（行为层回归门）

数值不变量不等于"有意义"（纯随机但确定的噪声也满足全部数值不变量）。增加行为层度量：
- 活跃故事线数（v1.1：持续的关系弧线 / 目标推进链；v1.2：恩怨/情/夺位状态机链条）
- 关系图非平凡边占比
- 目标推进率
- 动作分布熵（过高=随机游走，过低=死循环）
- 平均每角色每步可交互对象数（过低=角色孤立自循环）

跑 N 步后这些指标须落在合理带内才算通过；并抽样人读编年史做"是否像一段江湖故事"的定性 gate。

### 10.4 相遇/互动密度保障

离散节点 + 微缩规模 + 邻近过滤过严会让角色长期互不相遇 → 退化为闲修炼自循环。明确相遇判定 + 密度调参：用客栈/门派集会等**高汇聚节点**强制制造相遇；把"平均每角色每步可交互对象数"作为可观测指标。

| 成本（v1.1） | mock `ILlmClient` 累计 token/调用数，断言不超预算；LlmBrain 解析 + 越界 fallback（不打真 API） |

## 17. 需求规格（EARS 形式化，带 ID — 供 writing-plans 追溯）

对照 spec-driven development（Kiro/spec-kit）：本节是缺失的 `requirements.md` 腿。EARS 记法（Mavin, RE'09）：`【凡】Ubiquitous / 【在…期间】While / 【当…时】When / 【在含…变体】Where / 【若…则】If-Then`，每条 SHALL 一个响应。writing-plans 的 tasks 须回引这些 ID。范围 = v1.0。

**功能需求（F）**
- **R-F1**（凡）：内核 SHALL 程序化生成带背景设定的武侠角色人设，其属性满足"定和 + 单项封顶"。
- **R-F2**（当）：WHEN 调度器取出一个到期角色，内核 SHALL 由该角色 `IBrain` 从 `ALLOWED_ACTIONS` 选且仅选一个动作并执行。
- **R-F3**（当）：WHEN 一个动作产生效果，内核 SHALL 以 `DomainEvent` 记之，并投影到 Memory / Relations / Chronicle 三方。
- **R-F4**（在…期间）：WHILE 模拟运行，内核 SHALL 按事件驱动逻辑时钟连续推进，不设固定回合上限。
- **R-F5**（当）：WHEN 角色 `Age ≥ Lifespan` 或战斗致死，内核 SHALL 置其 `Alive=false`、移出调度、转入逝者档，且其 `CharacterId` 仍可被他者稳定引用。
- **R-F6**（在…期间）：WHILE 活跃人口低于配置下界，内核 SHALL 复用人设生成器以 spawn-RNG 涌现新角色。
- **R-F7**（凡）：内核 SHALL 输出按时序的江湖编年史与任意角色的人设卡。

**非功能需求（NF）**
- **R-NF1 确定性**（凡）：在全 RuleBrain + 单线程下，内核 SHALL 对同一 `(种子, 配置)` 产生逐字节一致的运行轨迹。
- **R-NF2 可复现**（当）：WHEN 从 World 快照反序列化续跑，内核 SHALL 产生与不中断运行逐字节一致的后续轨迹。
- **R-NF3 防卡死**（在…期间）：WHILE 单步预算为 `budget`，内核 SHALL 每次 `Advance` 至多处理 `budget` 个决策、其余顺延（永不 freeze）。
- **R-NF4 可移植**（凡）：Core SHALL 仅依赖 `netstandard2.1` 且不引用禁用 API（CI 门控），以零改写进 Unity。
- **R-NF5 健壮**（若…则）：IF LLM 调用超时/限流/返回越界动作，THEN 内核 SHALL 按错误类型退避或 fallback RuleBrain，绝不写入非法状态。
- **R-NF6 分布**（凡）：人设生成器 SHALL 使属性边际分布符合"偏中庸"目标（KS/卡方验收）。
- **R-NF7 上限**（凡）：内核 SHALL 保证任意时刻属性 `≤ 单项 cap`、无负值、记忆条目数 `≤ 上限`。

> 追溯：R-F* ↔ §6/§7/§8/§11；R-NF1/2 ↔ §4.1/§5.6；R-NF3 ↔ §7.5；R-NF4 ↔ §4.2；R-NF5 ↔ §9.4；R-NF6 ↔ §5.2；R-NF7 ↔ §5.4。

## 18. 质量目标（优先级 + 场景 — arc42 §1.2 / ISO 25010）

权衡冲突时按此优先级取舍（高者压低者）。

| 优先 | 质量属性（ISO 25010） | 验收场景 | 落点 |
|---|---|---|---|
| 1 | **确定性 / 可复现**（功能正确性·可靠性） | 同 `(种子,配置)` 两次跑逐字节一致；快照续跑复现 | §4.1, §5.6, R-NF1/2 |
| 2 | **可测试性**（可维护性） | 纯函数域 + 注入 RNG/Brain；Brain 只吃 `DecisionContext` 不吃 World | §4.4, §4.6, §12 |
| 3 | **可移植性 → Unity**（可移植性） | Core 裸 `netstandard2.1` 编译、BannedApi CI 红线 | §4.2, R-NF4 |
| 4 | **性能 / 可扩展**（性能效率） | 单步预算防 freeze；邻近 `O(节点占用)`；LOD 留 v1.1 | §7.5, R-NF3 |
| 5 | **可维护 / 可演化**（可维护性） | 端口反转、DomainEvent 单源、强类型 ActionChoice | §3, §4.3, §4.6 |

> 权衡规则示例：当"确定性"与"性能/便利"冲突，取确定性——故确定性路径禁 `float`、禁共享可变 RNG 并发（§5.6）。

## 19. 备选方案与取舍（Alternatives Considered — ADR 级精简）

每行即一条 Nygard 式 ADR 的浓缩（Context 见对应正文，Consequences 见"理由"）。

| 决策点 | 选定 | 否决的备选 | 否决理由 |
|---|---|---|---|
| 语言/栈 | C# / .NET | Python 优先 / 混合 / TS | Unity 目标下避免整体重写；纯逻辑核 C# 可写性足够（§3） |
| 认知架构 | Tick + 枚举动作 + 双脑 | Generative-Agents 全量 / 事件驱动剧本 + LLM 润色 | 全量太重、难确定性化、降级别扭；剧本式"真涌现"弱（§7/§9） |
| 时间模型 | 事件驱动调度（开放式） | 固定回合 foreach 全员 | 需"自由发展"+防卡死；固定回合 `O(N)` 齐发易卡（§7.1/§7.5） |
| PRNG | 自带整数 PRNG | `System.Random` | 跨 .NET/Mono/IL2CPP 同种子序列不同，破坏复现 + 移植（§4.1） |
| 决策真相源 | DomainEvent 单源 + 投影 | 动作各自写 Memory/Chronicle | 双写重复易漂移；LLM 润色回写污染确定性（§4.6） |
| 角色主键 | 不可变 `CharacterId` | 姓名 string 作 key | 武侠重名/改名→数据串台 + 全局可变状态（§4.5） |
| LLM 时序 | v1.1 分阶段引入 | v1.0 即上 LLM 涌现 | 成本/限流/漂移风险；先用 RuleBrain 验证"江湖好看"（§1.1） |
| 属性采样 | 无拒绝构造 + 偏中庸 | 序贯逐项 rejection / clamp 重撒 | rejection 在单纯形非均匀、偏离极端 build；clamp 重撒 boundary heaping（§5.2） |

## 20. ODD 协议对照（Grimm 2006/2010 — ABM 文档事实标准）

把设计映射到 ODD 的 Overview / Design Concepts / Details，用 **Design Concepts 清单做完整性 gate**。

| ODD 要素 | 本设计落点 |
|---|---|
| Purpose（目的） | 生成武侠人设并模拟涌现江湖（§1） |
| Entities & State variables | Character/Sect/World + 属性/关系/记忆（§6） |
| Process overview & Scheduling | 事件驱动调度、逻辑时钟、行动间隔（§7.1） |
| Emergence（涌现） | 关系/恩怨/故事线由局部动作涌现（v1.1 护栏 + 行为层指标 §10） |
| Objectives（目标） | 角色 `Goal` 驱动 RuleBrain 效用机（§6.3, §7.3） |
| Sensing（感知） | `DecisionContext` = 角色可感知子集（§7.2） |
| Interaction（交互） | 切磋/结交改关系；门派门规约束（§8） |
| Stochasticity（随机性） | 注入 `IRandom`，确定性可复现（§4.1, §5） |
| Collectives（集体） | 门派 `Sect`（§6） |

| Observation（观测） | 编年史 + 行为层指标（§11, §10.3） |
