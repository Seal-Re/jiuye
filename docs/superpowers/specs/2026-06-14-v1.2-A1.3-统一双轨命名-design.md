## 2. 命名两层模型（大境界名 + 小境界名）

境界全名 = **大境界名 · 小境界后缀**（如「炼气·后期」「金身·不坏金身」）。

- **大境界名**：每 `MajorTier` 一个（§1 backbone 或 §3 门类变体名）。
- **小境界名**：每大境界内 `SubLevelCount` 个。命名两策：
  - **标准前缀池**（SubLevelCount 2-4）：`初期 / 中期 / 后期 / 圆满`（截取前 N）；SubLevelCount=1 时无后缀（大境界名即全名）。
  - **per-major 自有名**（覆盖池）：migrated 路的细分有专名，如体修「炼气」大境界含小境界 `锻皮 / 淬肉`，佛修「金身」大境界含 `金身 / 不坏金身`。

**数据载体现状 + 裁决**：`RealmCurveDef.RealmNames[flatIndex]` 是**单列、按 flatIndex（=小境界粒度）**。
- SubLevelCount=1 的路（17 路）：`RealmNames[fi]` 既是小境界名也是大境界名（一一对应），无歧义。
- SubLevelCount>1 的路（体/命/符/佛）：`RealmNames[fi]` 是**小境界名**，大境界名**当前无独立载体**。

→ **裁决（A1.3 定名，A1.5 定存储）**：本稿在 §3 表中为每路**同时给出大境界名 + 小境界名**。存储二选一，留给 A1.5 渲染实现拍：
- **(推荐) 加 `MajorRealmNames[MajorTier]` 列**（纯加，类 SubLevelCount；大境界名独立，小境界名留 `RealmNames`，渲染 = `MajorRealmNames[major] + 小境界后缀`）。
- (备选) `RealmNames[fi]` 存「大·小」合成串，渲染期 split——不洁，弃。

A1.3-impl 先把 §3 的**小境界级**名规整进现有 `RealmNames`（safe rename）；`MajorRealmNames` 加列随 A1.5（含投影查询）落，避免本任务扩 schema。

## 4. 门类 flavor 政策裁决（一致性）

- **邪道 4 路（鬼/魔/妖/血）**：妖/鬼/魔 → **纯门类轨**（本稿已规整鬼 UT10/11、重写魔）；**血修例外保 backbone+血神**（血道「以血入道」近正统修真变体，且 backbone 名无 UT 错位，YAGNI 不强改）。政策：**去除 backbone 名与门类名的 UT 错位/混用**（魔必修、鬼/毒蛊已修），**纯 flavor 与 backbone 共存允许**（血/器/佛）只要 UT 不错位。
- **辅助路（丹/符/器/阵/驭兽/音/毒蛊）**：名归名（资历/门类 flavor），UT 归战斗当量（A1.4 重锚）。本稿**只规整名**，UT 不动。
- **顶境 flavor**：每路顶大境界宜有辨识度专名（剑仙/血神/妖圣/丹帝/魔尊…）；本稿补齐 法/阵 两个原缺 flavor 顶境。
