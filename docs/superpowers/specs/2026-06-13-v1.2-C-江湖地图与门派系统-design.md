# v1.2-C 江湖地图 + 门派系统 — 设计规格 (Design Spec)

- 日期：2026-06-13
- 状态：设计完成（含批判性自审 1 轮 + 修复 + walking-skeleton 边界），待 auditor 过 gate → writing-plans → developer
- 定位：补 v1.0 两处点名缺陷 —— ① 地图仅 3 抽象节点（`WorldFactory.cs:26-28` 硬编码 客栈/山道/市集，无地理/拓扑/资源）；② 门派仅 1 硬编码 `Sect(1,"无名谷")`（`Sect.cs` 仅 Id/Name/Disciples，无层级/门规/地盘/门派间关系/兴衰）。

## 0. 兼容基线（实读 v1.0 源码核实，凡冲突以源码为准）

| 接缝 | 源码事实 | 本设计约束 |
|---|---|---|
| 节点类型 | `WorldNode(NodeId Id, string Name)` 是 `sealed record`，仅 2 字段（`WorldNode.cs:3`） | **不改既有 2 字段顺序**；地理/类型/资源全走**侧表**或**纯加可空字段**（见 §2.4 两方案，采纳侧表） |

| 角色落点 | `Spawn` 用 `genRng.NextInt(3)` 落点（字面量 3，`WorldFactory.cs:47`） | cultivation/map-off 时**字面量 3 不动**（保 38 测试逐字节）；map-on 时改走 `genRng.NextInt(NodeCount)` 等价路径（§7.2 确定性分流） |
| 移动决策 | `RuleBrain.Travel`：`int to = ctx.Node.Value + 1`（线性 +1，**非拓扑感知**，`RuleBrain.cs:55`）；`TravelAction` 只查 `0<=To<NodeCount` 不查邻接（`TravelAction.cs:17`） | **关键裁决**见 §6：map-off 零改 RuleBrain；map-on 经 `DecisionContext` 注入邻接候选，Travel 选邻居（不破 RuleBrainTests，详见冲突裁决） |

| 四维契约 | `StatBlock` 恒 4 维 Σ=80 cap=30 min=5（`StatBlock.cs:12`，`LimitsConfig.cs:9-12`） | 地图/门派**零污染四维**（资源/声望/地盘全独立整数，不进 StatBlock） |
| 确定性纪律 | Dictionary 仅作索引不参与裁决；一切集合裁决前 `Sort`（`World.AtNode` `World.cs:74`）；禁浮点（`§4.1`） | 全继承：邻接表/门派表迭代前排序、整数百分比定点 `x*pct/100`、long 防溢出 |

**红线（不可破）**：纯整数禁浮点（IL 扫描覆盖 `Jianghu.Map`/`Jianghu.Sect` 命名空间）；不污染四维 Σ=80；map-off + sect-off 时 v1.0 的 38 测试逐字节不变；map-on/sect-on 同 (种子,配置) 逐字节复现 + Clone 续跑一致；新增不改 RuleBrain 效用代码（仅经既有通道）。

## 1. 概述与目标

把 v1.0 的「3 抽象节点 + 1 硬编码门派」升级为**程序化生成的江湖**：
- **地图**：区域（中原/西域/东海/北漠/南疆/蜀中…）→ 地点节点（城镇/客栈/门派驻地/秘境/洞天/险地/资源矿脉/坊市）→ 拓扑连通图（角色游历移动），随机但有上限整数确定性。
- **门派**：程序化生成多门派，含掌门/长老/弟子层级、门规、武学路线倾向（咬合 v1.2-A 修炼路线）、声望、地盘（咬合地图节点）、门派间恩怨/联盟（咬合 v1.2-B 戏剧 GrudgeLedger）、拜师入门/叛门、兴衰。
- **互锁**：地图秘境/洞天 = 修炼奇遇与传承来源（喂 v1.2-A 修炼路线/功法）；门派据地盘节点；戏剧弧的 Showdown 相遇密度由地图拓扑决定；门派恩怨注入 GrudgeLedger。

### 1.1 分阶段（本规格 = C.0 walking-skeleton）

| 阶段 | 范围 | 验收 |
|---|---|---|
| **C.0（本规格）** | ① 程序化地图生成器（区域/地点类型/拓扑连通/秘境洞天/资源点，caps + 整数确定性，扩 3→N 节点）；② 门派系统骨架（生成/3 层级/门规倾向/声望/地盘/门派间关系边/拜师叛门/兴衰最简一条）；③ 三处互锁接缝（秘境喂修炼 entry / 门派据地盘 / 门派恩怨注入 Grudge）；④ map-off+sect-off 回归安全网（38 测试逐字节） | 见 §10 验收清单 |
| **C.1** | 动态地图（节点占领易主/秘境刷新枯竭/坊市经济）；门派经济与资源链；门派大比/收徒大会 storylet；区域气候/地形 modifier 影响相遇与资源；地图 JSON 数据源 | — |

### 1.2 用户硬约束逐条对齐

1. **区域→地点→拓扑分层**：三层模型（Region / Node / Edge），§2。
2. **随机但有上限整数确定性**：区域数/地点密度/秘境频率全走 `MapLimits` caps + `mapRng`（Split(6)）整数采样，§5、§7。
3. **扩 v1.0 WorldNode（现仅 3 节点）**：N 节点纯加，3 节点退化为 `MapPreset.Legacy`（map-off 默认）保兼容，§2.6、§7.2。
4. **地点带类型 tag（供 entry/奇遇/资源/相遇密度）**：`NodeType` + `NodeTags`，§2.2、§3。
5. **秘境/洞天=修炼奇遇与传承来源（喂修炼路线）**：`SecretRealmDef` 携 `GrantPool`（功法/传承 ArtId/RealmBoost），经既有 EffectOp/CultivationState 落地，§3.3、§8.1。
6. **门派完整**：生成/层级/门规/武学倾向/声望/地盘/门派间恩怨联盟/拜师入门叛门/兴衰，§4。
7. **数据驱动可扩展不锁死**：加新门派/地点/区域 = 纯数据（追加 def），零改生成/移动/战斗核，§2.7、§4.8、§9（开闭验收）。

## 2. 地图模型（三层：Region / Node / Edge）

新命名空间 `Jianghu.Map`，落 `Jianghu.Core`（netstandard2.1，禁浮点）。

### 2.1 区域 Region（最外层，flavor + 生成参数 + modifier 锚）

```csharp
public readonly record struct RegionId(int Value);

public sealed record RegionDef(           // 数据驱动模板（CodeRegionSource 常量；C.1 出 JSON）
    string RegionKey,        // "zhongyuan"/"xiyu"/"donghai"/"beimo"/"nanjiang"/"shuzhong"
    string DisplayName,      // "中原"/"西域"… flavor，不进数值
    int Weight,              // 该区域被实例化的整数权重（caps 内加权抽）
    int NodeDensityPct,      // 地点密度（整数百分比，× MapLimits.NodesPerRegionBase）
    int SecretRealmBiasPct,  // 秘境频率偏置（× MapLimits.SecretRealmBasePct）
    int ResourceBiasPct,     // 资源点偏置
    int DangerTier,          // 险地基线 0..5（驱动相遇密度/险地频率）
    IReadOnlyList<string> FavoredNodeTypes,   // 该区域偏好的 NodeType key（加权）
    IReadOnlyList<string> FavoredPathKeys,    // 该区域偏好的修炼路线/门派倾向（互锁锚）
    IReadOnlyList<string> AmbientTags);       // 气候/地形 tag（C.0 仅 flavor + 险地基线）

public sealed record RegionInstance(         // 生成后冻结实例
    RegionId Id, string RegionKey, string DisplayName,
    int FirstNodeIndex, int NodeCount,        // 本区域占据的 NodeId 连续区间 [First, First+Count)
    int DangerTier);
```

**关键：NodeId 区间分配**。每区域占一段**连续** NodeId（前缀和分配，仿 v1.2-A2 `MinorStageOffsets`）。`RegionOf(nodeId)` = 在 `RegionFirstNodeIndex[]` 上整数二分。这保证 NodeId 仍连续 0..N-1（`TravelAction`/`Move` 边界检查不变）且区域归属 O(log R) 可逆纯整数。

### 2.2 地点节点 Node（类型 + tag + 容量 + 区域归属）

`WorldNode` record **2 字段不动**（兼容 `WorldNode.cs:3`）。地理/类型/资源走**侧表** `MapData`（决策 §2.4 采纳侧表方案）。

```csharp
public enum NodeType {        // 整数 enum，稳定值（驱动 entry/奇遇/资源/相遇密度）
    Town = 0,        // 城镇：高相遇密度、坊市可能、低危
    Inn = 1,         // 客栈：中相遇、过路歇脚、戏剧偶遇热点
    SectSeat = 2,    // 门派驻地：门派地盘锚、拜师入门点、门内相遇
    SecretRealm = 3, // 秘境：修炼奇遇/传承来源（喂 v1.2-A），高危高奖
    Cave = 4,        // 洞天：闭关/传承福地，突破加成（喂境界 A.2）
    Peril = 5,       // 险地：高危低相遇，险中奇遇
    ResourceVein = 6,// 资源矿脉：灵药/矿料采集点（喂丹修/器修资源）
    Market = 7       // 坊市：交易、情报、人多相遇
}

public sealed record NodeMeta(            // 侧表条目，按 NodeId.Value 索引（数组下标=NodeId）
    NodeType Type,
    RegionId Region,
    int DangerTier,                       // 0..5（相遇/奇遇/损耗整数门控）
    int EncounterDensity,                 // 相遇密度权重（驱动戏剧 Showdown / 偶遇 storylet）
    IReadOnlyList<string> NodeTags,       // 细分 tag（"灵药"/"剑冢"/"魔窟"…），供 entry/奇遇查询
    int? SecretRealmDefIndex,             // 若 Type∈{SecretRealm,Cave} 指向 SecretRealmDef
    int? ResourceDefIndex,                // 若 Type∈{ResourceVein,Market} 指向 ResourceNodeDef
    int? OwnerSectSlot);                  // 该节点归属门派槽（地盘，运行期可变 → 见 §4.7 可变性裁决）
```

**OwnerSectSlot 可变性裁决（自审 S1）**：地盘归属是**运行期可变**（门派兴衰会易主），但 `NodeMeta` 是 record 且 `MapData` 在冻结契约（§2.6）下浅拷安全。冲突解法：**`OwnerSectSlot` 不放 NodeMeta，移到 `SectLedger` 的可变侧**（§4.3 `SectState.TerritoryNodeIds`，门派→节点的正向可变表 + `MapData.NodeToSectIndex[]` 派生只读索引仅 C.1 易主时维护）。**C.0 地盘只读**（生成期一次性分配，运行期不易主），故 C.0 `OwnerSectSlot` 留在 NodeMeta 且 MapData 冻结浅拷安全；C.1 引入易主时再迁可变侧。本规格 C.0 锁定**地盘生成期分配后只读**，避免 Clone 风险（见 §4.7）。

### 2.3 拓扑边 Edge（无向连通图，邻接表）

```csharp
public sealed record MapEdge(int A, int B, int Cost);   // A<B 规范化；Cost=旅行整数代价(影响 ActionInterval, C.1)

public sealed class MapTopology {     // 生成后冻结
    public IReadOnlyList<MapEdge> Edges { get; }
    public IReadOnlyList<IReadOnlyList<int>> Adjacency { get; }  // 邻接表，每项已 Sort（确定性）
    public bool AreAdjacent(int a, int b);
    public IReadOnlyList<int> Neighbors(int nodeId);   // 升序，确定性迭代
}
```

**连通性硬保证**：生成器**必须**产出单连通图（否则角色被困子图、秘境永不可达、戏剧 Showdown 死锁）。算法 §5.3 用「随机生成树（保连通下界）+ 受 caps 限制的加边（增回路提升相遇）」，并在冻结前断言 BFS 全可达（INV-MAP-CONNECTED）。

### 2.4 决策记录：节点扩展用侧表 vs 改 record

| 方案 | 做法 | 选择 |
|---|---|---|
| A 改 WorldNode record | 给 `WorldNode` 加 Type/Region/Tags 字段 | **否决**：破 record 字段顺序与值相等性，`World.Clone` 的 `new List<WorldNode>(Nodes)` 浅拷语义需复核，且 v1.0 测试可能断言 record 形状 |
| **B 侧表 MapData**（采纳） | `WorldNode` 不动；`World` 加可空字段 `MapData? Map`，`NodeMeta[]` 按 NodeId 索引 | 纯加、零破既有 record；map-off 时 `Map==null` 完全 no-op（38 测试不变）；侧表数组下标=NodeId 是 O(1) 查询 |

### 2.5 World 接线（纯加可空字段）

`Sim/World.cs` 纯加：
- 字段 `public MapData? Map { get; }`（含 `RegionInstance[]`、`NodeMeta[]`、`MapTopology`、`SecretRealmDef[]`、`ResourceNodeDef[]`），可空。
- ctor 重载加 `MapData? map = null` 参数（默认 null → 既有 caller 与 38 测试签名/轨迹不变）。
- `Clone`：`Map` **生成后冻结（§2.6）→ 浅拷贝引用安全**（与现 `Nodes`/`Sect` 浅拷同纪律，`World.cs:159`）。**C.0 不深拷 Map**（前提是 Map 全程不可变，由 §2.6 冻结契约 + INV-MAP-IMMUTABLE 守护）。

### 2.6 地图冻结契约（INV-MAP-IMMUTABLE，C.0 安全前提）

C.0 地图生成后**全程不可变**：节点类型/区域/邻接/秘境池/地盘归属在生成期一次性定型，运行期只读。这与 v1.0「Nodes 浅拷」假设一致，使 `World.Clone` 对 Map 浅拷安全。
- 守护：`MapData` 所有集合为 `IReadOnlyList`/冻结数组；无任何运行期 mutate Map 的方法。
- **C.1 解冻代价**：一旦引入秘境枯竭/地盘易主，须把可变片迁出 MapData 到可变侧表并加入 Clone 深拷清单 + 续跑专测（本规格显式标注此边界，避免 developer 误在 C.0 引入可变性）。

### 2.7 可扩展（开闭）

- 加新区域 = 追加 `RegionDef`（CodeRegionSource 常量）。
- 加新地点类型 = 追加 `NodeType` enum 值 + 该类型的 `NodeTypeWeightDef`（区域偏好表自动纳入）；若需新奇遇语义才扩生成器（明确边界）。
- 加新秘境 = 追加 `SecretRealmDef`（其 GrantPool 引既有 ArtId/EffectOp，零改修炼引擎）。
- 生成器**不含任何区域/门派特例**（全数据查表 + caps 采样）。

### 3.1 NodeTags（细分，供 entry/奇遇查询）

`NodeTags` 是 string 列表（如 `["剑冢","上古"]`/`["魔窟","煞气"]`/`["灵药","千年人参"]`），flavor 取 jianshu 灵药剧毒/大山大川名库。**只进查询/Chronicle 文本，不进数值裁决**（数值走 NodeType/DangerTier/GrantPool 整数）。

### 3.3 秘境/洞天定义 SecretRealmDef（修炼奇遇/传承来源）

```csharp
public sealed record SecretRealmDef(
    string RealmKey, string DisplayName,    // "剑冢"/"藏经洞"… flavor
    int DangerTier,                          // 进入门槛/损耗整数
    EntryReqDef Entry,                       // stat 阈值 + tag anyOf（复用 v1.2-A EntryGate 思路）
    IReadOnlyList<GrantDef> GrantPool,       // 奇遇产出池（功法/传承/资源/突破助力）
    int VisitCooldown,                       // 同角色再访冷却（caps）
    int MaxGrantsPerVisit);                  // 单次最多得几样

public sealed record GrantDef(
    GrantKind Kind, string TargetKey, int Amount, int Weight);  // 整数加权

public enum GrantKind {
    ArtGrant = 0,       // 授一门具名功法（TargetKey=ArtId，进 CultivationState.ChosenArtIds）
    RealmAid = 1,       // 突破助力（TargetKey=资源/RealmBoost，喂 A.2 渡劫/瓶颈）
    ResourceGain = 2,   // per-path 资源（res:* delta，经 ApplyResource chokepoint）
    StatTemper = 3,     // 四维上限内淬炼（经 StatBlock.Apply，**仍守 cap，不破 Σ**）
    RosterGrant = 4,    // 灵兽/法宝/鬼兵条目（进 CultivationState.RosterIds）
    Inheritance = 5}    // 传承（埋戏剧/血脉钩，C.0 落 Chronicle + 可选 ArtGrant）
```

**奇遇落地经既有 chokepoint**（不旁路）：
- ArtGrant/RosterGrant → 写 `CultivationState.ChosenArtIds/RosterIds`（v1.2-A 既有结构）。
- RealmAid → 经 v1.2-A2 渡劫/瓶颈资源 chokepoint。
- ResourceGain → 经 v1.2-A `ApplyResource(key,delta,def)` 钳制。
- StatTemper → 经 `StatBlock.Apply(k,delta,limits)`（**钳到 cap，溢出丢弃，Σ 不变** `StatBlock.cs:19-25`）。
- 产 `SecretRealmExplored` / `ArtObtained` / `InheritanceReceived` DomainEvent（§8.3）。

**散修（cultivation-off / Cultivation==null）**：秘境奇遇对无修炼态角色仅产 StatTemper（守 cap）+ Chronicle flavor，不创建 CultivationState（保 off 轨迹纯净）。

## 4. 门派系统

新命名空间 `Jianghu.Sect`（注意与既有 `Jianghu.Model.Sect` 同义不同层 → 新类型 `SectDef`/`SectState`/`SectLedger`，**保留** v1.0 `Model/Sect` 不删，§4.6 迁移）。

### 4.1 门派定义 SectDef（数据驱动模板）


### 4.4 门派间关系（恩怨/联盟，互锁戏剧）

- **生成期**：按 `SectAlignment` 矩阵 + `AlignmentRivalryBias` 确定性建初始 `SectRelation`（正↔邪默认 Hostile/Rival；同盟约偏好 Allied）。地盘相邻（地图拓扑相邻节点属不同派）→ 加 Rival 偏置（地图×门派互锁）。
- **注入 GrudgeLedger（互锁 v1.2-B）**：Hostile/Rival 门派对 → 在 seed 控制下用 `mapRng` 子流为双方代表成员（掌门/长老）预置一对强 `Grudge`（B-spec §5 预注入冤孽的门派版，B-spec openQuestions 点名「SectFeuds 借 jianshu 名库」留 v1.2.1 → **本规格 C.0 接通最简一条**：每对 Hostile 门派注入 1 笔掌门级 Grudge，喂戏剧复仇弧）。

### 4.5 兴衰（SectPhase，C.0 最简一条）

C.0 兴衰由**成员数 + 声望 + 掌门在否**三整数确定性判定（每次 `World.Advance` 末尾 `SectDirector.Pump` 评估，节流）：

| 转移 | 条件（整数） |
|---|---|
| → Defunct | 在世弟子+长老+掌门 == 0（绝门） |
| Declining → Flourishing | 成员 ≥ FlourishThreshold 且 Prestige ≥ 阈值 |
| Flourishing → Declining | 掌门寿尽且无合格递补 **或** 成员 < DeclineThreshold |
| Rising → Flourishing | 成员 ≥ FlourishThreshold |

- **掌门递补**：掌门 `CharacterDied`（监听既有 `Lifecycle.Tick` 死亡，B-spec 同款监听）→ 声望最高在世成员递补（声望→武力→Id），产 `SectLeaderSucceeded`；无人 → `LeaderId=null` + Phase→Declining。
- **绝门处理**：Defunct 门派地盘节点 `OwnerSectSlot` 置空（C.0 只读前提下绝门是**单向终态**，不重分配 → 不破冻结；C.1 才做易主/吞并）。

### 4.6 v1.0 Sect 迁移（不删旧类型）

## 5. 地图生成算法（伪代码，全整数确定性）

`MapGenerator.Generate(ulong worldSeed, MapLimits caps, IRandom mapRng, IRegionSource regions, ISecretSource realms) -> MapData`

### 5.1 主流程

```
// 全部经注入的 mapRng（= root.Split(6)），整数采样，集合裁决前 Sort
1. 定区域数 R = clamp(caps.RegionMin + mapRng.NextInt(caps.RegionMax - caps.RegionMin + 1),
                       caps.RegionMin, caps.RegionMax)
2. 加权抽 R 个 RegionDef（按 Weight，前缀和整数轮盘；VariedSelector 去重保多样，复用 B-spec Step0 原语）
   → regionsPicked[]（抽后按 RegionKey 排序固定 RegionId 0..R-1，确定性）
3. 为每区域定节点数：
   nReg = clamp( caps.NodesPerRegionBase * region.NodeDensityPct / 100
                 + mapRng.NextInclusive(-caps.NodeJitter, caps.NodeJitter),
                 caps.NodesPerRegionMin, caps.NodesPerRegionMax )
   累加 → 前缀和分配连续 NodeId 区间（RegionFirstNodeIndex[]）；总 N = Σ nReg ≤ caps.NodeHardCap
4. 为每节点定 NodeType（§5.2 加权 + 秘境/资源频率 caps）
5. 建拓扑（§5.3 连通生成树 + caps 内加边）
6. 实例化秘境/资源 def 索引（SecretRealm/Cave→SecretRealmDef；ResourceVein/Market→ResourceNodeDef）
7. 冻结 + 断言：连通(BFS)、NodeId 连续、每区域 ≥1 节点、Σ 秘境 ≤ caps.SecretRealmHardCap
8. return MapData
```

### 5.2 节点类型分配（频率 caps）

```
for each node in region:
   // 先按 caps 保下界关键节点（每区域至少 1 个 Town、门派区至少 1 SectSeat）
   // 再加权抽 NodeType：weight = baseTypeWeight[type] × regionFavorMul(region, type) / 100
   // 秘境/洞天受全局频率闸：
   if drawnSecret and (secretCountSoFar >= caps.SecretRealmHardCap
                        or regionSecretCount >= caps.SecretRealmPerRegionCap):
        retype to fallback (Inn/Town)   // caps 优先于随机（硬上限）
   secretFreq = caps.SecretRealmBasePct × region.SecretRealmBiasPct / 100   // 整数定点
```
- **频率带 caps**：`SecretRealmBasePct`（全局秘境基频）/`SecretRealmPerRegionCap`（每区上限）/`SecretRealmHardCap`（全图硬上限）/`ResourceVeinBasePct`/`MarketPerRegionCap`/`SectSeatTotalCap`。随机有上限 = 任何频率乘子后**先 clamp 再用**，且全局/区域双重硬 cap 拦截（自审 M3）。

### 5.3 拓扑生成（保连通 + 受控回路）

```
// 阶段1：随机生成树（保连通下界，N 节点 N-1 边）
order = [0..N-1]; mapRng.Shuffle(order)              // 既有 RandomExtensions.Shuffle，确定性
for i in 1..N-1:
   j = mapRng.NextInt(i)                              // 连到已加入集合的某点 order[j]
   addEdge(order[i], order[j], cost=edgeCost(...))
// 阶段2：受 caps 限加回路（提升相遇密度、避免纯树长路径）
extra = clamp(N × caps.ExtraEdgePct / 100, 0, caps.ExtraEdgeHardCap)
for k in 1..extra:
   a = mapRng.NextInt(N); b = mapRng.NextInt(N)
   if a!=b and not adjacent(a,b) and degree(a)<caps.MaxDegree and degree(b)<caps.MaxDegree:
        addEdge(a,b,...)
// 阶段3：区域内聚 + 跨区主干（确保区域间至少 1 边，否则区域成孤岛）
ensureInterRegionBridges()   // 每对相邻 RegionId 至少 1 跨区边（确定性选边界节点）
// 阶段4：邻接表每项 Sort 升序（确定性迭代）；BFS 断言全连通（失败→抛，生成期硬错）
```

**连通失败处理**：阶段4 BFS 若发现不连通（理论上阶段1 树 + 阶段3 桥保证连通，断言为防御性），抛 `InvalidOperationException`（生成期 fail-fast，非运行期，不影响确定性轨迹）。

### 5.4 随机有上限四件套（地图版，全整数）

集中进 `MapLimits`（init-only record，仿 `LimitsConfig`，默认值过 `Validate`）：
1. **数量 caps**：RegionMin/Max、NodesPerRegionBase/Min/Max、NodeHardCap、SecretRealmHardCap/PerRegionCap、ResourceVeinCap、SectSeatTotalCap、ExtraEdgeHardCap、MaxDegree。
2. **频率 caps（整数百分比）**：SecretRealmBasePct、ResourceVeinBasePct、ExtraEdgePct、各 region BiasPct（乘子后 clamp）。
3. **抖动 caps**：NodeJitter（±整数，确定性 `NextInclusive`）。
4. **门派 caps**：SectCountMin/Max、SeatsPerSectMin/Max、SectPrestigeCap、AlliancePerSectCap。

`Validate` 追加断言：RegionMin∈[1,RegionMax]；NodesPerRegionMin≤Base≤Max；各 Cap≥0；ExtraEdgePct∈[0,100]；MaxDegree≥2；SectSeatTotalCap ≤ 预期 SectSeat 节点数（可行域非空，仿 `LimitsConfig.Validate` 可行域断言 `LimitsConfig.cs:31-34`）。

## 6. 移动 / 决策互锁（最关键裁决：RuleBrain 兼容）

**问题**：`RuleBrain.Travel` 现做 `int to = ctx.Node.Value + 1`（线性 +1，非拓扑感知 `RuleBrain.cs:55`），`TravelAction` 只查 `0<=To<NodeCount` 不查邻接（`TravelAction.cs:17`）。引入真图后，线性 +1 会走到非邻接节点（瞬移）、或走出有效范围（被 TravelAction 静默丢弃成原地）。

**裁决（参照 B-spec「零改 RuleBrain 效用代码」纪律）**：

| 方案 | 做法 | 选择 |
|---|---|---|
| A 改 RuleBrain 效用 | 在 `Evaluate(Travel)` 内查邻接选目标 | **否决**：破 RuleBrainTests，违 v1.0/v1.2 零改 RuleBrain 纪律 |
| **B 经 DecisionContext 注入邻居候选**（采纳） | `DecisionContext` 纯加 `IReadOnlyList<int> Neighbors`（map-off 时为既有线性候选 `[Node+1]`，保旧轨迹）；`RuleBrain.Travel` 改为「选 Neighbors 中确定性一个」——**但 RuleBrain 改动极小且 map-off 完全等价旧行为** | 见下「兼容性精算」 |
| C 改 TravelAction 加邻接闸 | `TravelAction` 查 `AreAdjacent` | **采纳为附加**：map-on 时非法目标返回空事件（原地），map-off 时 `Map==null` 跳过邻接闸走旧 `0<=To<NodeCount`（38 测试不变） |

**兼容性精算（核心）**：v1.0 黄金轨迹是 map-off（3 节点 Legacy）。要保 38 测试逐字节，必须使 map-off 下 RuleBrain 的 Travel 选择与 `Node+1` **逐字节等价**。
- 解法：`DecisionContext.Neighbors` 在 **map-off** 时由 `World.BuildContext` 填入 `[ (Node.Value+1) ]`（单元素，即旧行为）；RuleBrain 选 `Neighbors[确定性索引]`，单元素时恒选旧目标 → **map-off 逐字节等价**。
- map-**on** 时 `BuildContext` 填入真邻接 `Map.Topology.Neighbors(Node)`（升序），RuleBrain 用既有 `_rng.NextInt(50)` 平手打破之外**新增**一个确定性邻居选择（如 `Neighbors[_rng.NextInt(Neighbors.Count)]`）——这是 map-on 的**新确定性轨迹**（非回归，由新 INV-DET-Map 专测覆盖）。
- **RuleBrain 改动边界**：仅 `Evaluate(Travel)` 内目标计算从 `Node+1` 改为读 `ctx.Neighbors`。为守「零改 RuleBrain 效用」精神，**裁决：把邻居选择下沉到 `DecisionContext`/`World.BuildContext`，RuleBrain 只读 `ctx.Neighbors[0]`（map-off 单元素=旧值；map-on 由 BuildContext 已按 mapRng 预选好放 [0]）** → RuleBrain 的 Travel 一行从 `new NodeId(Node+1)` 改为 `new NodeId(ctx.Neighbors[0])`，map-off 下 `Neighbors[0]==Node+1` 逐字节等价。**这是本规格唯一触碰 RuleBrain 的一行**，且 map-off 等价 → RuleBrainTests 须复核但应全绿（gate 显式验证）。

> **自审 M1（致命，已修复）**：若不注入邻居而直接让真图 + 旧 `Node+1`，角色会瞬移到非邻接节点或卡死（`Node+1` 越界被丢弃→永不移动→秘境/相遇全断）。方案 B+C 组合解决：BuildContext 注入合法邻居 + TravelAction 邻接闸双保险。

**ActionInterval 与 EdgeCost（C.0 不耦合）**：`Lifecycle.ActionInterval`（`Lifecycle.cs:16-20`）C.0 **不**接 EdgeCost（接了会改 Scheduler 出队时序→冲击确定性面）。EdgeCost 仅存储留 C.1（旅行耗时）。C.0 移动瞬时（与 v1.0 同）。

## 7. 与 v1.0 集成 · 确定性

### 7.1 RNG 流隔离

`WorldFactory.CreateInitial` 在 `dramaRng = root.Split(5)`（B-spec 占）**之后**新增 `var mapRng = root.Split(6)`。不扰 `Split(1..5)` 顺序 → v1.0 + 戏剧既有 RNG 拓扑零变化。

### 7.2 map-off / sect-off 严格 no-op（38 测试逐字节安全网）

```
WorldFactory.CreateInitial(seed, limits, initialCount,
    MapPreset map = MapPreset.Legacy, bool sects = false)
```

### 8.1 地图秘境 → 修炼路线（喂奇遇/传承）
