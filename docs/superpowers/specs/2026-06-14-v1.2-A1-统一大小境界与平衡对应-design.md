## 1. 统一大境界主阶梯（canonical，冻结 UT 锚集）

**一界两层**（九野 WorldBible §3）：凡人 → 武夫（武侠层）→ 修士（仙侠层），共享一根 **UT 0-12 战力当量轴**。大境界**双轨称谓**（同 UT 不同 flavor 名），主阶梯 9 个大境界（MajorTier 0..8），UT 锚集冻结 = **{0,2,4,6,8,9,10,11,12}**（critique 实测 19/21 路天然落此集，非通胀）：

| Major | 仙侠称谓（修士轨） | 武侠称谓（武夫轨） | **UT** | 段 | 武夫可达 |
|---|---|---|---|---|---|
| 0 | 炼气 | 淬体 / 不入流 | **0** | 凡人入门 | ✓ |
| 1 | 筑基 | 二流 | **2** | | ✓ |
| 2 | 金丹 | 一流 / 绝顶 | **4** | | ✓ |
| 3 | 元婴 | 先天 / 宗师 | **6** | | ✓ |
| 4 | 化神 | 大宗师 | **8** | 武夫近顶 | ✓ |
| 5 | 炼虚 | **陆地神仙** | **9** | **武夫封顶** | ✓（CanAscend=false 止此） |
| 6 | 合体 | —（武夫不可达） | **10** | 修士独有 | ✗ |
| 7 | 大乘 | — | **11** | | ✗ |
| 8 | 渡劫→**飞升离场** | — | **12** | 顶 / 离场 | ✗ |

- **UT 步进**：major 0→4 每阶 +2（偶数主阶，密度走小境界）；major 4→8 顶段压缩 8/9/10/11/12（顶段稀，符合「越高越难分」）。
- **飞升离场**：修士达 major8 圆满 → 飞升，转 `World.Ascended` 档离场（UT12 硬封顶，YAGNI 不开第二位面，防数值通胀，WorldBible §1.1）。

## 2. 小境界 SubLevel + schema 变更

每大境界含 `SubLevelCount` 个小境界，**承载 UT 不变下的战力密度**（决策③）。小境界命名（全局共享前缀池）：

- 标准 4 小境界：`初期 / 中期 / 后期 / 圆满`（SubLevelCount=4）。
- 顶段可减（如 major8 渡劫 SubLevelCount=1，无细分，直接圆满即飞升）。
- 允许 SubLevelCount=1（该大境界无小境界）。

**RealmCurveDef schema 变更**（A.0 四列 → A.1 多列，纯加；这是迁移实质工作，非改名）：
```csharp
public sealed record RealmCurveDef(
    IReadOnlyList<int> RealmMultipliers,   // [flatIndex] 战力倍率（细到小境界）
    IReadOnlyList<int> UnifiedTierOf,       // [flatIndex] → UT（同大境界各小境界共享该大境界 UT）
    IReadOnlyList<string> RealmNames,       // [flatIndex] 展示名（仅 CLI，改名不破 determinism，§7）
    IReadOnlyList<int> RealmThresholds,     // [flatIndex] 升入阈值
    // —— A.1 新增（五/六列等长 + 标量）——
    IReadOnlyList<int> SubLevelCount,       // [MajorTier] 每大境界小境界数（长度 = 大境界数；前缀和投影输入，P5）
    bool CanAscend,                         // 修士=true / 武夫=false（P3，决策②）
    int MaxMajor);                          // 该路最高大境界索引（武夫≤5 即 UT9 封顶；修士=8）
```
- `RealmCurve.Validate` 扩：`RealmMultipliers.Count == UnifiedTierOf.Count == RealmNames.Count == RealmThresholds.Count`（flatIndex 等长，沿用）+ `Σ SubLevelCount == flatIndex 数`（前缀和闭合）+ `SubLevelCount.Count == MaxMajor+1` + 每 `SubLevelCount[i] ≥ 1` + `UnifiedTierOf` 非降 + `CanAscend==false ⇒ UnifiedTierOf.Last() ≤ 9`（武夫封顶硬校验）。
- PathValidator（M4）同步扩这些断言。21 路 standalone 测试同步。

## 3. 武夫 / 修士分层（决策②）

- **修士线**（21 条修真路 + 飞升）：`CanAscend=true`，`MaxMajor=8`，可达 UT12 后飞升离场。
- **武夫线**（凡俗武林 / 无修真路的武者，**A.1 字段就位、武夫 ladder 数据 A.1+ 或随凡人层落**）：`CanAscend=false`，`MaxMajor=5`，顶 = 陆地神仙 UT9。
- **体修归属裁决**：`ti_xiu_hengshi`（炼体）= **修士炼体一脉**（CanAscend=true，UT12 肉身成圣），用金身变体名（§4），**非** UT9 武夫——WorldBible §3.1 的「武夫 UT9 封顶」指**武侠层无修真路的凡人武者**，不指炼体修真路。故体修 A.0 的 UT12 顶**保留**（与决策②不冲突：武夫=path-less 层，体修=path 层）。
- **禁同路同 UT 多 major**（P6 佛修）：佛修 flatIndex8/9 都→UT12（金身/不坏金身）→ 合并为**一个 major（金身）+ 两小境界（金身/不坏金身 SubLevel）**，顶段只占一个 UT12；mul 150→225 改由小境界承载（UT 不变）。除非显式 plateau 声明 + 共享寿元/劫测试，否则不允许。

## 4. 21 路 realm 指派（统一主阶梯 + 变体名 + 辅助锚战斗当量）

**绝大多数路共用 §1 主阶梯**（同 UT 映射），仅**称谓**按门类换 flavor（同 UT 不同名，非 21 套割裂数值）：

| 类 | 路 | 称谓轨 | CanAscend/MaxMajor | UT 锚 |
|---|---|---|---|---|
| 剑/法/魂/雷/命/因果 | 主流修真 | 仙侠轨（炼气→飞升） | true / 8 | 主阶梯 0..8 → UT{0,2,4,6,8,9,10,11,12} |
| 炼体/佛 | 体 ti_xiu / 佛 buddhist | **金身炼体变体名**（淬体/易筋/金钟罩/金身/不坏金身/肉身成圣） | true / 8 | 同主阶梯 UT（佛顶段合并 P6） |
| 邪道 | 鬼/魔/妖/血 | **邪道变体名**（鬼：聚煞/养鬼/鬼将/鬼王/鬼仙；妖：开灵/化形/妖丹/大妖/妖圣；魔/血同理） | true / 8 | 同主阶梯 UT |
| 儒 | ru_xiu_haoran | **文位变体名**（童生/秀才/举人/进士/翰林/大儒/半圣） | true / 8 | 文位映射主阶梯 UT |
| **辅助** | 丹/器/阵/符/驭兽/音/毒蛊 | 仙侠轨（修为境界） | true / 8 | **UT 锚到真实战斗当量**（决策①，见下） |

**辅助路锚定（决策① / P2 核心）**：辅助路（丹/器/符 等）**直面战斗当量低**，其 `UnifiedTierOf` 锚到**实际 1v1 战斗力对应的 UT**（如丹修战斗≈化神当量 → 顶 UT8，**不锚 UT12**），不再让 mul=5 的丹帝吃 UT12 的寿元/劫/碾压。其「丹帝/符王/阵宗」**地位**走 **path-local rank**（已存在的 `itemTier`/`recipeCount`/`flameTier`/符储备 等资源），与 UT 解耦。修为境界**名**可仍叫「化神/炼虚」（flavor），但 UT 封到战斗当量。
> 具体每辅助路战斗当量 UT 由 §5 平衡标定确定（INV-CROSS 验）。

## 5. 平衡对应（UT = 战力当量，INV-CROSS 强制）

**核心契约**：同一 UT 上，任两条**战斗路**的典型角色 1v1 胜率 ∈ **[40%, 60%]**（INV-CROSS）。对治 P1 的 24-70× 失真。

- **平衡方法**：A.0 per-path `RealmMultipliers` × 战力公式系数是 build 期各自拟的（未对齐）。A.1 **归一化**：以一个 per-path「同 UT 典型角色战力」基准函数，调每路 `RealmMultipliers`/公式权重，使同 UT 典型战力收敛到同一量级带（±一定%）。
- **INV-CROSS 测试**（实现期 gate）：对每个有 ≥2 条战斗路锚定的 UT，构造各路典型角色（四维按生成期 Σ=80 中庸 + 该 UT 对应 realm + 标准功法配置），跑 N 次 1v1，断言两两胜率 ∈ [40,60]%。未过的 UT 禁止宣称可比 → 回调该路该 UT 的 mul/系数。
- **辅助路豁免**：辅助路（丹/符/器）**不进 INV-CROSS 战斗对拍的「必胜率」要求**（其设计就是战斗弱），但其 UT 锚必须 = 其真实战斗当量（即「丹修锚 UT8」意味着它 1v1 约等于 UT8 战斗路的下限，而非 UT12）。即辅助路按「战斗当量下限」锚 UT，地位另算。
- **碾压闸**：UT 差 ≥ K（如 2）→ 高 UT 碾压（必胜）。这要求 UT 真反映战斗当量（决策①保证），否则丹帝 UT12 会碾压真 UT12 战斗修士=荒谬（P2 已对治）。
- **寿元/天劫绑 UT**：`LIFESPAN_TABLE[UT]`、`TribGate ∝ UT`——因辅助路 UT 已锚战斗当量，丹修≈UT8 则吃 UT8 寿元/劫（合理），不再吃 UT12（P2 对治）。

## 6. 三轴解耦 + 可程序化（前缀和投影实例化）

- **flatIndex**（权威）：曲线索引，A.0 不变（RealmMultipliers/Thresholds/RealmNames 按此索引）。
- **(MajorTier, SubLevel)**（投影，不存）：由 `SubLevelCount` 前缀和确定性投影——
  ```
  offsets[0]=0; offsets[m]=offsets[m-1]+SubLevelCount[m-1]
  encode(major,sub) = offsets[major] + sub        // (大,小) → flatIndex
  decode(flatIndex):  major = 最大 m 使 offsets[m] ≤ flatIndex; sub = flatIndex - offsets[major]
  INV-REALM-1: decode(encode(m,s)) == (m,s)  // 投影可逆（先写此测试再落命名）
  ```
- **UnifiedTier**（跨路战力刻度）：`UnifiedTierOf[flatIndex]`（同 major 各 sub 共享该 major UT，§1 表）。
- **数据结构**：`RealmLadderDef`（§2 扩后的 RealmCurveDef）即承载体；小境界名 = 全局前缀池 + per-major 覆盖（可选）。加新路/改名/调小境界数 = 加数据（L0）；加 CanAscend/MaxMajor/SubLevelCount 列已是 A.1 一次性 schema 变更（L1 诚实标注）。

## 7. 数据迁移口径（A.0 → A.1）

1. **RealmNames = 纯 flavor**：Chronicle（整数 NewRealmIndex 渲染）与 StateSnapshot（INV-DET-Cult，整数 RealmIndex）**不读名** → **改名不破 INV-OFF-38 / INV-DET-Cult 逐字节**（一行论证：渲染链 `Chronicle.cs` 用整数、`Program.cs` CLI 才查名）。故统一命名可放心替换。
2. **加列 = schema 变更（实质工作量）**：RealmCurveDef 加 SubLevelCount/CanAscend/MaxMajor → 21 路构造器全改 + `RealmCurve.Validate` 扩多列校验 + PathValidator(M4) 同步 + 21 路 standalone 测试同步。这是 A.1 迁移的真活，非改字符串。
3. **规整以实际数组为准**（非注释）：A.0 有注释/数组 drift（如 `BodyHenglianPath.cs` 注释 9 名 vs 数组 10 含「金身」）→ 规整时以数组长度为准，先 grep 注释/数组一致性 + U+FFFD 乱码扫描。
4. **辅助路 UT 重锚**：按 §4/§5 把辅助路 `UnifiedTierOf` 从「资历 UT12」降到「战斗当量 UT」——这会改其寿元/劫档（变合理），需重跑相关测试。
