### B1. 奇遇Storylet库 — 全库 storylet 的 Arc 字段(1102/1103/1201/1202/1203/1301/1302/1303/1501/1502/1601/1602/1701/1702/1801-1804/1901/1902/11002/11003 标 Arc=Encounter)

### M1. 初始势力Landscape库 #10 血河魔宫据地「西陲·血煞原」+ #09 玄铁慕容世家(西陲·镔铁坞)+ 多条「夹在商会与魔道之间」描述 vs 地理骨架命名库 Region 3 西陲·万器谷地

- **修**: 要么在地理骨架库的 Region 3 西陲(或另一大区)追加「血煞原」魔窟地标 + 「镔铁坞」坞堡地标(L0 追加 LandmarkDef 行)并使两库一致; 要么把血河魔宫/慕容世家改锚到 GeoCanon 已存在的地标。注意: 若血河魔宫已按上一条改回百蛊渊(南疆), 则西陲魔道存在的问题随之消解, 但慕容世家的「镔铁坞」仍需在地理库落一个地标锚点。

### M2. HistoryAnchor 库 anchor101(玄昊大劫·道枢崩裂, 决战中心=中州, 鬼雾)+ anchor102(神魔互戕, 北漠, 妖兽潮) vs 地理骨架命名库 地标 Id8 玄昊古战场(北漠, HazardSeed=劫烬)
- **问题**: 库间矛盾: 玄昊大劫主战场到底在哪、什么灾劫种类两库不一致。地理库明文 Id8「玄昊古战场」在**北漠**, 注「三百年前玄昊大劫**主战场**遗迹」, HazardSeed=**劫烬**(地理库 §0 词表把劫烬专属绑玄昊大劫)。但历史库 anchor101(玄昊大劫本体)把决战中心放在**中州**(「神京所在中枢曾为决战中心, 灵机损耗最深」)并写**鬼雾**hazard; 而历史库写北漠的是 anchor102(一个泛化的神魔互戕battlefield), hazard 是**妖兽潮**, 不是劫烬。结果: (a) 玄昊大劫主战场, 地理库说北漠、历史库说中州; (b) 北漠的 hazard, 地理库种 劫烬、历史库 anchor102 写 妖兽潮; (c) 历史库全表无任何 anchor 在北漠落「劫烬」, 与地理库 Id8 的 HazardSeed=劫烬 对不上(地理库声明 HazardSeed 由 HistorySeeder 展开, 但历史库没有对应展开条目)。
- **修**: 对齐两库的玄昊大劫地理落点: 决定主战场是北漠(随地理库 Id8)还是中州(随历史库 anchor101 决战中心叙事), 二者统一。并让历史库为地理库 Id8 北漠玄昊古战场补一条 PlaceBattlefield+WriteRegionHazard(Kind=劫烬, Tag=北漠) 的锚点, 使 HazardSeed=劫烬 有对应的 HistorySeeder 展开来源; 同时消解「中州决战中心鬼雾」与「北漠主战场劫烬」的双主战场叙事(可改其一为次要战场)。

### M3. 奇遇Storylet库 1302 灌顶 Effect `RealmDelta(+1)` + 备注「Mentor 灌顶后 RealmDelta(-N)」; 1902 渡劫 `RealmDelta(+1)`; 11001/1403 等
- **问题**: `RealmDelta` 算子在 canon 无定义且语义可疑。canon 的三轴境界是 flatIndex/(MajorTier,SubLevel)/UnifiedTier 解耦(§3, A2-FINAL), 升境走 A2 的渡劫/晋阶 action(CultPump), 战力台阶=大境界×MajorMul。`RealmDelta(±1)` 既未说明改的是 RealmIndex(per-path 大境界 0..N)还是 UnifiedTier(0-12跨路刻度), 而本库其它谓词同时读 `realmIndex` 与 `unifiedTier` 两个量——若 RealmDelta 改 RealmIndex, 则跨 12 路语义不一(各路 N 不同, +1 在剑修=跳一个大境界、在魂修 7-major 可能跳 2 个 UT); 若改 UnifiedTier 则破三轴解耦(UT 是 per-path UnifiedTierOf 映射派生量, 不可直接 ±)。备注「Mentor RealmDelta(-N)」让前辈掉境更无机制依据(canon 无掉大境界机制, A3 渡劫失败只损资源不掉境)。这是在 storylet 层发明了一个绕过 CultPump 的升/降境通道。
- **修**: 用 canon 既有机制表达灌顶/渡劫破境: 渡劫(1902)直接走 A3 §渡劫机评劫(本库附录也说「走 v1.2 渡劫机评劫」, 那就让效果只产 break-progress/触发既有晋阶, 不引入 RealmDelta 原子); 灌顶(1302)若要跨阶, 应表述为「灌注大额 per-path 主资源 + breakAid 推满使其在既有晋阶 action 中破境」, 而非一个新 RealmDelta 算子。Mentor 的能量守恒改为转 Ascended/陨落(由 Lifecycle)而非 RealmDelta(-N)。需在 canon 层确认是否新增 RealmDelta 结构化 op, 否则属悬空算子。

### M4. 奇遇Storylet库 谓词 `Self.archetype ∈ {Martial, Cultivator}`(1501)、`Self.archetype`(1701等) 与 NamingPoolLibrary §2.4 `archetype ∈ {Martial, Cultivator}`
- **问题**: `archetype` 的取值集 {Martial, Cultivator} 引用了一个 canon 不存在的枚举组合, 混淆了两个不同的 canon 概念。源码 Persona.cs 的 `ArchetypeKind` 实为 **{Martial, Knight}**(武痴/游侠), 决定行为偏好(重修炼 vs 重游历)。而 {Mortal, Martial, Cultivator} 是 canon §3.1 的 **IdentityLayer**(身份层, 读 CanAscend 派生, 凡人/武夫/修士)。本库把两者拼成 {Martial, Cultivator} 当 archetype 用——Martial 在 ArchetypeKind 与 IdentityLayer 里恰好同名但语义不同(一个是「武痴行为型」, 一个是「武夫身份层」), Cultivator 则只属 IdentityLayer。这会让 implementer 不知该读 Persona.Archetype 还是 IdentityLayerOf(ch)。
- **修**: 明确区分: 若 storylet 要按「能否飞升/凡武修」门控, 用 IdentityLayer(Mortal/Martial/Cultivator, 经 IdentityLayerOf 派生); 若要按「武痴/游侠行为偏好」门控, 用 ArchetypeKind(Martial/Knight, 读 Persona.Archetype)。把 `archetype ∈ {Martial, Cultivator}` 改为对应的单一概念枚举, 不要混名。

### M5. 奇遇Storylet库 · 全库 Effect 行（如 1101 `AdjResource(swordWill,+30)`(钳≤cap=20+5×realm)、1401 `FormGrudge(Self→Foe,Kind=Slaughter,Intensity=85)`、1403 `TransferFortune(败→胜,30)`）
- **问题**: effect 数据行无法干净落成 canon 的 `Effect(Kind,From,To,Amount,Tag)` 单 Amount 形状。canon Effect 只有一个 `int Amount` + 一个 `int Tag`。而本库多数 effect 需要承载 2 个以上整数语义：FormGrudge 需 (GrudgeKind 枚举 + Intensity)；AdjResource 需 (resourceKey 字符串 + Δ)，且 swordWill 这类 key 是 string 不是 int Tag；TransferFortune 需 (源 + 目标 + Δ) 三元。单 (Amount,Tag) 装不下，要么改 Effect record 字段（改核心），要么 Tag 编码多义魔数（不可复现风险 + 破『可程序化』可读性）。
- **修**: 对每条 effect 显式给出到 canon Effect 四元组的映射：resourceKey/GrudgeKind 用整数码表（在库内附 key→int 码字典，纳入确定性裁决排序），Δ 落 Amount，目标落 To（RoleRef 已有 From/To 两槽，可承 TransferFortune 的源目标）；FormGrudge 的 Kind+Intensity 若塞不进单 Amount，则申明需扩 Effect record（L1）。不能停留在 `FormGrudge(Kind,Intensity)` 这种双参伪签名。

### M6. 奇遇Storylet库 · 1302 灌顶 `RealmDelta(+1)`、1902 渡劫 `RealmDelta(+1)`，及 §对齐核对『RealmDelta 极罕见且门控最严』
- **问题**: RealmDelta 直接增减 RealmIndex(flatIndex)，绕开既有渡劫 chokepoint。A2-FINAL（2026-06-13-v1.2-A2…-design.md:95,118）钉死：境界推进唯一合法路径 = Phase6 Breakthrough 状态机，`TribScore≥Gate → major++; flatIndex=offsets[major]`，且 RealmIndex 是 flatIndex 编码（小境界经前缀和投影）。一个 storylet effect 直接 `RealmDelta(+1)` 既非已定义 op，又跳过 TribulationResolver/飞升状态机，会把 flatIndex 设成非法值（INV-REALM-1 decode(encode) 可能破）。1902 自己也说『走渡劫机评劫…RealmDelta(+1)』——即真正的破境必须经渡劫机，RealmDelta 是多余且危险的旁路。
- **修**: 删除 RealmDelta 作为 storylet 直接算子。破境一律通过触发既有 Breakthrough/AscensionGate 状态机：storylet 只产『进入渡劫裁决』的信号（或喂 BreakProgress/BreakAid 到位让下次 Pump 自然渡劫），由 cult5 渡劫机判 TribScore≥Gate 后 major++。1302 灌顶若要跨阶，必须经飞升/渡劫 chokepoint + 守恒（Mentor 同步 RealmDelta(-N)/转 Ascended），且申明这需要核心支持（非 L0 数据行）。

### M7. 12路功法补遗库 · §0 schema 列『tribResist(只在突破劫 ResistTerms 注入)』为允许 TargetKey；12 路道心顶阶 art（dh_sw_wulou/dh_fo_puti/dh_so_yangshen 等）effect『渡劫/心魔劫 ResistTerms 注 (daoHeart,+w)』
- **问题**: 『往 ResistTerms 注入一个 term』不是 canon 任何 EffectOpKind 能做的事，tribResist 也不是 CultivationState.Resources 里的资源 key。A2-FINAL（line151-161）钉死 TribScore=Σ(src×weight in TribulationDef.ResistTerms)−ThreatPenalty+Roll，ResistTerms 是 **per-path TribulationDef 数据表里静态声明的 TermDef 列表**，加载期 fail-fast 校验 Src key 存在；它不是运行期可被 art EffectOp 动态追加的容器。而且 A3 表里魂/鬼/佛的心魔劫 ResistTerms 本就已含 (daoHeart,×4)——本库自己也说『坐实 A3 来源，不改其权重』。所以这些 art 的『注入』要么是无机制效果的 no-op（term 早已静态存在），要么需要『per-character 可变 ResistTerms 或新 EffectOpKind』这一核心改动。当前写法把一个 no-op/需改核心的东西伪装成 L0 EffectOp 数据行。
- **修**: 二选一：(a) 承认这些顶阶 art 对 ResistTerms 无独立机制贡献（daoHeart 项已在 TribulationDef 静态存在），把 effect 改为真正能落地的算子（如 AddResource(daoHeart,+w) 提升 daoHeart 存量，从而间接抬高已有 (daoHeart,×4) 项的贡献），删除『注 ResistTerms』与 tribResist key；(b) 若确要让 art 动态改某人渡劫抗性，明确这是新增『可变 per-char ResistTerm』机制 = 核心改动（L1+），不在本库 L0 scope。

### M8. 奇遇Storylet库 · 1102 道枢碎片 `AdjFortuneSelf(+40)`（注『碎片气运来自全局未分配池，非凭空增发』）；同型 1601/1602/1801 等 `AdjFortuneSelf(+X)`
- **问题**: 守恒律只在散文里声明，算子层未机制化。canon §7.5/INV-FORTUNE-CONSERVE 要求气运守恒（Σ Faction.Fortune + Σ 个体 fortune + 未分配池 == 预算）。`AdjFortuneSelf(+40)` 是单边加法，没有任何配对的『未分配池 -40』算子；运行期 storylet 也没有到 gen 期 FortuneTotalBudget『未分配池』的引用通道（那是 HistorySeeder 账）。对比本库 TransferFortune 是诚实双边（源减目标增），AdjFortuneSelf(+) 是纯增发，长跑破守恒/通胀。
- **修**: 凡 storylet 给个体/势力凭空加 fortune 的，必须配机制化的守恒源：要么从一个运行期存在的『未分配气运池』侧表整数扣减（需先定义该池为 World 字段并进 Clone），要么改成从某来源 TransferFortune（如碎片气运来自被吞噬的全局池实体）。不能用散文『非凭空增发』掩盖单边算子。AdjFortuneSelf 仅允许用于已守恒转移的目标侧。

## MINOR (5)

### M1. 奇遇Storylet库 1601 Precondition `Self.atSiteKind == Self.atSiteKind`(恒真) 与 11003/1602 等多处「同节点」判定写成 Self 与自身比较
