## 1. 戏剧弧状态机（复仇 6→5 状态）

复仇母题（范式 I 六阶段：受害→逃脱→蓄力→归来→搏斗→胜利）映射为确定性状态机。因 v1.0 灭门时受害者多在场、且 v1.2.0 不引入致死路径，"逃脱"合并入"受害"，落为 **5 状态**（折中四镜：lens 3/4 同主张 5 态）：

```
ArcStage: Victimized → BuildUp → Hunting → Showdown → Resolved | Abandoned
```

| Stage | 进入门控（整数） | 效果（产 DomainEvent） | NextDelay |
|---|---|---|---|
| Victimized | 点火即入（已有强恩怨 g.Intensity ≥ GrudgeIgniteThreshold） | `ArcStageEntered`；恩怨写入参与者 Memory（负 valence，大恨不被淘汰，依 `MemoryStore` 按 \|Valence\| 淘汰，`MemoryStore.cs:13-24`） | FirstStageDelay |
| BuildUp | 进入即记 `BuildUpBasePower`（自 `Power=Force*2+Internal+Constitution`，与 `RuleBrain.SelfPower`/`SparAction.Power` 同源 `SparAction.cs:14-15`） | `ArcStageEntered`；**Goal 覆写为 `Advance`**（让 RuleBrain 自发疯狂修炼涨战力，`RuleBrain.cs:44-50` 既有 Advance 加权 1500） | BuildUpDelay（长） |
| Hunting | `当前Power ≥ BuildUpBasePower + GrowthNeeded` **且** 仇人在世 | `ArcStageEntered`；**镜像负 Relations** 使 RuleBrain `notFoe` 项（affinity ≤ -50 → -500，`RuleBrain.cs:64`）自发偏好切磋仇人 | 中 |
| Showdown | 复仇者与仇人同节点（由上面两钩自发促成；超时则强制结算"狭路相逢"） | 复用 `SparAction` Power 公式判定胜负；`RevengeConsummated`（**非致死**：重创=关系崩坏到极值 + 恩怨化解/转移）；`Relations.Adjust` 到 -100 | 短 |
| Resolved / Abandoned | Showdown 完 / 仇人先寿尽 / 复仇者先寿尽 | `ArcStageEntered(Resolved)` 或 `ArcAbandoned(reason)`；Completed=true 退场 | — |

**死锁兜底**：若复仇者多 tick 未自发与仇人同节点（仇人一直游历躲开），弧在 `Showdown` 停留超过 `ShowdownTimeout` 后由 `TryAdvance` 直接强制结算一次，避免永久占用并发槽（四镜共识风险）。

## 2. 事件库 storylet（声明式 schema）

统一四镜的 `BeatTemplate`/`Storylet`/`IStorylet`/`ArcTemplate` 为单一声明式 schema（范式 A QBN + D beat schema）。**纯函数、无可变态、单例可单测**。

```
IStorylet {
  int Id;                       // 稳定整数键（cooldown/去重）
  ArcKind Arc; int Stage;       // 归属弧 + 弧内阶段
  long CooldownTicks; int CooldownScope;   // Global/PerActor/PerPair/PerSect
  int BaseWeight;               // 整数基础权重（范式 F）
  bool OncePerArc;
  IReadOnlyList<Predicate> Preconditions;  // 全 AND，整数阈值门控
  IReadOnlyList<Effect> Effects;           // 声明事件意图，由 director 翻译成 DomainEvent
  string ChronicleTemplate;     // 仅渲染层，不进数值路径
}
Predicate(RoleRef Subject, DramaVar Var, CmpOp Op, int Threshold)  // 纯整数比较
Effect(EffectKind Kind, RoleRef From, RoleRef To, int Amount, int Tag)
```

`DramaVar`→世界状态映射集中在唯一 `Resolve(RoleRef, DramaVar, ctx)->int`（仿 `BuildContext` 的纯映射，`World.cs:122-132`）。**v1.2.0 storylet 库用 C# `static readonly` 常量**（编译期校验 + 零 IO + 确定性，避 GBK/CRLF 历史坑），但抽 `IStoryletSource` 接口预留 JSON 适配层于 CLI 层（net8.0）启用——核库永远只吃接口、保持零 IO 纯度（采纳 lens 2 裁决）。

**v1.2.0 种子 storylet（恩怨线优先，6 个）**：101 门派血洗（注入 gen0 复仇弧）/ 102 恩怨偶遇（同节点 → 若 Hunting 推进 Showdown，否则关系恶化加码）/ 103 奇遇·秘境得功法（范式 M，BuildUp 阶段加速蓄力，落 `CharacterTrained` 复用既有投影 `World.cs:141-143`）/ 104 追杀（推进 Hunting）/ 105 比武招亲（埋 Insult 种子，情仇钩留 v1.2.1）/ 106 门派大比（批量切磋）。flavor 名取 jianshu 名库（范式 N），**只进 Chronicle 文本，绝不进整数数值路径**。

## 3. 选择器（点火 + 推进，全整数）

`DramaDirector.Pump(world)`（每次 `Advance` 末尾调一次，**非每 tick 全扫**）：

```
// A. 推进到期弧（戏剧调度器最小堆 (NextWakeAt, ArcId)，仿 Scheduler.cs:16-20）
while DramaScheduler.HasDue(Clock) and stepped < DramaBudget:   // DramaBudget 防卡死，呼应 Advance budget
   arc = PopDue(); if not ArcStillValid(arc): emit ArcAbandoned; continue
   arcRng = dramaRng.Split(arc.Id)            // 弧私有子流：顺序无关（范式 E）
   res = template[arc.Kind].TryAdvance(arc, world, arcRng)
   foreach e in res.Events: ProjectAndChronicle(e)
   if res.Completed: ActiveArcs.Remove(arc) else { arc.Stage=res.Next; reschedule(arc) }
   stepped++
// B. 点火节流
if Clock < _nextIgnitionCheckAt: return
_nextIgnitionCheckAt = Clock + IgnitionCheckInterval
if ActiveArcs.Count >= MaxConcurrentArcs: return     // 强度门控之一
// C. 候选收集（只扫强恩怨，不扫全员）+ 加权抽取
cands = []; foreach g in Grudges.AboveIntensity(GrudgeIgniteThreshold):   // 账本侧索引 O(强恩怨数)
   if HasActiveArcFor(g.Holder) or not IsAlive(g.Holder) or not IsAlive(g.Target): continue
   if OnPairCooldown(g.Holder, g.Target, Clock): continue
   cands.Add(candidate(g, weight = BaseWeight * IntensityMul(g.Intensity) * ArchMul(holder)))
if cands.empty: return
picked = WeightedPick(cands, dramaRng)     // 点火串行消费主流（顺序敏感）
CreateArc(picked); SetPairCooldown(...); emit ArcIgnited; schedule(arc)
```

**确定性铁律**：所有候选集合在抽取前 `Sort by (Weight desc, Id asc, CharacterId asc)`；`WeightedPick` 用前缀和 + 整数轮盘。**禁止任何 Dictionary/HashSet 枚举顺序参与裁决**（四镜共识 + 符合 `World.AtNode` 已有的 `list.Sort` 模式 `World.cs:74`）。点火主流串行消费，弧内推进用 `dramaRng.Split(arcId)` 私有子流（顺序无关 → 支持有上限随机排序不破连贯）。

## 4. 随机有上限（四件套全整数）

集中进 `LimitsConfig`（init-only，默认值不破既有 `Validate` `LimitsConfig.cs:28-38`）：
1. **cooldown**：`IgnitionCheckInterval`（点火节流）+ `ArcPairCooldown`（同 (复仇者,仇人) 对子，scopeKey 用 Szudzik 配对码避哈希碰撞）+ storylet 级 `CooldownTicks`/`OncePerArc` + 各 stage `NextDelay`。
2. **加权抽取**：`BaseWeight × 整数乘子`（不合格权重置 0，范式 G）；**乘子用整数定点 `w*mul/100` 并每步 clamp + 兜底 `w>=1`**，防小数值区截断为 0 导致候选集全空 / 除零（lens 3 已识别的风险）。
3. **并发上限**：`MaxConcurrentArcs`（默认 3）+ `MaxArcsPerCharacter`（默认 1，一人一时一条复仇主线）。
4. **强度门控**：`GrudgeIgniteThreshold`（≥才点火）+ `MassacreGap`（灭门需碾压差）+ `EscapeRatioPct`（整数百分比，禁浮点）。

`Validate` 追加：`MaxConcurrentArcs>=1`、`GrudgeIgniteThreshold in [0,GrudgeCap]`、`EscapeRatioPct in [1,100]`、`DramaBudget>=1`。

**VariedSelector 依赖（Step 0 先决）**：经全仓 grep 证实 `VariedSelector` 在源码中**不存在**，仅见于设计 spec（`docs/.../-design.md` §5.1/§5.4/§5.5）。spec 规定其语义：状态=每元素 `long` 使用计数，在最小计数集内用注入 `IRandom` 均匀抽、抽后 +1。Step 0 须按此 spec 形式化该共享原语并供 storylet 去重复用。

## 5. 背景驱动冤孽（恩怨账本 + 师承/血缘传递）

**GrudgeLedger（独立有向恩怨表，与 Relations 并存）**：采纳 lens 3/4，**不**把恩怨复用为 `Relations` 负值的真相源（否则日常切磋 `SparAction` 的 `-4`（`SparAction.cs:30`）累积会误把口角当灭门）。结构：`List<Grudge> _grudges`（确定性迭代）+ `Dictionary<long,List<int>> _byHolder`（仅查询加速，不参与裁决顺序）。`Grudge(Id, Holder, Target, Kind{Insult/Maiming/Slaughter}, Intensity[0,100], OriginTick, Generation, Cause, InheritedFrom?)`。合并规则确定性幂等：同 (holder,target) 取 `Kind=max, Intensity=max, Generation=min, OriginTick=首次`。**镜像一笔上限内负 Relations**（如 `-Intensity/3`，钳制）使 RuleBrain `notFoe` 项自然产生敌意摩擦——这是 drama→decision 的唯一受控通道。

**预注入"冤/孽"（范式 K）**：v1.2.0 简化为 `WorldFactory.CreateInitial` 在 seed 控制下用 `genRng` 独立子流（不碰 domain/spawn/drama 流）预置 1~2 对强恩怨 + 若干师徒边，保证首刀必有一条复仇线可观测。完整"门派对立矩阵自动注入"（SectFeuds，借 jianshu 名库）留 v1.2.1。

**师承/血缘传递（walking-skeleton 的"深"）**：戏剧专属角色关系走 **DramaProfile 侧表**（`DramaState.Profiles`，含 `CharacterId? Master/Bloodline`），**不**侵入 `Character`/`Persona` record 字段顺序（否则破 v1.0 测试，`Persona.cs:4`/`Character.cs`）。继承时机（最简一条）：复仇者**寿尽**（`Lifecycle.Tick` 的 `Age>=Lifespan` 死亡 `Lifecycle.cs:24-29`）且其恩怨未了 → 在世子嗣/弟子继承一条 `Grudge(Cause=Inherited)` 指向仇人，产 `GrudgeInherited` → 点燃下一条复仇弧 → **跨代恩怨链**（命中验收标准）。继承强度 `= 原强度 × InheritDecayPct/100`（整数衰减），`MaxGeneration=3` 封顶防无限链。继承人选取：年龄→武力→Id 三级确定性排序。

## 6. 与 v1.0 集成（DomainEvent / Scheduler / RuleBrain / Relations / World.Clone）

**戏剧引擎与 RuleBrain 的关系（唯一裁决）**：戏剧引擎是**独立事件流序列器**，**v1.2.0 不修改 RuleBrain 任何代码**（保 `RuleBrainTests` 不变）。它通过两条既有通道间接影响决策：(a) 覆写 `Character.Goal`（`set` 可变 `Character.cs:13`）→ BuildUp 设 `Advance` 让 RuleBrain 既有 1500 权重自发修炼；(b) 镜像上限内负 `Relations` → 触发 RuleBrain 既有 `notFoe`（affinity ≤ -50）项使其偏好切磋仇人。**不新增 `GoalKind.Revenge`、不新增 `ActionType.Vendetta`、不致死**（见 conflictsResolved）。Goal 覆写须在弧 Resolved/Abandoned 后**还原原 Goal**（防角色永久卡复仇态）；MaxArcsPerCharacter=1 防多弧抢同一角色 Goal。

- **Scheduler**：弧**不进** `Scheduler`（其项语义是 `(CharacterId, At)` 角色调度 `Scheduler.cs:7`，混入弧污染确定性出队序与快照）。弧用**独立 `DramaScheduler`**（同结构最小堆，键 `(NextWakeAt, ArcId)`，仿 `Snapshot/LoadFrom` `Scheduler.cs:48-49`），由 `Pump` 驱动。
- **DomainEvent 单源**：戏剧层只产不可变 record 事件，走同一 `Project + Chronicle.Append`。新增事件（继承 `DomainEvent(long Tick)`）：`GrudgeFormed / GrudgeInherited / ArcIgnited / ArcStageEntered / RevengeConsummated / ArcAbandoned`。投影规则纯加 case：`GrudgeFormed/Inherited`→写账本+负 valence Memory；`RevengeConsummated`→关系崩坏+恩怨化解/转移（非致死，复用既有 `Relations.Adjust`）。
- **Relations**：只经 `Adjust` 钳制（`Relations.cs:12-18`），不旁路。
- **World.Clone（R-NF2 命门）**：当前 `Clone` 深拷 alive/brains/Chronicle/Relations/Scheduler/RNG/Lifecycle（`World.cs:148-163`），**未含任何 drama 态**。必须扩展深拷贝清单：`GrudgeLedger.Clone()`、`DramaDirector`（含 `List<ArcInstance>` 逐个 `Clone()`、`DramaScheduler` snapshot/load、`dramaRng` 经 `CloneRng` `World.cs:165`、`_nextArcId/_nextGrudgeId/_nextIgnitionCheckAt`、对子 cooldown 表、VariedSelector 计数表）、`DramaState.Profiles`。任一漏拷即 R-NF2 静默漂移、只在长跑暴露 → Clone 清单须逐项对账 + 专测。

## 7. 确定性整数

- **RNG 流隔离**：`WorldFactory.CreateInitial` 新增 `dramaRng = root.Split(5)`。**不消费保留的 `_domainRng=root.Split(2)`**（保其"reserved"注释语义干净 `World.cs:27`），也不改 `Split(1..4)` 顺序 → v1.0 既有 RNG 拓扑零变化。INV-RNG：决策阶段（RuleBrain）绝不碰 dramaRng；戏剧阶段绝不碰 brain/domain/spawn 流。
- **黄金轨迹兼容**：`DeterminismTests` 比对的是 `Chronicle.Lines` 文本（`DeterminismTests.cs:8-13`）。**戏剧系统对空库/`MaxConcurrentArcs=0`/无候选时严格 no-op（连 dramaRng 都不消费、不产 Chronicle 行）** → 既有 38 测试逐字节不变。一旦弧真点火产事件，含戏剧的种子轨迹会变——这是预期的，新增 INV-DET-1/2-Drama 专测覆盖含弧种子（不动既有 R-NF1/R-NF2 的旧种子断言）。
- **整数纯度**：`Jianghu.Drama` 全程禁 float/double，整数百分比定点 `x*pct/100`，加权 total 用 `long` 防溢出。**注意**：现 `CorePurityTests` 仅检查引用程序集（`CorePurityTests.cs:8-13`），**尚无 IL float 扫描**——本设计需新增该能力，不能假装复用。
- **NextLong 依赖**：`IRandom` 仅有 `NextInt(int)`（`IRandom.cs:7`），无 `NextLong`。若加权 total 可能 > int.MaxValue，须在 `RandomExtensions`（现仅 `Shuffle` `RandomExtensions.cs:8`）新增确定性 `NextLong(this IRandom, long maxExclusive)`（两段 32 位拼接 + 拒绝采样），不改 `IRandom` 接口。v1.2.0 权重设计上 < 2^31，可暂用 `NextInt`，但须 `MaxArcWeightSum` 上界守门。

## 8. 测试策略

新增约 14 个测试，分层对应 gate：
- **INV-DET-1（R-NF1 扩）**：含戏剧推进的种子，同种子两跑 Chronicle 逐字节一致。
- **INV-DET-2（R-NF2 扩，最重要回归门）**：预置强恩怨 fixture（保证点火复仇弧），tick=N Clone 续跑 vs 完整跑逐字节一致（复制 `DeterminismTests.Snapshot_continue_equals_uninterrupted` 结构 `DeterminismTests.cs:20-33`）。
- **既有 38 测试不变**：空库 no-op 证明（Step 5/6 gate 前置）。
- **INV-FLOAT**：新增 IL 扫描，`Jianghu.Drama` 命名空间零浮点。
- **INV-CAP**：ActiveArcs ≤ MaxConcurrentArcs；每角色弧 ≤ MaxArcsPerCharacter；对子 cooldown 内不二次点火；恩怨强度恒 [0,GrudgeCap]、继承单调不增。
- **INV-PERF（spy 计数）**：N=300 角色但仅 2 条恩怨，跑 1000 Advance，断言 `FindIgnitions` 恩怨遍历 = O(强恩怨数) 而非 O(N)——证明不每 tick 全扫。
- **INV-STATE-MACHINE**：弧只走合法转移，非法转移抛异常。
- **INV-CHAIN（验收）**：预置强恩怨，跑足够 tick，Chronicle 出现 `ArcIgnited → RevengeConsummated → GrudgeInherited → 第二条 ArcIgnited` 跨代链并最终全收束。
- **INV-NO-DEADLOCK**：Showdown 超时强制结算，弧不永久卡。
- **WeightedPick / NextLong**：边际频率 + 拒绝采样 + long 不溢出 + 确定性。
- **VariedSelector**：Note/Usage/UsageMultiplier/Clone 计数正确（Step 0）。

**回滚策略**：`_drama` 可空 + `MaxConcurrentArcs=0` 严格 no-op → 任一步破黄金轨迹即关 drama 退回纯 v1.0，38 测试为安全网。`Project/Chronicle` 改 Project（最高危）必须先证空库轨迹不变再继续。

## 冲突裁决（conflictsResolved）

- [最高优先·唯一裁决] 戏剧引擎与 RuleBrain 的关系：四镜分歧（lens 1 SetGoalEff 间接+v1.2.1 加趋仇效用项；lens 3 新增 ActionType.Vendetta+改 RuleBrain.Evaluate 加 case；lens 4 D1 独立事件流不改 RuleBrain + D2 Goal 覆写，并自问是否用 RevengeBiasBrain 装饰器）。裁决：戏剧=独立事件流序列器，v1.2.0 零改 RuleBrain（保 RuleBrainTests 不变）。仅经两条既有通道间接影响：(a) 覆写 Character.Goal=Advance 触发 RuleBrain.cs:47 既有 1500 修炼权重；(b) 镜像上限内负 Relations 触发 RuleBrain.cs:64 既有 notFoe(affinity≤-50→-500) 项偏好切磋仇人。否决 ActionType.Vendetta（污染动作集+改 RuleBrain+需致死路径）、否决 GoalKind.Revenge<id>（Goal record 无 target slot，Goal.cs:4，扩它破 v1.0 测试）、否决 RevengeBiasBrain 装饰器（v1.2.0 不需要，徒增 Clone/确定性面）。依据：RuleBrain.cs 实读证实 Goal+Affinity 通道已足够驱动'自发挥刀'。
- 致死/灭门路径：lens 1/2/3 假设 SpawnHeir/CharacterDied(Cause==Slain) 链式孵化，但源码 CharacterDied 仅 (Id,Age)（DomainEvent.cs:12）、死亡只由 Lifecycle.Tick 的 Age>=Lifespan（Lifecycle.cs:24-29），无任何 slain 语义。裁决：采纳 lens 4 D9——v1.2.0 Showdown 非致死（重创=关系崩坏到-100 + 恩怨化解/转移），杀人路径留 v1.2.1。理由：引入致死会触碰 Lifecycle/人口下界 PopulationLow(触发 MaybeSpawn 暴涌)/spawn，成倍放大确定性风险。
- 恩怨载体：lens 2 主张复用 Relations 负值（option A，Grudge=-Affinity）零 schema 变更；lens 3/4 主张独立 GrudgeLedger。裁决：独立 GrudgeLedger 为真相源（lens 3/4），因 SparAction.cs:30 日常切磋累积 -4 会误把口角当灭门(lens 2 自承此风险)；但保留 lens 2 洞见——镜像一笔上限内负 Relations 驱动 RuleBrain notFoe，取两者之长。
- RNG 流：lens 1/4 主张新增 root.Split(5)；lens 2/3 主张消费保留的 _domainRng(=root.Split(2))。裁决：root.Split(5)（lens 1/4）。源码证实 _domainRng 是 Split(2) 且被传入 World 但从未读取（World.cs:27,36）；新增 Split(5) 不扰 Split(1..4) 顺序，且保 _domainRng 的'reserved'注释语义干净、可追。两方案对空库 no-op 时都保黄金轨迹，但 Split(5) 语义更清晰。
- 复仇阶段数：lens 1 用 6 态（含独立 Escape/Building）；lens 3/4 压缩为 5 态（无独立逃脱）。裁决：5 态（Victimized→BuildUp→Hunting→Showdown→Resolved/Abandoned），因 v1.2.0 非致死、受害者多在场，逃脱并入受害，减少状态机魔数与卡态面。
- 张力曲线/Façade salience：lens 1 重度依赖整数张力曲线选 beat（范式 C）；lens 4 明确推迟到 v1.2.1（首刀纯整数加权 F/G 够用）。裁决：采纳 lens 4——v1.2.0 用纯整数加权抽取 + 5 态机本身提供弧形，不引入张力曲线（避免一批未标定魔数 RevengeCurve/各 TensionEffect 导致卡态/秒收束），salience 排序留 v1.2.1 多母题并行时再加。

## 风险（risks）

- 确定性回归（最高危）：drama 介入 Advance 改变 Chronicle/Memory 投影时序，若 RNG 流隔离或 Clone 深拷贝有疏漏，R-NF1/R-NF2（DeterminismTests 比对 Chronicle.Lines）破裂且既有 38 测试黄金轨迹漂移。缓解：dramaRng=Split(5) 不碰既有流、空库严格 no-op（连 Split 都不调）、Step 6/8 强制先写 INV-DET 专测红再实现、Clone 清单逐项对账。
- Clone 漏拷 = R-NF2 静默破坏：drama 新增 7+ 类续跑敏感态（GrudgeLedger/ActiveArcs/DramaScheduler/dramaRng/3 个计数器/对子cooldown表/VariedSelector计数/DramaProfile），World.Clone 现完全未含（World.cs:148-163），任一漏拷只在长跑暴露。缓解：Clone 清单逐项 + 含活跃弧的专测。
- Goal 覆写副作用回流：弧改 Character.Goal 影响 RuleBrain 后续所有决策，弧结束须还原原 Goal 否则角色永久卡复仇态；多弧抢同一角色 Goal。缓解：MaxArcsPerCharacter=1 + Goal 覆写/还原往返测试。
- Showdown 死锁：复仇者靠 RuleBrain 自发靠近仇人，若仇人持续游历躲开，弧永卡 Showdown→并发槽耗尽→后续弧无法点火。缓解：ShowdownTimeout 超时强制结算（狭路相逢）+ INV-NO-DEADLOCK 测。
- 恩怨繁殖/继承失控：跨代链若衰减或上限设松，灭门→继承→再继承膨胀拖垮 Ledger/Clone 性能。缓解：InheritDecayPct 整数衰减 + MaxGeneration=3 封顶 + MaxConcurrentArcs + 对子 cooldown + 行为层活跃弧数监控。
- 魔数标定：GrudgeIgniteThreshold/各 stage Delay/GrowthNeeded/EscapeRatioPct/MaxConcurrentArcs 全是初值，不当会导致弧卡态或秒收束、或'全员复仇'过载。缓解：集中 LimitsConfig 便于扫参；MinFiredInState/ShowdownTimeout 兜底；建议独立标定阶段(类比 calibration init)。
- 相遇密度依赖：Showdown 需复仇者与仇人同节点，v1.0 仅 3 节点(WorldFactory.cs:26-28)小规模，可能长期不相遇。缓解：Goal=Advance/Hunting 引导 + 超时强制结算；根治需 v1.0 相遇密度同调。
- 权重整数截断/溢出：乘子整数定点 x*mul/100 在小值区截断为 0 致候选全空/WeightedPick 除零；连乘可能溢出 int。缓解：每步 clamp + 兜底 w>=1 + MaxArcWeightSum 上界 + 极端乘子组合测试。
- CorePurity 浮点扫描尚不存在：现 CorePurityTests 仅查引用程序集，无 IL float 扫描，drama 误引浮点不会被现有测试拦住。缓解：Step 1 起新增 IL float 扫描覆盖 Jianghu.Drama 命名空间。

## 强项（strengths）

- 四镜对核心架构高度收敛——戏剧=独立事件流序列器、挂 Advance 末尾 MaybeSpawn 前、弧不进 Scheduler、DomainEvent 单源、空库 no-op 回退——综合后裁决面小、争议集中在可清晰仲裁的 4 点，plan 可落地性强。
- 与 v1.0 源码契合度经实读验证：Goal 覆写 + 负 Relations 镜像驱动 RuleBrain 既有 notFoe/Advance 权重，是零改 RuleBrain 即可'自发挥刀'的真实通路（RuleBrain.cs:44-64 实证），非臆测。
- 确定性硬约束全程在控：新增 Split(5) 不扰 Split(1..4)、空库 no-op 保黄金轨迹、Clone 清单逐项对账、Dictionary 仅索引不参与裁决——每条都对应 v1.0 既有模式（World.AtNode 的 Sort、CloneRng、Scheduler.Snapshot）。
- walking-skeleton 聚焦一条深线（恩怨/复仇含师承/血缘跨代继承），避鬼谷八荒广而浅；同一声明式 schema 已证可扩隐藏血脉/奇遇/情仇（范式 L/M/H）只加数据不改引擎。
- '随机有上限'四件套全整数（cooldown/加权抽取/并发上限/强度门控）集中 LimitsConfig，呼应 v1.0 既有上限集中纪律，可扫参可验证。
- TDD 竖切：11+1 步每步独立 gate、错误路径（无候选/仇人先死/超时/继承触底/权重全0）均有显式裁决，回滚策略(_drama 可空 + MaxConcurrentArcs=0 no-op)以 38 测试为安全网。

## 开放问题（openQuestions）

- 复仇 Showdown 致死在 v1.2.0 是否接通？本设计裁定非致死（重创+恩怨化解/转移），但'灭门→复仇'母题本质需致死才有张力。若产品要求首刀即致死，需新增 death-cause 机制 + 处理仇人死后弧立即 Resolved/死者恩怨账本清理/调度器移除/人口下界联动——建议作为独立评估，本刀默认非致死。
- 预置冤孽来源：v1.2.0 用 WorldFactory seed 控制预置 1~2 对强恩怨（确定可观测）vs 让恩怨从 SparAction 既有 RelationChanged 自然累积到阈值（更涌现但首刀可能长跑不出戏）。需定'多快看到第一条复仇弧'的产品期望。
- DramaProfile 师徒/血缘边的来源：v1.2.0 仅靠 WorldFactory 预注入 + 可选涌现挂师（Lifecycle.MaybeSpawn 用 spawnRng），真正婚配生子留 v1.2.1。确认 v1.2.0 血缘传递只走预注入的既有父子边、世代传递主要靠再传弟子，是否接受？
- 弧产 DomainEvent 是否进 R-NF1 黄金 snapshot 基线：DeterminismTests 现比对 Chronicle.Lines 全文。确认含戏剧的新种子用新 INV-DET 专测、旧种子断言不动；谁负责'含弧种子'的基线固化。
- Memory 投影粒度：哪些 drama 事件投影进参与者 Memory？MemoryStore cap=16 按 |Valence| 淘汰（MemoryStore.cs），全投影会挤占；需定灭门/搏斗/大仇得报为高 valence 必留、ArcStageEntered 等不投影。
- DramaLimits 并入 LimitsConfig（一处验证、符合集中原则）还是独立 record（避免 v1.0 record 膨胀）？倾向并入，待确认。
- OnDomainEvent/Pump 内孵化的新弧/新继承在同一 Advance 内是否立即参与后续 pulse，还是延后到下一 Advance？影响确定性时序与单步成本，需明确语义（本设计默认 Pump 内推进相 A 处理本批、点火相 C 产的新弧延后到下次 Pump）。
