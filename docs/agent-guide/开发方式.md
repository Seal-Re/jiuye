# 开发方式

> 本项目的开发纪律。不是可选建议——是红线 A.3/A.5/A.7/A.9 的具体落地方式。  
> 新 agent 在写任何代码前先读这里。

---

## 核心仪式：TDD（测试驱动）

每个功能单元的开发顺序是固定的：

```
① 先写差分测试（RED — 先失败）
② 写实现让测试绿
③ 跑全量套件确认无回归
④ 提交（附测试绿计数 + git sha）
```

**"差分测试"的定义**：不是"结果 ≠ 0"，而是"装备 vs 剥离产生可观测的不同结果 + 语义正确"。这对应 §10 覆盖账的第1+2条。只会非零的测试抓不住"效果挂上了但不生效"的 bug。

**全量套件必须跑**：每次提交前 `dotnet test --nologo`，不能只跑单路测试。当前基线 1051 绿，不退。

---

## 证据门（A.3）

标 `done` 的三件事缺一不可：

| 要素 | 怎么给 | 示例 |
|---|---|---|
| 全量测试绿 | 贴测试计数 | `1051绿/0失败` |
| git 已提交 | 贴 sha | `ea1c3f9` |
| 产物存在 | 文件路径可验证 | `design/gdd/combat-system.md` |

**主控独立核验（不信 subagent 自报）**：subagent 说"已完成"，主控自己跑 `dotnet test` + `git log` 核 sha，才算过。

## WIP 纪律（A.5）

doing 状态同时 ≤ 2。超限的后果：
- 禁止开新任务
- 已在做的先转 blocked（标阻塞点+原因）或做完

当前 sprint-status.yaml 是真相：每次开会话先读它。

## 分层实现：Core 整数 → Unity 浮点

这是本项目最重要的架构规则（对应 B.2）：

```
Jianghu.Core（netstandard2.1）= 纯整数确定性
  ↓ 零改写进 Unity（引用）
Unity 宿主层 = 可用浮点/实时帧/玩家输入
```

**任何放入 `Jianghu.Cultivation` 命名空间的代码禁浮点**。战斗 q_core 模型用整数 clamp，不用 `exp()`。IL 扫描在测试里守。

如果某功能需要浮点/实时——放 Unity 宿主层，不放 Core。

## Modules 工厂（B.9）

所有战斗效果模块用工厂构造，**禁裸写 `new EffectOp(七参)`**：

```csharp
// ✅ 正确：工厂构造，自动保证 ratio-Kind Amount2≥1 / Trigger / Rarity
Modules.PenFromResource("swordWill", 4)
Modules.CounterMul("evil", 3, note: "破阴邪×3")
Modules.Control("soulLock", 1)
Modules.Drain("itemTier", 2)
Modules.Special("luobao")

// ❌ 错误：裸构造，容易漏 Amount2 导致注册期 AssertValid 抛
new EffectOp(EffectOpKind.PenFromResource, "swordWill", 4, "note")
```

新算子的流程：
1. 在 `Modules.cs` 加一个工厂方法
2. 在 `ModuleResolver.cs` 加对应 case 分支
3. 不改既有积木

## Special handler 纪律（B.9 扩展）

唯一档签名机制走 `SpecialModuleRegistry`：

```csharp
// 注册（SpecialModule.cs 静态构造）
Register(new LuobaoModule());

// handler 必须满足：
// ✓ 纯整数（IL 扫描）
// ✓ 副作用经 ctx.ApplyResource（不直写 dict）
// ✓ 不读 daoHeart/innerDemon（B.5）
// ✓ 无 RNG
```

## Skill 调度（A.9）

会话开始时 session-start.sh 会输出推荐 skill，按照推荐使用。主要场景：

| 场景 | 推荐 skill |
|---|---|
| 查当前 sprint 进度 | `/sprint-status` |
| 实现 story | `/dev-story`（需 combat GDD + ADR 就位）|
| 战斗/平衡验证 | `/balance-check` |
| 提交前 gate | `/smoke-check` |
| 补设计文档 | `/design-system [系统名]` |
| 检查范围蔓延 | `/scope-check` |

大方向决策交用户（A.1），工具性/核验性操作自主调度。

---

## 提交规范

Conventional Commits 格式：

```
feat(scope): 简短描述
fix(scope): 
chore(scope): 
docs(scope): 
design(scope): 
```

必须引用 story/task id（在 commit body），贴测试计数 + sha 证据。
`Co-Authored-By` 行要加。

---

## 坏味道清单（见到就警惕）

- `"已完成"` 后面没有 sha 和测试计数 → 不算 done
- 开了第 3 个 doing 状态的任务 → 违反 WIP ≤ 2
- 新 EffectOp 直接裸写七参 → 违反 B.9
- 战斗计算里出现 `Math.Exp` / `(float)` / `(double)` → 违反 B.2
- `daoHeart` 进了 `EffectivePower` 计算 → 违反 B.5
- 任务延后但 TASKS.md/epic 没标 `deferred` → 违反 A.8
