# 红线速查

> 给新 agent/新会话用。**规则 + 原因** 都在这里——只知道规则而不知道原因，迟早触雷。
> 完整原文在 `CLAUDE.md`；这里是快速参考 + 每条的 WHY。

---

## A. 流程红线

### A.1 每回必给下一步选项
**规则**：回复结尾给 2-4 个方向 + 推荐 + 一句理由，绝不"任务完成然后停止"。  
**为什么**：项目曾多次发生 agent 自作主张跳大方向，或做完一阶段就静默停止让用户茫然。决策权在用户，agent 的职责是把选项铺开，不是替用户定。

### A.2 真相源 = production/
**规则**：唯一真相 = `production/epics/` + `sprint-status.yaml` + `stage.txt`。TASKS.md 是派生指针，不持有真相。  
**为什么**：历史上 TASKS.md 和口头记录多次出现分叉，导致虚报/漏做。单一真相源消除歧义——冲突时以 `production/` 为准。

### A.3 完成 = 机器证据
**规则**：`done` 前必须：①全量测试绿（贴计数）②git 已提交（贴 sha）③产物存在。不信 subagent 自报，主控独立核验。  
**为什么**：subagent 有"完成话术"倾向——用自然语言说"已完成"但代码其实没过。本项目曾出现 batch1 已完成但台账仍写"待启"的断链。机器证据是唯一可信依据。

### A.4 状态枚举强制
**规则**：任务状态 ∈ `{todo, doing, review, done, blocked}`。done 的 DoD 清单必须全勾。blocked 必记阻塞点+原因+依赖。  
**为什么**：无状态或模糊状态导致无法自动化审计。A1.4 曾被静默 defer 而不标 blocked，发现时已影响后续任务链。

### A.5 WIP ≤ 2
**规则**：doing 状态同时不超过 2 个。超限禁开新任务，先做完或转 blocked。  
**为什么**：人和 agent 都有上下文切换成本。多任务并行导致每个都做一半，验证成本翻倍，bug 互相掩盖。"做完再开"是吞吐量最优的简单规则。

### A.6 定期审计
**规则**：每阶段边界/每 5-8 步，对账 production/ vs 实际。  
**为什么**：本项目发现过 done 标了但测试没跑、doing 挂了三天没动的情况。审计是防止台账腐化的唯一手段。

### A.7 单点 fix ≤ 3 次
**规则**：同一个 bug fix 尝试不超过 3 次，超限标 `known_issue` 上报人审。  
**为什么**：死循环 fix 是时间黑洞。3 次还没解决说明根因判断有误，继续猜测解法只会越改越乱。

### A.8 诚实标 defer
**规则**：任何计划内任务延后，必须显式标 `deferred` + 依赖，不得静默移走。  
**为什么**：A1.4（辅助路 UT 重锚）曾被静默 defer 到"标定阶段"——台账看不出来，后来发现时发现有依赖链断掉。静默移走 = 假装没发生 = 债务积累。

### A.9 主动调度 skill
**规则**：不等指示，主动扫 `.claude/skills/` 匹配当下工作并 invoke。  
**为什么**：红线 A1 要求给选项，A9 要求主动用工具——合力保证 agent 既不自作主张（大方向交用户）又不消极等待（工具性操作自发起）。session-start.sh 每次启动都会提示推荐 skill，参照执行。

## B. 技术红线

### B.1 联网只 headless
**规则**：只用 WebSearch/WebFetch API，禁止前台浏览器窗口（Playwright browser_navigate）。  
**为什么**：前台浏览器在 CI/headless 环境下会挂起，且 Playwright 曾触发过 prompt-injection（网页内容喂给模型）。

### B.2 整数确定性（最重要的技术红线）
**规则**：`Jianghu.Cultivation` 命名空间禁浮点。同种子逐字节复现。新随机流升 World 字段+进 Clone。IL 扫描守。  
**为什么**：jiuye 是涌现模拟，可复现性是核心保障——同一种子必须在任何机器/任何时间产生完全相同的结果。浮点有平台差异（x86/ARM/IL2CPP）。一旦破坏，调试成本极高且无法在 Unity 端验证。

### B.3 off 逐字节
**规则**：cultivation-off 默认路径必须与 v1.0 逐字节一致。改 v1.0 文件后必验。  
**为什么**：off 路径是兼容性保障。有 38+ 测试 + sha256 实证保护。一旦偏移意味着历史存档无法重放。

### B.4 不舍弃任何路径
**规则**：21 路全入册，加路=数据行，none dropped。  
**为什么**：每一路都代表一种"武侠职业"设计，是世界观完整性的体现。漏路=世界观破洞，且后续平衡标定依赖全 21 路数据。

### B.5 道心解耦
**规则**：`daoHeart`/`innerDemon` 禁进 `EffectivePower`（仅突破劫 ResistTerms）。  
**为什么**：道心/心魔是 A.2 阶段才接的"内在状态"，混入战斗 power 计算会破坏"战力=外在修为"的设计边界，且让平衡标定无法控制（内心状态随 NPC 行为变化，战斗结果就不稳定）。

### B.6 Σ=80 仅生成期
**规则**：角色四维总和=80 只在生成期固定，后续功法不污染四维。侧表纪律。  
**为什么**：Σ=80 是平衡锚——保证初始属性分布可控。功法改四维会让后期角色与初始假设脱节，破坏 INV-CROSS 标定。

### B.7 subagent 模型分层
**规则**：实现/核验/平衡裁决类 subagent 用 Opus 4.8。CCGS 诊断/格式类 skill 自带廉价 tier 保留（产物派生+主控复核）。  
**为什么**：廉价模型产出代码/裁决可能引入难以发现的细微错误。主控永远用最强模型，派生性操作可降档——关键是区分"产出代码"和"产出建议"。

### B.8 可视化分轨
**规则**：游戏世界=像素(Pillow)；UI=精细化古风(SVG/HTML-CSS)。程序化只做变换/派生/拼装，原创基础件交手绘/AI。  
**为什么**：两轨审美完全不同，混用会两边都不伦不类。像素做不出水墨感，矢量做不出像素锐利感。分轨也让 AI 出图工具（SunshineFlow）专注静态件，不越界做动画角色。

### B.9 模块化·可插拔
**规则**：战斗效果/功法/战技经 `Modules` 工厂构造，唯一档经 `SpecialModuleRegistry`。禁裸写 `new EffectOp(七参)`。  
**为什么**：21 路 × 每路 N 招的战技定义量极大，裸构造容易漏参（Amount2/Trigger/Rarity）。集中工厂=集中校验=一处改动覆盖所有路。B.9 落地后发现的 bug 比原来少了一个数量级。
