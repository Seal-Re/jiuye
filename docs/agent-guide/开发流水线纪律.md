# 开发流水线纪律（阶段·里程碑·配置驱动）

> 本项目的阶段性开发纪律。据用户「轮次提示词」指令（2026-07-02）+ 联网检索工业化游戏开发规范 + 本项目确定性红线三方对账确立。
> **不是可选建议——是防「过度开发/过早雕琢」的流程红线**，与 A.1（大方向交用户）/A.5（WIP≤2）/A.8（诚实 defer）同级。
> 新 agent 在选择"做什么"之前先读这里：**先确认当前阶段，再确认这件事属不属于当前阶段该做的**。

---

## 0. 一句话纪律

**严禁跳过阶段直接做最终品质的雕琢。** 前一阶段未验收，不进入下一阶段。尤其：**平衡数值的详细收敛调优、美术打磨、UI 美化，不在 Pre-Production 做**（属 Production→Alpha 及以后）。

> 本项目已犯的典型过度开发：`balance-003` C1 收敛闸门（同 UT 胜率 [40,60]% 硬调优）在无 Vertical Slice、核心循环未验证有趣时被反复尝试——这是把 Alpha 期的数值收敛提前到了 Pre-Production。已 defer 到 Alpha（见 `production/epics/balance-cross/story-003-*.md`）。

## 1. 阶段序列 = CCGS 七阶（AAA 四阶术语映射其上）

用户指令用的是经典 AAA 四阶术语（Prototype / Vertical Slice / Alpha / Beta）。本项目跑在 CCGS 七阶骨架上（`/gate-check` 强制准出判据）。二者是**同一条流水线的两套词汇**，映射如下——**不新造平行结构，一律走 CCGS 闸门**：

| AAA 术语（指令） | CCGS 阶段（`/gate-check`） | 准出判据（业界共识，见 §4 源） |
|---|---|---|
| Concept | Concept → Systems Design | 概念文档 + 系统拆解 |
| Prototype | Technical Setup → **Pre-Production** | 占位/粗数值验证**核心机制可行**（throwaway，非最终品质） |
| **Vertical Slice** | **Pre-Production → Production 闸门** | **生产品质**的端到端极小切片，验证"做得成 + 长这样"（去风险 greenlight） |
| **Alpha** | **Production**（feature complete） | 所有系统框架实现，可玩含全主要功能；**数值收敛/平衡冻结的重点期** |
| Beta | Polish | feature + asset complete，只修 bug，功能/资产/代码冻结 |
| Gold/Release | Release | 可交付量产 |

**当前阶段**：`production/stage.txt` = **Pre-Production**（2026-07-02 订正，原误标 Production）。
订正依据（三方证据）：① CCGS `Pre-Production → Production` 闸门要求**已验证的 Vertical Slice**——本项目无（圆桌 Seat C 证）；② `project-stage-detect` 的 `src/ 10+ 文件 → Production` 是纯文件数启发式，从不核真闸门，误标源于此；③ 用户指令"平衡推迟到 Alpha"——若真在 Production，平衡本就该做，矛盾反证阶段错标。

## 2. 三条论断的修正（指令原文 → 对账后纪律）

用户指令明确要求"结合联网+本地+推断**修上述论断**"。以下三条经查证**方向对但过绝对**，据本项目性质修正：

### 2.1 「原型禁一切数值/平衡计算」→ 修正为「原型用占位/粗数值验证机制，不投入最终平衡调优」

业界原型定义是"占位资产、聚焦核心验证、省略最终打磨/健壮性"，**非禁一切数值**。占位数值是原型常态。
**本项目尤其要调整**：九野是**整数确定性涌现模拟**，"核心玩法"本身就是数值与关系的涌现——原型阶段回避数值 = 回避了要验证的东西。
- ✅ 该做：用粗数值/占位曲线验证"涌现是否发生、是否有趣"（如 21 路能否在世界里长出恩怨/门派/复仇）。
- ❌ 不该做：同 UT 胜率 [40,60]% 这种**收敛级数值调优**（属 Alpha）。

### 2.2 「平衡推迟到 Alpha 及以后」→ 修正为「平衡拆 Viability/Fairness 两类：Viability 前置、Fairness 推迟 Alpha」

业界共识：game balance 是"iterative process, during development and also afterwards, needs many iterations"，贯穿 Production→Alpha→Beta→post-launch。**但"平衡"一词太粗，混了两类本质不同的东西**（2026-07-03 用户指令订正）：

| 类别 | 定义 | 阶段 | 例 |
|---|---|---|---|
| **Viability（可运转性）** | 实体能否走完成长线——生命周期流转正常、破境率/寿命消耗匹配、基础数值满足自然成长规律 | **Pre-Production 核心前置**（内核是否 work） | 破境率（600 步大部分实体应能破境 ≥1 次）、寿命 vs 修为累积速度 |
| **Fairness（公平性）** | 流派间强度对等 | **推迟 Alpha** | 21 路同 UT [40,60]% 胜率（balance-003）、RealmMultipliers 重校准 |

**为什么必须拆**：上层剧情（Drama）强依赖实体核心属性（战力/寿命/状态）。若 Viability 未通（如战力涨不动→破境卡住），复仇状态机会因"数值不满足门控"陷入死锁——**这正是本项目实测的"闭关到老死"**。所以 Viability 不是"雕琢"，是"内核是否 work"，**Pre-Production 必须先跑通**；把它 defer 到 Alpha 是错的。
- ✅ Viability 调平：Pre-Production 就做（破境率/寿命/生命周期流转）。
- ✅ 平衡**观测**（`BalanceMatrixDump`、`/balance-check`、advisory gate）：全程可跑。
- ⏸️ Fairness **收敛调优**（advisory [35,65]→硬闸门 [40,60]、21 路 RealmMultipliers 重校准）：Alpha。
- 判据：**Viability + 观测 gate 全程/前置；Fairness 收敛 gate 到 Alpha 才拧硬。**

> ⚠️ 修订记录：本节初版（2026-07-02）把"破境率低"一股脑归 Alpha，2026-07-03 用户指令订正——破境率是 Viability，属 Pre-Production 前置，不该 defer。

### 2.3 「必须用 JSON 配置」→ 修正为「数值外置是共识，但格式不必 JSON、数值绝不浮点、配置变更须版本锁定」

数据驱动（外置数值、解耦逻辑）是行业共识，利于迭代/modding（Doom WAD 范式）。**但"必须 JSON/浮点"对确定性项目错误**：
- 确定性 lockstep/replay 要求**逐位复现**；**浮点跨平台/编译器/OS 不可复现**（AoE 1500-archers、gafferongames 实证）——这正是红线 **B.2 禁浮点 + IL 扫描** 的由来。
- **本项目配置纪律**：① 外置数值须**整数/定点**（非浮点 JSON）；② 配置变更 = 破坏逐字节复现，等价 schema 迁移，须**显式版本锁定**（承 RngStreamIds 冻结·append-only 精神）；③ 权衡 schema 脆性——确定性项目外置配置代价**高于**普通游戏，宜"结构可读 + 数值整数化 + 严格版本门"。
- 现状注记：当前数值多在 `paths/*.cs` 代码内（`RealmCurveDef` 等）。迁移到外置整数配置是**可选的 Alpha 期重构**，不是 Pre-Production 的当务之急；迁移时必过 off 逐字节回归。

### 2.4 开发顺序：内核绝对前置于剧情（2026-07-03 用户指令）

Drama 是**架设在基础运行逻辑之上的上层应用**。恩怨状态机（立誓/寻仇/手刃）**强依赖实体核心属性**（战力/寿命/当前状态）。若基础内核（破境概率/战力成长/切磋目标选择）未跑通，上层剧情将因数值不满足门控而**死锁**（实测："闭关到老死"）。故开发顺序：

1. **核心系统**：实体生命周期管理、基础数值流转、基础行为树（打坐/切磋）。
2. **Viability 调平**：调平破境率与寿命消耗，确保大部分实体能正常经历完整成长线（见 §2.2）。
3. **剧情模块**：在此基础上接恩怨状态机，验证"跨代复仇""手刃仇人"等边界。

> 注：`Vendetta Urge`（drama 停滞超时强制推进，`2f85281`）虽属 Step 3，但作为**状态机防御性兜底**（死锁安全网）保留——治标合理，但**治本在 Step 2**（内核数值调平后，门控自然满足，不再依赖强制推进）。

### 2.5 表现层（Godot）接入闸口（2026-07-03 用户指令；引擎 Unity→Godot 4.x .NET，ADR-0004）

**采用逻辑与表现分离（Model/View）**：`Jianghu.Core`（netstandard2.1）= Model（纯逻辑数据层）；Godot 4.x .NET = View（只读状态、渲染 UI，如江湖记事本/属性面板；边界见 ADR-0004 §9）。
- **当前阶段（Headless Simulation）**：维持现状——纯代码控制台跑数百 Tick 步进，纯靠数据输入 + 日志输出验证核心状态机与数值闭环。
- **接入触发条件（唯一闸口）**：**当无头数据日志证明核心机制无死锁**（如：复仇逻辑能正常走到 Showdown 节点，且基础战力/破境分布符合预期分布）时，方可接 Godot。
- **接入纪律（ADR-0004）**：Godot 节点只读 Core（`WorldBridge`→`[Signal]`）、回写只经命令端口、`Godot.*` 不进 Core、`World.Advance` 走固定时间步累加器（`delta` 不进 Core）。**2D 等距地图当前不开发**，仅文档预留 iso 坐标系红线。
- **为何暂不接**：过早接入引擎渲染/UI 绑定会拖慢底层逻辑迭代速度。先闭环 Model，再叠 View。

## 3. 里程碑构建 + 目录结构（已基本合规，注记）

**里程碑构建**：业界每阶段产出**可独立运行的 build**（First Playable/Alpha/Beta/Gold 皆是具体可交付构建，非纯文档）。本项目 CLI (`dotnet run --project src/Jianghu.Cli`) 即当前可运行 build，配 1051 测试绿 + git sha（红线 A.3 证据门）。**纪律**：每过一个 CCGS 闸门，确认 CLI build 可跑 + 全量绿 + 打 tag/记 sha。

**目录结构**：本项目三程序集布局（`Jianghu.Core` netstandard2.1 纯逻辑 / `Jianghu.Cli` net8.0 薄壳 / `tests/` 含 Determinism 专轨）**已符合** .NET (`src/`+`tests/`) 惯例 + "可复用 Core + 后期引擎宿主"分层。BannedApiAnalyzers 守 Core 无 `Godot.*`/`UnityEngine.*`/`System.Random`——即"零改写进 Godot"的前提。**纪律**：模拟逻辑只进 Core；宿主（CLI/后期 Godot）只做解析/渲染/输入，不含逻辑。
