> 给新 agent/新开发者的核心架构知识。读这里比从代码里推断快得多。

## 一句话总结

jiuye 是一个**确定性整数涌现模拟器**：给定种子，NPC 们按各自修炼路线自主行动，江湖自然演化，同种子产出完全相同的历史。

## 核心类型

### World（`Sim/World.cs`）
模拟状态的根容器。持有：
- `IReadOnlyList<Character>` 所有角色
- `Clock`（模拟时钟）
- `Pcg32` 随机流（种子确定）

所有副作用经 `IWorldMutator` 接口（不直写 World）。

### Character（`Model/`）
角色数据：
- `Persona`（姓名/头衔/标签/门派）
- `StatBlock`（Force/Internal/Constitution/Insight，Σ=80 起始）
- `Cultivation`（修炼状态，可 null）

### CultivationState（`Cultivation/`）
角色的修炼当前状态：
- `PathId`：选了哪条路
- `RealmIndex`：当前境界（0..MaxMajor）
- `Resources`：路线专属资源（swordWill/thunderCharge/qixue…）
- `Flags`：二值/整数标志
- ⚠️ `DaoHeart`/`InnerDemon`/`Comprehension`：A.2 道心层才用，**禁进 EffectivePower**（B.5）

### CultivationPathDef（`Cultivation/`）
路线的静态定义（只读数据）：
- `PathId`：唯一标识（如 `sword_immortal`）
- `ArtCategories`：功法类目（每类 ≥4 具名功法）
- `CombatSkills`：战技列表（含 EffectOp 模块）
- `Curve`：境界曲线（倍率/UT映射/名称/阈值）
- `Power`：战力公式项
- `Resources`：路线专属资源定义

## 战斗模块系统（B.9 的落地）

```
战技（CombatSkillDef）
  └── OnUse: EffectOp[]   ← 效果模块列表
           ↓
    ModuleResolver.ApplyOnUse(dmg, op, ctx)
           ↓
    按 EffectOpKind 分支：
      PenFromResource → dmg += res×mul/div
      CounterMul      → 有 tag 时 dmg × num/den（上界3/2）
      DrainResource   → 夺取资源
      Control         → 控场（批4接）
      Dot             → 持续伤（批4接）
      Special         → SpecialModuleRegistry[handlerId].Apply(ctx, op)
      ...
```

**CombatContext** 是 chokepoint：所有资源读写必须经它（`ReadResource`/`ApplyResource`），不直写 dict。

## 21 路修炼路线

每条路 = 一个 `static class XxxPath` 文件，`Def` 属性返回完整 `CultivationPathDef`。

| 属性 | 意义 |
|---|---|
| `SituationalTags` | 属性/形态 tag（`evil`/`ghost`/`melee`…），**非对手 PathId** |
| `AttackDimension` | 攻击维（physical/spirit）|
| `EntryGate` | 入路条件 tag（`sword_root`/`body_root`…）|

当前已结构化的招牌招（batch2/3 完成）：
- 所有 21 路的签名招式已从 `AddPenInteger` 占位迁为 `Modules.*` 结构化模块
- 8 个 Special handler 已实现：落宝/炸阵/金身态/律场/夺舍/夺心/断链/逆演栈
- 剩余 24 个次要占位为显式 defer（需 FULLSTRUCT 底层，A.8 已标注）

## 平衡标定（INV-CROSS）

战力量级锚：
- `HP = pe = PowerEngine.Evaluate(state, stats, def, limits)`
- 同 UT（修为等级）的不同路线，`pe` 同量级
- C1 契约：同 UT 战斗路 1v1 胜率 ∈ [40,60]%

**INV-CROSS 是 balance-cross epic 的实测 gate**，不是设计假设。harness 在 `tests/.../BalanceMatrixDumpTests.cs`。

## 关键接口（扩展点）

### IBrain（`Decide/IBrain.cs`）
```csharp
public interface IBrain
{
    ValueTask<ActionChoice> DecideAsync(DecisionContext ctx, CancellationToken ct);
}
```
**这是 LLM 脑接入的唯一入口**（未来 Epic-LLM）。现在只有 `RuleBrain`（规则 AI）。接入 LLM = 实现此接口，Core 不变。**不要绕过这个接口直接改 World**。

### IWorldMutator（`Sim/World.cs`）
World 的写入接口，所有 Action 通过它改状态（`ApplyResource`/`Move`/`AdjustRelation`等）。不要拿 World 的内部集合直接写。

### IAction（`Actions/IAction.cs`）
战斗/修炼/旅行等所有行动的基接口。新行动 = 实现此接口 + 在 Scheduler 注册。

## off 路径（B.3）

cultivation-off（默认）= 走 legacy `SparAction` 旧公式（`Force×2+Internal+Constitution`）。

- off 路径完全不接触 Cultivation 系统
- 任何改动都不能影响 off 逐字节
- 38+ 测试保护，改 v1.0 文件后必须验

## 其他已接线系统（Drama / Map / Faction）

修炼是核心纵深，但 World 主循环还接了三个 Feature 系统（默认 off，各走独立 RNG 子流，off 逐字节守）：

| 系统 | 命名空间 | 关键类型 | RNG 流 | CLI 开关 | 主循环接入点 |
|---|---|---|---|---|---|
| 戏剧引擎 | `Jianghu.Drama` | `DramaDirector`(Pump)、`GrudgeLedger`(恩怨账本)、`RevengeArc`(5 态机)、`IgnitionScanner`、`StoryletSelector` | `Split(6)` | `--drama` / `--drama-feuds` | `World.Advance` 末尾 `_drama.Pump` |
| 地图 | `Jianghu.Sim` | `WorldMap`、`WorldMapFactory`、`KruskalMstGenerator`、`IGeoQuery` | `Split(7)` | `--map` | `WorldFactory` 建图 + `World` 持有 |
| 门派 | `Jianghu.Sim` | `SectLedger`(Pump)、`SectLedgerFactory`、`IFactionQuery` | `Split(8)` | `--faction` | `World.Advance` 中 `Faction.Pump` |

- **戏剧**：drama-001~013 全落，1062 绿。NPC 自发恩怨→复仇弧→跨代继承（父债子偿）。**RuleBrain 零改动**——drama 仅经覆写 `Goal=Advance` + 镜像负 `Relations` 间接影响决策。
- **接线纪律**：三系统全部进 `World.Clone()` 深拷（漏拷=长跑静默漂移）；off（不开开关）时严格 no-op，连各自 rng 都不消费。
- 详细设计见 `design/gdd/drama-system.md`；状态见 `production/epics/index.md`（drama-engine=Done / map-system=Wired / faction=C.0 Done）。
