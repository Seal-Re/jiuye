## Overview

战斗系统决定两个角色（或多个 NPC）如何分出胜负。它**分两层**：

- **结算层（Core，纯整数确定，headless 可跑）= 当前阶段实现**：速度（speed）决定行动序列（CTB 风），每个行动单位轮到时选一个**操作**（攻/防/控/资源/签名），经 `ModuleResolver` + `SpecialModuleRegistry` 把该操作的 `EffectOp` 模块作用到战斗状态。NPC×NPC 的江湖涌现战斗全在这一层，确定可复现（红线 B.2）。**常规防御（运气/护体真气/招架/结界/闪避概率档）在此层做**。
- **即时层（Unity 宿主，后期）= 当前 Core 不实现**：少数**瞬时反应操作**（闪避/格挡/打断/反弹/夺舍续命/逆演）的"按键时机窗口"是 **Unity 阶段**的玩家体验，**含浮点正态判定的 q 模型放 Unity 宿主层**（不受红线 B.2 约束——B.2 只管 `Jianghu.Cultivation`）。**Core 阶段（现在）即时类操作退化为"概率档/阈值"的纯整数结算**（见 §即时操作的 Core 退化），不引浮点、不引实时帧。玩家按键的真即时手感等 Unity。

> **复审对治（独立 design-review 2026-06-16）**：原稿把"浮点 q 正态模型"放 Core → 撞红线 B.2（`exp()` 浮点）。**修订：即时窗口的浮点 q + 按键时机整个 defer Unity；Core 只做纯整数概率档/阈值**。Core/Unity 不再共用同一浮点数学——Core 用整数退化版（NPC 用），Unity 用浮点窗口版（玩家用），两者是"同一操作的两档保真度"，非同一公式。

核心区分（用户定）：**不是所有防御都即时**。常规防御（运气防御/护体真气/招架值/结界）是**回合内的选择**，在 Core 正常结算；只有"瞬时反应类"（闪/格/打断/反弹/续命/回滚）才有即时窗口。

战斗能做什么由**修了什么功法**决定（功法门控，§14.1）——stat 只调强弱，不自带能力（没轻功不能闪、没内功心法没护体真气）。

## Player Fantasy

- **当前阶段（v1 涌现模拟）**：玩家是"江湖的观察者/编剧"，看 NPC 们按各自功法路数自动厮杀，涌现出"剑仙一击破百、毒修阴蛊缠身、体修硬抗反震、命修逆天改命"的武侠桥段。战斗的爽感来自**门派各异的质感**和**涌现的戏剧性**（A 压制 B 三招，C 濒死夺舍翻盘）。
- **后期阶段（Unity 玩家介入）**：玩家操控一个角色时，常规出招是回合策略（选招、调度资源），但**关键时刻的闪/格/打断是即时按键**——在敌方杀招落下的一瞬精准格挡（铁山靠反震）、看准蓄力读条点穴打断、濒死时电光石火间夺舍续命。**"该出手时凭反应，该谋算时凭策略"**——回合谋略 + 即时反应的复合快感，正是 33 号远征队的精髓移植到武侠。

参考：宝可梦/最终幻想（速度序列）、八方旅人 CTB、33 号远征队（回合骨架 + 即时 parry/dodge 窗口）、北大侠客行（通用武学维 + 门派武功）。

## Detailed Design

### 核心循环（速度序列 + 操作结算）

```
战斗开始：每方 HP = pe（PowerEngine.Evaluate，已校准；同 UT 同量级 = 平衡锚）
          每个单位算 speed（见公式）→ 入行动队列（CTB：行动条/ATB 累积）

行动循环（直到一方 HP≤0 或达回合上限 N）：
  1. 取队列中 行动条最先满 的单位 A（speed 高者更频繁行动）
  2. A 选一个【操作】（NPC 由 AI 决策；玩家由输入）：进攻/防御/控制/资源/签名
  3. 结算该操作的 EffectOp 模块（经 ModuleResolver / SpecialModuleRegistry）
     - 若是【攻击】且目标有【即时反应窗口操作】待命 → 触发窗口判定（见下）
  4. 应用结果（伤害/控制/资源移动/状态），状态进 [Min,Cap] 钳
  5. A 的行动条清空，按 speed 重新累积
胜者 = HP 高者；平 → Tiebreak(CharacterId)
```

> **与 R2 原稿 §4 的差异 + C1 保障机制（复审对治 B2）**：原稿用"3 回合同时结算"消先手偏置（保 C1 同 UT 胜率∈[40,60]%）。本设计改速度序列（快者多动），**会引入先手优势**——这是真实风险，不能用"对称性"一句带过。**C1 保障靠下列 4 条具体机制**（非口号）：
> 1. **速度差封顶**：同 UT 内 speed 差设 cap（`SPEED_SPREAD_CAP`，如±15%）——同 UT 战斗路 speed 不允许拉开到一方碾压行动数。speed 大分化只发生在跨 UT（越级本就该压制）。
> 2. **行动数守恒校正**：回合上限 N 按"双方总行动数配额"定，而非固定回合——快者单次行动更频但每击经 pe 量级锚衰减，**总输出∝pe 不∝行动数**（pe 同 UT 同量级 = 平衡锚，承 INV-CROSS）。
> 3. **无早退末击红利**：HP≤0 即止，但**最后一个行动结算完才判负**（不让先手者靠"抢先清零"白赚——同 R2 原意，移植到序列模型：致死击仍让对方完成其待发的同 tick 反应/已蓄结算）。
> 4. **INV-CROSS 实测校准**：speed/cap/N 是标定旋钮，由 balance-cross epic 跑 1000 场对拍把同 UT 胜率压进 [40,60]%（**实测 gate，非设计假设**；C1 是硬契约，speed 模型若实测破窗则收紧 cap 或回退同时结算）。
>
> **诚实标注**：C1 在速度序列下能否守住 = **balance-cross 标定阶段的实测裁决**。本 GDD 给机制与旋钮，最终由对拍数据定；若 4 条机制仍压不进 [40,60]%，回退 R2 同时结算（保留为 fallback）。

### 即时窗口模型（用户定；分 Core 整数退化版 + Unity 浮点窗口版）

当攻击方发动一次攻击、防御方持有对应的**即时反应操作**（闪/格/打断/反弹/续命）时——

#### Core 阶段（现在实现，纯整数，NPC 用）

**Core 不做时机窗口，退化为"成功度 q ∈ [0,100] 的整数计算"**（无浮点、无实时帧、无 exp）：
```
q_core = clamp( base + 品级×tierGain + 悟性×insGain − 难度penalty + 确定扰动(Pcg32 split), 0, 100 )
```
- 全整数；`确定扰动` 来自 cultivation 对拍 RNG 的 split 流（确定性 split，同种子逐字节复现，红线 B.2/B.3）。**首版可设扰动=0 走纯确定**（承 R2 §4"首版纯确定无 RNG"——若启用 split 扰动需显式升 World 字段+进 Clone，并标注为对 R2 决策的修订）。
- `q_core` 作**乘子**：成功（q 高）→ 该操作完整效果（闪避→入伤×(100−q)/100、格挡→反震、打断→对方蓄招不结算）；q 低→效果衰减。
- 变量表见 §Formulas「即时操作成功度 q_core」。**无 `exp`、无正态、无 Δ/σ**——这些是 Unity 版的事。

#### Unity 阶段（后期，玩家按键真即时手感）

玩家介入时，把 q 升级为**真时机窗口**：按键时刻 vs 窗口中心偏差 Δ → 浮点正态 `q = 100·exp(−Δ²/2σ²)`。
- **浮点 q 放 Unity 宿主层**（C# 但非 `Jianghu.Cultivation` 命名空间，不受 B.2 IL 扫描约束）。
- 窗口宽度 = 操作类型基值 × 功法品级系数（品阶越高窗口越宽/容错越高，见 §Formulas window 表）——**这张表 Unity 用**。
- **同一操作两档保真度**：Core 整数 q_core（NPC/无玩家时）↔ Unity 浮点 q（玩家在场时）。引擎据"是否玩家操控该角色"二选一。NPC 永远走 Core 整数档。

> **复审对治**：原稿"Core/Unity 共用同一浮点窗口数学"违 B.2。**修订：Core 整数 q_core（现在）/ Unity 浮点窗口 q（后期），两档独立**。Core 永不含 exp/正态/实时帧。

### 功法门控防御（承 R2 §14.1）

战斗能做什么 = 修了什么功法定。防御分模式：
- **硬抗**（人人，floor）= 体质削减伤害。
- **护体真气**（需修内功心法）→ 减伤（回合 choice）。
- **闪避**（需修轻功/身法功法）→ 即时窗口（没轻功不能闪）。
- **格挡/反弹**（需修横练/护体类）→ 即时窗口。
- **打断**（需修特定功法：点穴/截拳/雷法）→ 即时窗口。
公共武技池（神行/吐纳护体/基础招）= 散修兜底，靠 pick 得基础能力。

### 操作与模块系统的关系

每个操作 = 一组 `EffectOp` 模块（已实现，batch2/3）。即时窗口判定结果 `q` 作为**乘子**作用到该操作的模块效果上（q=1 满效果，q 低衰减）。NPC 的 q 由 AI 采样，玩家的 q 由按键。**模块系统不变，只在"即时类操作"的结算前插一个 q 乘子**。

## Formulas

### speed（行动频率）

`speed = base + Insight×Ks + 身法功法bonus − 负重penalty`

| 变量 | 类型 | 范围 | 说明 |
|---|---|---|---|
| base | int | 100 | 基础行动速度常数 |
| Insight | int | 1–40 | 悟性（身法/先手主属性，§2 战斗维映射） |
| Ks | int | 2 | 悟性→speed 权重（调优旋钮） |
| 身法功法bonus | int | 0–60 | 修轻功/身法功法的加成（功法门控） |
| 负重penalty | int | 0–40 | 重甲/重兵器减速 |

**行动条**：每"tick"所有单位 `行动条 += speed`；先到 `ACTION_THRESHOLD`(=1000) 者行动，行动后清零。speed 高 → 行动条满得快 → 行动更频繁（CTB 标准）。**纯整数确定**。

### 即时操作成功度 q_core（Core 阶段，纯整数，现在实现）

`q_core = clamp( BASE + tier×TIER_GAIN + Insight×INS_GAIN − atkPress + 扰动, 0, 100 )`

| 变量 | 类型 | 范围 | 说明 |
|---|---|---|---|
| BASE | int | 40 | 基础成功度（调优旋钮） |
| tier | int | 1–5 | 防御功法品级（ArtDef.Tier）——越高越易成功（容错=品级，承用户"按品级调"） |
| TIER_GAIN | int | 8 | 品级→成功度斜率 |
| Insight | int | 1–40 | 悟性（反应/身法主属性） |
| INS_GAIN | int | 1 | 悟性→成功度斜率 |
| atkPress | int | 0–40 | 攻方"压制度"（攻方招式 Tier×k + 速度差）——攻势越猛越难防 |
| 扰动 | int | 0 或 ±N | **首版=0（纯确定，承 R2）**；若启用 = Pcg32 split 流确定取值（升 World 字段+进 Clone，B.2） |

- **输出 q_core∈[0,100]，作整数乘子**：闪避→入伤×(100−q_core)/100；格挡→反震=入伤×q_core/100×reflectRatio；打断→q_core≥INT_THRESHOLD(如60)则对方蓄招不结算（阈值二段，非浮点）。
- **全整数、无 exp、无正态、无实时帧**——红线 B.2 守。NPC×NPC 全走此式，确定可复现（B.3）。

### 即时时机窗口 q（Unity 阶段，浮点，后期，玩家用）

> ⚠️ **Unity 宿主层实现，非 Core。浮点在此层合法**（不在 `Jianghu.Cultivation`，不受 B.2 IL 扫描）。Core 阶段不实现本式。

`q = round( 100 × exp( −Δ² / (2σ²) ) )`（玩家真按键时机；NPC 仍用上面的 q_core 整数式）

| 变量 | 类型 | 范围 | 说明 |
|---|---|---|---|
| Δ | float | 0–window | 按键时刻偏离窗口中心（仅玩家，实时采集） |
| σ | float | window/3 | 标准差 |
| window | int | 见表 | 窗口宽度 = 操作类型基值 × 功法品级系数 |

**window 宽度表**（Unity 用）：

| 操作 | 基值 window₀(ms) | 品级系数 |
|---|---|---|
| 格挡 parry | 200（窄，高回报） | ×(100+tier×15)/100 |
| 闪避 dodge | 320（中） | ×(100+tier×12)/100 |
| 打断 interrupt | 260（要踩蓄力） | ×(100+tier×15)/100 |
| 反弹 reflect | 230 | ×(100+tier×12)/100 |
| 夺舍续命 | 160（极窄，生死一线） | ×(100+tier×20)/100 |
| 逆演回滚 | 200 | ×(100+tier×18)/100 |

> tier=功法品级（1–5）。品阶越高窗口越宽 → 容错越高（承用户"按品级调窗口"）。Unity 阶段玩家体验；NPC 仍走 Core 整数 q_core。

### 伤害（承 R2 §4，即时类操作加 q 乘子）

`dmg = clamp( (raw − mit) × (100+kè)/100 × q_factor, Floor, PowerCap )`
- `raw/mit/kè` 同 R2（攻击维/招式 Tier/招架闪避/相克）。
- `q_factor`：若防御方触发即时反应，按 q 衰减入伤（格挡 q 高→入伤大减；闪避 q 高→近乎免伤）。非即时操作 q_factor=1。

## Edge Cases

- **若 speed 完全相等**（同路镜像）：行动条同时满 → 按 CharacterId 序定先后（确定 tiebreak）。
- **若一击致死但防御方有"硬抗/不灭金身"**：先判被动触发（HP 将≤0 时），成功则 HP 保留至阈值、消耗续命资源；失败正常死亡。
- **若多个即时反应同时可触发**（既能闪又能格）：按功法门控优先级（玩家选/NPC 取 q 期望最高者），只触发一个。
- **窗口外应对（Δ>window）**：q=0，操作失败（闪空/格崩），且可能有后摇惩罚（落空硬直）。
- **NPC vs NPC 无玩家**：即时类操作走 **Core 整数 q_core 计算**（非浮点窗口），纯 Core 结算，headless 可跑、逐字节复现（首版扰动=0 纯确定）。
- **夺舍续命失败**（遇雷/纯阳/佛光在场）：确定性判定（非掷骰），失败=永久退场。
- **逆演回滚**：只撤上一次交锋结算（伤害/夺运/胜负），**Karma/天谴债不回滚**（防无限重开）。
- **off（cultivation-off）**：无 Cultivation，不入本系统，走 legacy `SparAction`（红线 B.3 逐字节守）。
- **速度无限叠（haste 类）**：speed 设 cap，防一方无限连动。

## Tuning Knobs

| 旋钮 | 默认 | 安全域 | 调高/调低影响 |
|---|---|---|---|
| ACTION_THRESHOLD | 1000 | 500–2000 | 高→行动慢节奏长；低→快节奏 |
| Ks（悟性→speed） | 2 | 1–4 | 高→悟性主导先手；低→各路趋同 |
| 回合上限 N | 不固定（HP 决） | — | 防战斗无限拖（cap 兜底） |
| window₀（各操作） | 见公式表 | ±50% | 宽→即时操作好做（偏休闲）；窄→硬核 |
| 品级系数斜率 | tier×12~20 | — | 大→高功法反应碾压；小→品级影响弱 |
| Q_MAX | 100 | — | 即时反应满效果上限 |
| Floor/PowerCap | 承 INV-CROSS | — | 保底伤/伤害上限 |
| NPC base_skill | 难度档 | 0–100 | 高→NPC 反应强（高难）；低→NPC 笨 |

## Visual/Audio Requirements

> 当前 v1 为 CLI 文本快照；以下为 Unity 阶段（即时窗口的呈现是即时层的核心体验）。

- **即时窗口视觉**：敌方杀招起手时，目标身上/招式轨迹给"窗口提示"（如闪光带、读条收束圈），窗口中心有明确视觉峰值（33 号式 parry 提示）。

## Acceptance Criteria

- **GIVEN** 两 NPC 同 UT 不同路，**WHEN** 跑 1000 场 Core 战斗，**THEN** 胜率∈[40,60]%（C1 契约，纯 Core 确定可复现）。
- **GIVEN** A speed 远高于 B，**WHEN** 战斗，**THEN** A 行动次数显著多于 B（CTB 生效）。
- **GIVEN** 防御方修了轻功且持闪避操作，**WHEN** 被攻击且 q 采样高，**THEN** 入伤大幅衰减；**未修轻功**则无闪避能力（功法门控）。
- **GIVEN** NPC×NPC 无玩家，**WHEN** 即时类操作触发，**THEN** 走 Core 整数 q_core 计算（无浮点/无 exp）、headless 跑通、同种子逐字节复现（B.2）；首版扰动=0 时纯确定。
- **GIVEN** 攻击方蓄力中，**WHEN** 防御方在打断窗口内 q 高，**THEN** 蓄招被打断、不结算。
- **GIVEN** 防御方濒死且会夺舍，**WHEN** 夺舍窗口 q 成功且无雷/纯阳/佛光，**THEN** 续命；**否则**永久退场。
- **GIVEN** cultivation-off，**WHEN** 切磋，**THEN** 走 legacy SparAction、与 v1.0 逐字节一致（B.3）。
- **GIVEN** 逆演触发，**WHEN** 回滚，**THEN** 撤上一次交锋但 Karma 不回滚。
- **GIVEN** 品级越高的同类闪避功法，**WHEN** 同等偏差应对，**THEN** 高品级 q 更高（窗口更宽，容错验证）。

## Open Questions

- **窗口宽度具体数值**：window₀ 表是初值，需 INV-CROSS 标定阶段对拍校准（属 balance-cross epic）。
- **NPC q 采样的难度曲线**：不同江湖角色（杂兵 vs 高手）的 base_skill 怎么分布？依赖 AI/难度系统设计。
- **即时窗口在多人混战**（>2 角色）如何处理：先按 1v1 落地，群战窗口规则待评估。
- **攻击方是否也有窗口**（用户②留口：蓄力重击在窗口内释放伤害更高/连击踩点）：本版聚焦防御侧即时窗口，进攻侧即时留作扩展。
- **逆演/夺舍/落宝的 Special handler**：batch3 已 wiring 8 个，但"真濒死判定/栈回滚"依赖 batch4 回合态（已显式 defer）。

### 复审遗留项（独立 design-review 2026-06-16 提出，记此不静默 A.8）

- **C3 Evade 公式统一**：模块化稿 §3.2 的 `Evade`（OnDefend，身法值≥攻方命中−阈则减免）与本 GDD 的 q_core 需统一——**裁决：Evade 走 q_core 整数式**（身法/悟性入 INS_GAIN，攻方命中入 atkPress），废"身法值≥命中"的独立阈值表述，batch4 实现时以 q_core 为准。
- **C4 回合上限 N**：非固定回合，按"双方总行动数配额"算（见 C1 机制②）；具体配额值 = balance-cross 标定旋钮。Formulas 已补 ACTION_THRESHOLD；N 的语义 = 行动队列总配额上限（防无限拖），默认值待标定。
- **C7 功法门控反向门（破坏性变更，需标注）**：本 GDD 要求"无对应功法→该能力为 0"（没轻功不能闪）。现 `SparAction` 用统一 Power 公式、无门控。**batch4 实现 DuelEngine 时需落 M5 反向门控**（Evade/护体模块来自功法，无功法则模块不存在），这是对现有战斗的破坏性升级，已在 batch4 story（story-003）记。off 路径（legacy SparAction）不受影响（B.3）。
- **操作分类学 32 项的"已实现 vs 设计待建"边界**：表中操作分两类——**已有 CombatSkillDef + EffectOp 实现**（batch2/3 的攻击/控制/资源类）vs **设计待建**（即时类闪/格/打断的 OnDefend 触发、夺舍/逆演的 batch4 回合态）。batch4 实现时以"已实现优先、即时类 Core 退化、玩家手感 Unity"分期落地。
