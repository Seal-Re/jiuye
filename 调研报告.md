# 武侠角色游戏 · 前期调研报告

> 目标：设计一款"武侠世界观 + 角色可通过 API 行动（每角色有背景设定）或按简单 AI 逻辑自主行动 + 随机生成但有上限约束的数据"的简单游戏。
> 方法：deep-research 工作流（5 路并行搜索 → 抓取 24 源 → 提取 117 条断言 → 3 票对抗验证 → 合成，106 个 agent）+ 后台无头浏览器抓取知乎/机核/百科（严格不开前台窗口）。
> 日期：2026-06-13

---

## 一、结论速览（TL;DR）

1. **"角色通过 API 行动"的工业范式 = function-calling 从枚举动作表里选**（GIGAX/NPC-Playground），而非放任自由文本——这天然实现"有上限/受约束"。
2. **"简单 AI 逻辑自主行动"的现成骨架 = MUD 的心跳循环**（`HeartOfWorld:Tick()` → 每角色 `HeartBeat()`）。每 tick 查状态→选动作→执行。
3. **"随机但有上限"分两层**：① 属性侧 = 总和固定 + 单项封顶（北大侠客行：四维总和 80、单项≤30、系统天花板 50/觉醒 60）；② 抽取侧 = 最少使用加权随机去重（AI Dungeon `variedRandomSelector`）。
4. **"角色背景"= 结构化人设字段 + per-角色隔离记忆**（GIGAX 的 `PROFILE/MEMORIES/ALLOWED_ACTIONS`；Inner-Self 按角色名键控的私有记忆）。
5. **世界观天然容器 = 门派**：门派既约束可用武功，又用"门规/行为准则"把背景直接编码成硬 AI 规则（北大侠客行十二大门派 + 门规；苍茫孤影行门派准则带优势劣势）。
6. **硬性避坑（对抗验证 0-3 否决）**：**不要**用裸自然语言 prompt 取消脚本/行为树来驱动决策。LLM 只负责"选/生成"，动作约束、状态持久化、一致性必须由 prompt 之外的工程结构（枚举动作 API + 状态机 + 记忆模块 + 确定性解析）承担。

## 二、按方向分述

### 方向 1 · 武侠文字游戏 / MUD / IF / AI Dungeon

**代表案例**
- **北大侠客行（PKUXKX）**：1996 起持续运营至今的金庸世界观 MUD，华语 MUD 长寿标杆。门派为核心组织武功与世界观——"十二大门派"（武当/华山/明教/全真/天龙寺/桃花岛/少林/古墓/丐帮/峨嵋/日月神教/星宿）+"十二小门派"；每门派有公共武功、门规（少林只收男徒、峨嵋只收女徒）、拜师习武子系统。角色系统 = 天赋属性/个人特征/年龄/道德/等级多维。〔来源：https://www.pkuxkx.net/wiki/ ；佐证 触乐 chuapp.com/282887〕
- **机核《从头做一个 MUD》（Lua 实现）**：给出可直接照搬的极简架构——`HeartOfWorld`（世界心脏，定时 `Tick()`，每个对象 `HeartBeat()` 心跳）；对象模型 `SpaceObject`(可嵌套容器)→`Room`(title/desc/exits/channel)→`Char`→`Player`(继承+存档)；命令系统(玩家输入文字→解析→查 `cmd_tab`→调 `cmd_fun`)；战斗 = tick 驱动 + 招式克制 `tiger>monkey>crane`(石头剪刀布)算伤害。〔来源：https://www.gcores.com/articles/181122 ；开源 daxiaohan.coding.net/public/luamud〕
- **AI Dungeon 生态**（两个工程范式可直接复用）：
  - **Inner-Self mod**：给每个故事角色独立的记忆/目标/秘密/规划/自我反思；per-NPC 私有分段记忆（按角色名键控、彼此隔离）；角色能自行修订/裁剪心智状态（LLM 输出 `assign/forget/rename` 算子→确定性解析写回 brain card）。〔来源：https://github.com/LewdLeah/Inner-Self〕
  - **Atkana 脚本**：`variedRandomSelector`（最少使用加权随机、不重复直至循环、历史持久化到 state）；`extractFields` 正则 `([A-Z ]+):(.[^;]*);` 解析定界人设字段。〔来源：https://github.com/Atkana/AI-Dungeons-Scripts〕

**对本项目**：MUD 的 tick+命令系统 = "API 行动"与"自主行动"的共同底座；AI Dungeon 的"名键控隔离记忆 + 定界字段人设模板"是**无需向量库**的轻量起步路径。

### 方向 2 · 武侠 / 江湖 TRPG 桌游

**代表案例**
- **苍茫孤影行（Walking Alone in a Vast Expanse）**：受 D100 地牢启发的**单人武侠纸笔 RPG**，已发行（v0.2.0, 2025-01）。骰子 + 六格地图(hexcrawl)；核心循环由**门派选择 / 武功选择 / 任务选择**三轴构成；门派施加行为准则约束角色在事件中的决策（**带相应优势与劣势**）；武功决定战斗风格；升级不靠经验，而靠完成所学武功的特定任务。〔来源：https://rottenwalker.itch.io/walking-alone-in-a-vast-expanse〕
- 中文 TRPG 社区资源：ttrpg.cn、gameres 论坛跑团规则讨论。〔http://www.ttrpg.cn/ ；https://bbs.gameres.com/thread_9784_1_1.html〕

**对本项目**："门派行为准则 → 优势/劣势"是把"背景"直接编码进可计算 AI 逻辑的**桌游级先例**。三选择轴(门派/武功/任务)可作为角色生成与行为驱动的骨架。注意：桌游机制迁到 API/agent 需"翻译"而非直接移植。

### 方向 3 · 武侠小说 / 网文世界观要素

**核心要素**
- **门派体系**：门派是武侠世界的基本组织单元，承载武功传承、门规、地理、恩怨。〔https://zh.wikipedia.org/zh-hans/門派_(武俠)〕
- **"侠客多有官方背景"套路**（强约束模板）：段誉=大理世子、楚留香=藩王外孙、李寻欢=朝廷官员；门派常受朝廷补贴；主流叙事"反贪官不反皇帝"。可作随机背景生成的合法性约束。〔来源：知乎侠客史料库 https://www.zhihu.com/question/402703112〕
- 角色设定模板 / 网文套路拆解：〔https://zhuanlan.zhihu.com/p/653190295 ；https://zhuanlan.zhihu.com/p/705189824〕

**对本项目**：门派 = 世界观+约束的双重容器；"官方背景/恩怨/师承"是生成角色背景时的关系性约束维度。⚠️ 本轮小说/网文侧来源较薄（见开放问题），功法/境界/恩怨体系的结构化模板需补充。

### 方向 4 · 贴吧 / 社区文字 RPG

- 知乎侧 MUD 讨论确认仍在运营的中文武侠 MUD：北大侠客行、《武林》MUD（含同人小说）。〔https://www.zhihu.com/question/24976404〕

### 方向 5 · 武侠模拟/沙盒的随机有上限系统

**代表案例与机制**
- **北大侠客行 · 天赋属性系统（"随机生成但硬上限"教科书级范例）**：创建时随机生成数值，但带硬约束——**每项属性上限 30、系统天花板 50（觉醒可达 60）、四主属性总和固定 80**；装备附加效果生成时随机但在明确数值区间内（暴伤 5%–50%、攻速 10%–30%、闪避 10–50 级）。〔来源：https://www.pkuxkx.net/wiki/ (newbie/gift, newbie/zhiyezhuangbei)〕
- **鬼谷八荒**：修仙体系 + 山海经背景，炼气期模块化进阶，开放世界沙盒（首月 180 万份）。命格/属性养成。〔https://www.gamersky.com/handbook/202102/1359792_2.shtml ；https://www.zhihu.com/question/442111017〕
- **侠客风云传**（《武林群侠传》脉络）：属性/数值养成。〔https://zhuanlan.zhihu.com/p/567808519〕

**对本项目**：分两层落地——**属性用"总和固定 + 单项封顶"约束，事件/掉落用"最少使用加权随机"去重**。注意 `variedRandomSelector` 只解决"不重复"，不限数值幅度；数值封顶要另用 PKUXKX 式上限。

### 方向 6 · LLM / API 驱动 NPC（"角色通过 API 行动"核心）

**奠基范式**
- **Generative Agents（斯坦福，UIST'23）**：扩展 LLM 以 ① 自然语言记忆流、② 随时间综合为高层反思、③ 动态检索记忆来规划行为。25 个自主智能体在《模拟人生》式沙盒中，从单一种子意图(一个 agent 想办情人节派对)**无脚本涌现**出散播邀请→结识→约会→协同到场的多智能体社交。〔来源：https://arxiv.org/pdf/2304.03442〕
  - ⚠️ 失败模式（论文自录）：幻觉编造、仅 5/12 受邀者实际到场、instruction-tuning 致过度合作 → 对"恩怨/门派对立"的江湖需护栏。
- **Voyager（NVIDIA 等）**：纯黑盒 GPT-4 API 调用驱动具身行动，**无需微调参数**，"代码即动作空间"。三组件：① 最大化探索的自动课程、② 存可执行代码的技能库、③ 融合环境反馈/执行错误/自我验证的迭代提示。〔来源：https://arxiv.org/abs/2305.16291 ；开源 github.com/MineDojo/Voyager〕
  - ⚠️ 局限：仅处理可自动验证任务，自我验证可能无法从错误恢复 → 角色动作宜限定为可验证类。

**工程模式（可直接复用）**
- **受约束动作 API = function-calling 选动作**（GIGAX/NPC-Playground）：模型读"场景状态 + 近期事件 + 可用动作列表 + 角色心理画像与记忆"，输出枚举动作之一并带实体引用参数（如 `say NPC1 "Hello, Captain!"`）。模型卡逐字字段：`PROTAGONIST PSYCHOLOGICAL PROFILE / MEMORIES / ALLOWED ACTIONS`。〔来源：https://huggingface.co/blog/npc-gigax-cubzh ；https://huggingface.co/Gigax/NPC-LLM-3_8B〕
- **人设固化进权重（省 token/抗注入脆弱）**：LoRA 微调小模型(SLM)编码每个 NPC 的语气/行为约束/可知知识；每 NPC 配两个 ChromaDB 记忆库（会话 + 世界知识），运行时热插拔（切换<0.03s）；单共享底模可驱动多个同型 NPC（如多个客栈老板）各带不同上下文。〔来源：https://arxiv.org/pdf/2511.10277〕
- **OCEAN 行为体**：无状态 LLM + 自动提示生成器，由 NPC 特质/游戏状态/环境动态构建上下文 prompt；大五人格喂入产出可辨别的特质一致行为（验证为"可行性演示"级，n=8）。〔来源：https://etasr.com/index.php/ETASR/article/view/12631〕

**❌ 硬性避坑（对抗验证 0-3 否决）**：「仅靠编码人格特质的自然语言 prompt 即可驱动 NPC 决策、从而完全取消脚本/行为树」被否决。**纯 prompt 不能单独承载行为逻辑**，必须配 function-calling 受限动作集 + 状态机/记忆 + 确定性解析层。〔来源同上 etasr 12631 (refuted)〕

## 三、对本项目的设计启示（架构建议）

**推荐：确定性引擎为脊柱，LLM 为可插拔的"决策选择器 + 文本生成器"**。同时满足"API 行动"与"简单 AI 逻辑自主行动"，并尊重 0-3 否决的避坑结论。

```
角色 Character = {
  背景 persona:  { 门派, 师承, 官方背景, 性格(OCEAN/标签), 目标, 秘密 }   // 定界字段，可喂 LLM
  数据 stats:    { 武力/内力/... 单项≤cap, 四维总和=固定值 }            // 随机生成+硬上限
  记忆 memory:   per-角色隔离(按名键控)，可被角色自行增删改           // Inner-Self 式
  门派规则 code: 门规→硬性行为约束(优势/劣势)                          // 背景→AI 硬规则
}

世界循环（每 tick）:
  for 每个 NPC:
    ctx   = 收集(场景状态 + 该角色记忆 + 该角色 ALLOWED_ACTIONS)
    动作   = 决策(ctx)         // 规则分支 OR LLM function-call，二选一/混合
    执行(动作)               // 动作来自枚举集，参数引用实体 → 天然有上限
    更新(状态, 记忆)
```

- **"角色通过 API 行动"** → 落为枚举动作集 + 实体引用参数的 function-calling（GIGAX 模式）。
- **"简单 AI 逻辑自主行动"** → tick 循环 + 规则/行为树；LLM 可选地接进决策点，但必须有状态机兜底。
- **"随机但有上限"** → 属性层(总和固定+单项封顶) + 抽取层(最少使用加权去重)，两层独立。
- **世界观** → 以门派为容器组织武功(动作约束)+门规(行为硬规则)+恩怨(关系图)。

## 四、待定的关键决策（"其他约束待定"的真正分叉点）

1. **LLM 集成路线**：纯黑盒 API（零训练、低门槛、依赖外部配额/延迟）vs 本地 LoRA 固化人设+模块化记忆（可离线、多 NPC 共享底模、需显存训练）vs **纯规则无 LLM**（最简、可控、零成本）。三者成本/延迟/一致性未量化对比。
2. **游戏规模**：单人 solo（太吾/苍茫孤影行式，1 主角 + 脚本 NPC）vs 多智能体涌现江湖（Generative Agents 式，N 个自主角色相互作用）。后者更炫但有行为漂移/剧情失控风险。
3. **平台形态**：CLI 纯文字（MUD 式）vs 其他界面。
4. **随机上限的具体数值**：四维总和、单项 cap、事件去重池大小等。
5. **护栏**：对"恩怨/门派对立"如何在保留涌现性的同时防漂移（缺现成方案）。

## 六、置信度与时效声明

- 全部发现以一手来源(arXiv 同行评议 / 官方仓库源码 / 官方博客/模型卡 / 游戏官方 wiki)支撑，多为 3-0 一致票，证据强度高。
- 需打折：OCEAN 行为验证(n=8，2-1)读作"可行性演示"；Voyager 终身学习(2-1)自主性仅限可验证任务；苍茫孤影行无任何 AI 表述（仅"单人+骰子"分支成立，勿宣称其有 AI 角色）。
- 时效：Fixed-Persona SLM(2025-11)、OCEAN(2025-10)、GIGAX(约2026-01)较新，细节可能演进；PKUXKX 主页当前宣传"三十年"。
- Inner-Self 为 ~55 star 个人 mod（逻辑隔离非物理隔离），原型级，生产化需自建隔离。
